# AGENTS.md

Instructions for AI agents working on this repository. Keep this file current.

## Documentation Discipline (MANDATORY)

Every change MUST update three artifacts in lockstep:

1. **`CHANGELOG.md`** ‚Äî add entry under `## [Unreleased]` for every user-visible or operator-visible change. On tag, rename `Unreleased` to the new version with the release date, and open a fresh `Unreleased` section. Categories: `Added`, `Changed`, `Fixed`, `Removed`, `Deprecated`, `Security`.
2. **`README.md`** ‚Äî keep `Status`, `Quickstart`, `Architecture`, `Repo Layout` honest. If a phase ships, update the Status block. If a flag/command changes, update the Quickstart.
3. **`AGENTS.md`** (this file) ‚Äî when a new workflow, convention, or guard-rail emerges, add it here so the next agent inherits it.

If a PR touches code but skips any of these three, it is incomplete.

## Release Workflow

1. Land all `Unreleased` work; verify `cargo fmt && cargo clippy -- -D warnings && cargo test --workspace` and `az bicep build` clean across all modules.
2. Edit `CHANGELOG.md`: rename `## [Unreleased]` ‚Üí `## [X.Y.Z] - YYYY-MM-DD`, add a fresh empty `## [Unreleased]` block at the top, update the link references at the bottom.
3. Edit `README.md` Status block if phase advanced.
4. Bump version: `crates/*/Cargo.toml`, the `--azcluster-version` CLI default in `crates/azcluster-cli/src/main.rs`.
5. Commit, tag `vX.Y.Z`, push tag ‚Äî CI publishes the release.
6. Live-validate on `paul-azcluster`/`southafricanorth` before declaring done.

## v0.19.1 deploy UX (`azcluster resume`)

- v0.19.0 overloaded `azcluster deploy` to detect a pending marker and switch to "resume" mode. That was confusing ‚Äî "deploy" reads as "deploy again". v0.19.1 replaces it with an explicit verb: **`azcluster resume --name <name>`**.
- New invariant: **`deploy` is strictly linear**. It writes the pending marker, submits ARM, and (without `--no-wait`) calls `finalize_deploy()` inline + deletes the marker. There is no resume branch in `deploy()` anymore.
- New invariant: **blocking deploys also write+delete the pending marker** (around the ARM submission). If the operator's shell dies mid-deploy (terminal disconnect, hibernate, OOM kill), `azcluster resume --name <name>` recovers ‚Äî same code path as `--no-wait`.
- `finalize()` reads the pending marker, polls ARM until terminal, loads `ClusterSecrets`, queries `az group show --query location` to recover the missing `location` field (PendingDeploy schema doesn't persist `location`), builds a synthetic `DeployArgs`, calls `finalize_deploy()`, deletes the marker. On Failed/Canceled it bails with the standard delete-+-remove-pending-file hint.
- `status` now appends `-> run azcluster resume --name <name> once ARM state is Succeeded` to the pending block, and the no-state footer points at `resume` instead of re-running `deploy`.
- `resume_deploy()` function deleted. `PendingDeploy` schema unchanged from v0.19.0 (still 8 fields: `cluster, deployment_name, resource_group, started_at, monitoring_enabled, accounting_enabled, shared_storage, grafana_location`) ‚Äî backward-compatible with v0.19.0 markers (validated live by `azcluster resume` finalising the `v19walk` pending marker that was created with v0.19.0).

## v0.19.0 deploy UX (`--no-wait` + resume + enhanced status)

- **`--no-wait`** is plumbed through `clap` on `DeployArgs` and appended verbatim to `az deployment sub create`. CLI exits after writing `~/.config/azcluster/clusters/<name>-pending.toml` (`PendingDeploy` schema: `cluster, deployment_name, resource_group, started_at (UTC iso8601), monitoring_enabled, accounting_enabled, shared_storage, grafana_location`). Secrets MUST be persisted **before** `az deployment sub create` returns ‚Äî ARM secure parameters are not retrievable from `properties.outputs`, so the resume path needs them on local disk. v0.19.0 enforces this by calling `secrets.save(...)` before building the `az` argv.
- **Resume**. Re-running `azcluster deploy --name <name> ‚Ä¶` with the pending file present jumps into `resume_deploy()` (skips template/SSH-key resolution, skips `az group create`, skips a fresh ARM submission). Polls `az deployment sub show --query properties.provisioningState` every 30 s (cap 90 min). On `Succeeded` it loads the persisted `ClusterSecrets` and calls `finalize_deploy()` (state file, timings capture, dashboard import), then `PendingDeploy::delete`. On `Failed`/`Canceled` it aborts with a recovery hint pointing at `azcluster delete` + manual pending-file removal ‚Äî does NOT auto-tear-down.
- **`finalize_deploy()`** is the single post-ARM path. Same call signature for blocking and resume flows. Takes the resolved RG explicitly (computed once at deploy start) so resume doesn't have to redo the `unwrap_or_else(|| format!("rg-azcluster-{}", name))` dance.
- **`azcluster status`** now tolerates the no-state-file case (pending-only) AND the pending-and-state-file case (post-finalize but pending marker not yet cleaned up). SSH probe uses `BatchMode=yes`, `ConnectTimeout=8`, `StrictHostKeyChecking=accept-new`, `UserKnownHostsFile=/dev/null`, `LogLevel=ERROR`. Scheduler probe goes via ProxyJump through login. Both probes are best-effort ‚Äî any SSH error prints `ERR (...)` and the command continues.
- **`azcluster delete`** now falls back from `ClusterState::load` to `PendingDeploy::load_optional`, so an aborted `--no-wait` deploy is still tearable down via the CLI. Both files are removed on success.
- Pending file path: `~/.config/azcluster/clusters/<name>-pending.toml`. State path: `~/.config/azcluster/clusters/<name>.toml`. Secrets: `~/.config/azcluster/clusters/<name>-secrets.toml`. Do not rename without grepping all three across `main.rs` + `cluster_state.rs`.
- **`Command::new("ssh").args(["host", "--", "bash", "-lc", "cmd"])` is BROKEN** (v0.19.0 fix). OpenSSH whitespace-joins all positional remote args before passing them to the remote login shell, so the remote sees `bash -lc cmd arg1 arg2` and bash parses it as `-c 'cmd'` (ignoring everything after). When you need to run a multi-token shell command via `ssh`, pass it as a single string positional arg: `cmd.args(["host", "tail -n1 /path 2>/dev/null || echo MISSING"])`. The login shell on the remote will then parse and exec it correctly. Live-bitten by `bootstrap_probe` in v0.19.0 before live-validation caught it.

## Azure / Infra Gotchas

- AzSecPack policy auto-attaches `AzSecPackAutoConfigUA-<region>` UAI to every VM/VMSS. Any PUT on a VMSS `identity` collection triggers `LinkedAuthorizationFailed` unless the caller has `Managed Identity Operator` on that UAI (out-of-band). **Use `az vmss scale --new-capacity`, not `az vmss update --set sku.capacity`.**
- Slurm configless mode does **not** distribute `plugstack.conf.d/`. Write `/etc/slurm/plugstack.conf` (absolute path) on every node that uses Pyxis (compute + login).
- Enroot needs `1777` on `/var/lib/enroot{,/runtime}`, `/var/lib/enroot-data{,/cache}`, and `/run/enroot` ‚Äî top-level AND subdirs.
- VMSS Flex VMs surface as `Microsoft.Compute/virtualMachines` named `vmss-<cluster>-<pool>_<hex>`, not under the VMSS resource.
- Image: `microsoft-dsvm:ubuntu-hpc:2404` (default), `2204` fallback.
- Slurm 25.11 + Pyxis 0.21.0 ABI match. NVIDIA Pyxis 0.24.0 exists; only bump if Slurm 26 is needed.
- Slurm 25.11 dynamic nodes: `slurmd --conf "...Partitions=<pool>"` is rejected ("Failed to parse nodeline"). Use the **NodeSet+Feature** pattern: emit `NodeSet=<pool>set Feature=pool_<pool>` and `PartitionName=<pool> Nodes=<pool>set ...` in `slurm.conf`; the compute slurmd registers with `--conf "...Feature=pool_<pool>"` and slurmctld places it in the matching NodeSet/partition.
- Pyxis spank library (`spank_pyxis.so`) must be installed on **every** node that may submit `srun` ‚Äî scheduler, login, and compute ‚Äî because `plugstack.conf` loads at submit time. Forgetting it on scheduler crashes any `srun` invoked from there with `Dlopen of plugin file failed`.
- The `microsoft-dsvm:ubuntu-hpc` image ships `nvidia-smi` even on CPU SKUs, so `command -v nvidia-smi` cannot be used as a GPU presence check. Use `nvidia-smi -L | grep -cE '^GPU [0-9]+:' || true` instead.
- **Managed Grafana region coverage**: `Microsoft.Dashboard/grafana` is NOT available in `southafricanorth` (and several other regions). Use `--grafana-location uksouth` (or another supported region) when the cluster region lacks AMG.
- **Monitoring Data Reader role GUID** is `b0d8363b-8ddd-447d-831f-62ca05bff136` (NOT the `...51537` value some docs list). Verify role GUIDs with `az role definition list --name "..."` before baking into Bicep.
- **AMW ingestion RBAC scope**: `Monitoring Metrics Publisher` MUST be granted on the AMW's **default Data Collection Rule**, not on the AMW account itself. Role at the AMW scope passes role-listing checks but the ingestion endpoint still returns `403 The authentication token provided does not have access to ingest data for the data collection rule with immutable Id 'dcr-...'`. The default DCR lives in the Azure-managed sister RG `MA_<amwName>_<location>_managed`.
- **AMW ingestion RBAC propagation**: 5-10 minutes on a freshly-created MI + DCR. After it propagates, `systemctl restart prometheus` is required - prometheus and IMDS cache the bearer token, and the cached token's authorization is decided server-side at request time but the connection state appears to survive the role landing. A direct `curl` test (`POST` empty body to the ingestion URL with an IMDS token) is the fastest way to confirm whether the failure is RBAC vs config.
- **Prometheus 3.x cloud-init perms**: `cat > /opt/prometheus/prometheus.yml <<EOF ... EOF` in a cloud-init `runcmd` produces a file with mode `0600` (root's umask). The non-root `prometheus` service user cannot read it -> "permission denied" at startup. Always follow with explicit `chmod 0644`.
- **Slurm conf files cloud-init perms** (v0.13.9 fix). Same root cause as the Prometheus gotcha above: cloud-init `runcmd` inherits a 0077 umask so `cat > /etc/slurm/slurm.conf <<EOF` produces 0600 files. `srun`/`sinfo` run as the submitting (non-root) user and parse `/etc/slurm/slurm.conf` + `/etc/slurm/plugstack.conf` locally on the submit host ‚Äî Pyxis spank plugins load at submit time, not exec time. With 0600 perms users get `error: s_p_parse_file: unable to read "/etc/slurm/slurm.conf": Permission denied` and `error: spank: Failed to open /etc/slurm/plugstack.conf: Permission denied`. Always `chmod 0644 /etc/slurm/*.conf` after writing them in cloud-init. Applies to scheduler (slurm.conf + plugstack.conf + cgroup.conf + gres.conf), login (plugstack.conf ‚Äî slurm.conf comes via sackd configless), and compute (plugstack.conf ‚Äî slurm.conf + gres.conf come via slurmd configless cache).
- **Compute `/etc/hosts` 127.0.1.1 breaks cross-node Gloo/PyTorch rendezvous** (v0.13.9 fix). The Ubuntu cloud-image default `/etc/hosts` has `127.0.1.1 <hostname>` after `hostnamectl set-hostname`. `torch.distributed.new_group(backend="gloo")` (Megatron-Bridge uses this for CP groups during init) calls `gethostbyname(hostname)` to learn its own advertised address, and Gloo `connectFullMesh` then expects every other rank to dial that address. With `127.0.1.1` every remote rank dials its own loopback and the rendezvous stalls with `Gloo connectFullMesh failed ... timed out connecting: SO_ERROR: Connection refused, remote=[127.0.1.1]:20901`. NCCL is unaffected (rendezvous is via `MASTER_ADDR`, set explicitly by Megatron-Bridge's launcher) ‚Äî the failure is at the PyTorch-level Gloo PG used for control-plane groups. Live-validated on `paul-azcluster-h100d` Llama 3.1 8B 2-node BF16 (job 16 failed pre-fix, job 20 succeeded post-fix). Fix in `cloud-init/compute.yaml.tmpl`: after `sed -i '/^127\.0\.1\.1/d' /etc/hosts`, write `${IPV4} ${NEW_HOSTNAME}` (eth0 IP, NOT 127.0.1.1). Scheduler/login don't need this ‚Äî slurm.conf resolves them via Azure DNS suffix.
- **Prometheus `azuread.managed_identity` remote_write**: works directly against the AMW DCE ingestion URL (`${dceEndpoint}/dataCollectionRules/${dcrImmutableId}/streams/Microsoft-PrometheusMetrics/api/v1/write?api-version=2023-04-24`). No AMA or DCR custom scrape config needed for VMs. Block shape:
  ```yaml
  remote_write:
    - url: "..."
      azuread:
        cloud: AzurePublic
        managed_identity:
          client_id: "<uai-client-id>"
  ```
  Audience is implicit (`https://monitor.azure.com/`). The UAI MUST be attached to the VM AND hold Metrics Publisher on the DCR (see above).
- **VMSS Flex + SystemAssigned identity** is rejected in subscriptions with AzSecPack/UAI-only policy (`InvalidParameter` on `identity`). VMs are fine; VMSS must use UserAssigned (or no MI).
- **Grafana dashboard JSON** lives under `grafana/dashboards/` and is provisioned by `bicep/modules/grafana-dashboards.bicep`. Keep panel ids stable across edits, use a datasource template variable named `DS_PROMETHEUS`, and never hardcode Azure Managed Grafana datasource UIDs.
- Phase 1+ test region: `southafricanorth`, RG `paul-azcluster`, max 2 GPU nodes.

If any of these fail, fix before claiming completion.
- **YAML write_files heredoc terminators**: cloud-init `write_files` with `content: |` block scalars strips the FIRST line's leading whitespace as the indent and removes that prefix from every subsequent line. Any `EOF` terminator MUST sit at exactly that base indent column in the template so it lands at column 0 in the materialised script - bash heredocs without `<<-` only recognise the closer at column 0. Extra indentation from being nested inside `if`/`for` (e.g. 6-space base + 2-space if-body = 8-space `EOF`) silently breaks the heredoc and chains errors until bash hits EOF with "syntax error: unexpected end of file". Always put `EOF` at the template's base indent column even inside `if` blocks.
- **AMG Grafana Admin for dashboard import**: `az grafana dashboard create` calls the Grafana HTTP API (`POST /api/dashboards/db`), which requires a Grafana role on the AMG. ARM contributor / owner is not enough. Grant the deployer principal `Grafana Admin` (`22926164-76b3-42b3-bc55-97df8dab3e41`) on the AMG resource scope, and expect 1-3 min propagation - retry with back-off on `401 NoRoleAssignedException`.
- **`Microsoft.Dashboard/grafana/dashboards@*` is NOT a real ARM resource**: BCP081 ("resource type does not have types available") is the warning, and ARM preflight returns `ResourceTypeRegistrationNotFound`. Import dashboards via the Grafana HTTP API (`az grafana dashboard create` or direct REST), not Bicep. The CLI does this post-deploy with the dashboard JSONs embedded via `include_str!`.
- **Shared filesystem modes**: `--shared-storage anf` (default) provisions Azure NetApp Files; `--shared-storage nfs-scheduler` exports `/shared` from the scheduler via `nfs-kernel-server`. The latter is test-only (no HA, SPOF on scheduler) but saves ~12 min off provisioning. Login + compute always mount with a 5-minute retry loop because the scheduler is still installing its NFS server while they boot in `nfs-scheduler` mode.
- **Deployment timing**: every successful `azcluster deploy` writes `~/.config/azcluster/deployments/<cluster>/<utc-stamp>.json` with per-resource `duration_seconds` and a `trend.tsv` for comparison. Inspect via `azcluster timings <cluster>`. Source: `az deployment operation {sub,group} list` recursed across nested modules; `properties.duration` parsed from ISO-8601 (`PT##H##M##S`).
- **`--no-monitoring` / `--no-accounting`**: monitoring + accounting default to ON. Pass these flags to skip them during rapid test deploys that don't depend on observability or accounting. Accounting (v0.13.0+) provisions an Azure Database for MySQL Flexible Server + runs `slurmdbd` on the scheduler.
- **MySQL Flex TLS CA**: Azure MySQL Flexible Server presents a DigiCert Global Root CA chain, which is **already in Ubuntu's `ca-certificates`** bundle. Set `StorageParameters=SSL_CA=/etc/ssl/certs/ca-certificates.crt` in `slurmdbd.conf`; do NOT try to download `https://dl.cacerts.digicert.com/DigiCertGlobalRootCA.crt.pem` ‚Äî that host returns a cert whose SAN doesn't match the hostname and `curl` aborts with `SSL: no alternative certificate subject name matches target host name`. v0.13.0/v0.13.1 had this bug; fixed in v0.13.2.
- **Slurm accounting (v0.13.0)**: MySQL Flexible Server (`mysql-<cluster>`, `Standard_B2ms`, MySQL 8.0.21, 50 GB autogrow, `publicNetworkAccess=Disabled`) is provisioned in a delegated subnet `10.42.8.0/29` (delegation `Microsoft.DBforMySQL/flexibleServers`). The CLI auto-generates the admin password from `/dev/urandom` (32 random chars from an ambiguity-free alphabet plus literal `Aa1!` to satisfy Azure's "‚â•3 character classes" policy ‚Äî Azure rejects passwords with only 2 classes) and passes it as a `@secure()` Bicep parameter. On the scheduler, the password lands in `/etc/azcluster/accounting.password` (mode 0600 root:root via cloud-init `write_files`) and is read into `slurmdbd.conf` (mode 0600 slurm:slurm) at boot, then the shell var is `unset`. TLS is required: the bootstrap downloads `https://dl.cacerts.digicert.com/DigiCertGlobalRootCA.crt.pem` to `/etc/slurm/ssl/` and `slurmdbd.conf` sets `StorageParameters=SSL_CA=‚Ä¶`. Bootstrap order MUST be: install `slurm-smd-slurmdbd` ‚Üí write `slurmdbd.conf` ‚Üí wait for `:3306` reachability (60√ó5s probe via `/dev/tcp`) ‚Üí start `slurmdbd` ‚Üí sleep 5s ‚Üí start `slurmctld` ‚Üí `sacctmgr -i add cluster`. Starting `slurmctld` before `slurmdbd` with `AccountingStorageEnforce=associations,limits,qos` set will cause slurmctld to refuse to start. Three MySQL server params are tuned per slurmdbd recommendations: `innodb_lock_wait_timeout=900`, `max_allowed_packet=16777216`, `log_bin_trust_function_creators=ON`.
- **`aka.ms/InstallAzureCLIDeb` is FORBIDDEN inside `set -euo pipefail` bootstraps** (v0.13.1 fix). The pipe script runs `apt-get install` internally without our `DPkg::Lock::Timeout=600`, so it races `apt-daily`/`unattended-upgrades` and aborts the entire bootstrap with `Could not get lock /var/lib/dpkg/lock-frontend`. Cloud-init still reports "finished" because runcmd pipes through `tee`. Always install azure-cli via the explicit MS apt repo (`packages.microsoft.com/repos/azure-cli/`) using our own `apt-get -o DPkg::Lock::Timeout=600 install -y azure-cli`. The MS GPG key (`/usr/share/keyrings/microsoft-prod.gpg`) is already imported earlier in the scheduler bootstrap for the slurm repo, so reuse it.
- **`tee` masks `runcmd` exit codes.** The compute/scheduler bootstraps run `/opt/azcluster/install-X.sh 2>&1 | tee -a /var/log/azcluster/install.log`. If the script aborts with a non-zero exit, the pipe still succeeds because `tee` (the last stage) returns 0. cloud-init then reports `finished` and you get a half-configured node that registers (or doesn't) with Slurm in a broken state. To diagnose, always `tail /var/log/azcluster/install.log` on the node and grep for `Error|fail|exit` rather than trusting `cloud-init status`.
- **`set -euo pipefail` + bash command substitution + pipelines is a sharp edge** (v0.13.4 fix). `drv=$(ethtool -i lo | awk ...)` aborts the whole script when the `awk` pattern matches nothing on a virtual interface, because the pipe's exit status under `pipefail` is non-zero and `set -e` kills the script. Always add `|| true` *inside* the `$(...)` for diagnostic pipes whose failure is expected and tolerated.
- **Slurm configless distribution does NOT push `gres.conf`** (v0.13.4 fix). `SlurmctldParameters=enable_configless` only distributes `slurm.conf`. Both scheduler (for slurmctld's `GresTypes=` validation) and every compute node (for slurmd's local Gres discovery) must have their own `/etc/slurm/gres.conf`. The simplest version that works on Azure is `AutoDetect=nvml` on both sides ‚Äî slurm-smd is built with NVML support, and slurmctld accepts NVML autodetect even though it has no GPUs itself.
- **Slurm caches a node's first registration** (v0.13.4 lesson). If a slurmd registers once with bad Gres/Parameters/CPUs, restarting slurmd alone does NOT re-trigger fresh registration; slurmctld holds the stale INVALID_REG state. Operator must `scontrol update nodename=<n> state=DOWN reason=... && scontrol delete nodename=<n>` from the scheduler, *then* restart slurmd on the compute node. Build automation around this if you're iterating on `--conf` strings.
- **`nvidia-smi --query-gpu=name` returns multi-token product strings** (v0.13.4 fix). On H100 it's "NVIDIA H100 80GB HBM3". Naive `awk '{print tolower($NF)}'` yields "hbm3" not "h100" ‚Äî Slurm registers `Gres=gpu:hbm3:8` and every `--gres=gpu:h100:N` job stays PENDING with reason `Resources`. Parse the first token matching `/^[A-Za-z][0-9]+$/` to pick out the SKU code (h100, h200, a100, v100, etc.).
- **NDv5 needs `Parameters=l3cache_as_socket` on slurmd's `--conf`, NOT `SlurmdParameters=` in slurm.conf** (v0.13.4 fix). Setting it via slurm.conf is silently ignored by slurmd (the option is per-node and read from the conf-server reply, not the global config). Setting it on the slurmd command line via `--conf "... Parameters=l3cache_as_socket"` works. Also: dropping `CPUs=` from `--conf` is required for autodetected S:C:T to honour the L3-as-socket grouping; explicit `CPUs=96` collapses everything back to one socket and confuses NCCL's GPU‚ÜîNIC affinity.
- **The 64 GB root disk on NDv5 is too small for container imports** (v0.13.4 fix). The `microsoft-dsvm:ubuntu-hpc:2404` image leaves ~7 GB free on `/` after base install. A single 10 GB Pyxis container import (e.g. `ghcr.io/azure/ai-infrastructure-on-azure/nccl-test:latest`) ENOSPC's mid-extraction. Always relocate `/var/lib/enroot` and `/var/lib/enroot-data` to symlinks under `/mnt` (the Azure ephemeral disk, ~956 GB NVMe-backed on NDv5) when `/mnt` is mounted.
- **HPC-X is the in-image MPI on Azure HPC/AI VMs** (v0.13.4 lesson). `/opt/hpcx-v2.25.1-gcc-doca_ofed-ubuntu24.04-cuda13-x86_64/` ships with OpenMPI + UCX + a bundled PMIx 4.2.x that ABI-matches Slurm 25.11's `mpi_pmix_v4.so`. Use it for bare-metal multi-node MPI/NCCL jobs by `source $HPCX_DIR/hpcx-init.sh && hpcx_load` then `srun --mpi=pmix ...`. The marketplace image also ships `/opt/nccl-tests/build/all_reduce_perf` prebuilt ‚Äî no need to compile NCCL tests yourself.
- **Cross-node containerised MPI via Pyxis needs slurmd PMIx env + enroot PMI hooks** (v0.13.6 fix; supersedes earlier "PMIx 4‚Üî5 ABI" claim; **live-validated v0.13.6 on `paul-azcluster-h100c`**: 2-node √ó 8-GPU NeMo container all-reduce reached a single 16-rank PMIx world and exited cleanly; orthogonal IB-visibility bug surfaced ‚Äî see next gotcha). The real failure mode on v0.13.4/v0.13.5 was NOT an ABI mismatch. Slurm 25.11 ships `mpi_pmix_v4.so` linked against PMIx 4.2.9; ND v5 host has PMIx 4.2.9; **NGC containers `nvcr.io/nvidia/nemo:25.07.02` and `nvcr.io/nvidia/pytorch:24.10/24.11/25.04-py3` ship HPC-X 2.20/2.21 ‚Üí PMIx 4.2.x** ‚Äî the ABIs already match. - **Cloud-init duplicate-writer bug** (v0.16.1 fix). When adding a new block to a `runcmd:` that writes to a path some earlier prototype block also writes to, the LAST writer wins because runcmd is sequential. v0.16 added a wrapper at `/usr/local/sbin/azcluster-healthcheck` that delegated to the new `/usr/local/bin/azhealthcheck` Rust binary, but left a pre-existing legacy inline-shell wrapper writing the same path further down `cloud-init/compute.yaml.tmpl` in place. The legacy block ran after the v0.16 block on every boot and overwrote it, so the Rust binary was never invoked. Lesson: when adding a cloud-init writer for a path, `grep -n '<path>' cloud-init/*.tmpl` first and consolidate. The legacy script ALSO hit the `command -v nvidia-smi` gotcha ‚Äî drained every CPU node every 5 min with `Reason=azcluster-healthcheck: nvidia-smi -L failed`. Live-reproduced on `paul-azcluster v16a` (southafricanorth, 2x Standard_D8as_v5) before the fix.

- **Heredoc inside cloud-init `content: |` block fails when delimiter is at column 0** (v0.18.0 lesson, sibling of the existing `write_files` heredoc terminator gotcha). The install-scheduler.sh script lives inside a `content: |` YAML block scalar with 6-space base indent. A bash heredoc body (e.g. `debconf-set-selections <<DEBCONF ... DEBCONF`) requires the closing delimiter at column 0 of the *materialized* script ‚Äî meaning it must be at column 6 in the template. Putting the body lines and closer at column 0 of the template ejects them from the YAML block scalar (YAML uses the first content line's indent as the base; lines with less indent terminate the scalar). YAML still parses (the line at column 0 starts a new top-level mapping key that gets ignored or errors elsewhere) but the materialized script is missing the heredoc body. Two fixes: (1) keep all heredoc body + closer at the YAML base indent (column 6) ‚Äî works if the consumer tolerates leading whitespace; (2) avoid the heredoc entirely ‚Äî use `printf '%s\n' "line1" "line2" ... | <cmd>`. We picked (2) for `debconf-set-selections` in scheduler.yaml.tmpl. Validate end-to-end before deploying: `sed -E 's/\{\{[A-Z_]+\}\}/PLACEHOLDER/g' cloud-init/X.yaml.tmpl > /tmp/x.yaml && python3 -c "import yaml; yaml.safe_load(open('/tmp/x.yaml'))"` for YAML, then extract each `*.sh` write_file and `bash -n` it.

- **`internalNsg` allows all intra-VNet traffic by default** (v0.18.0 verification, not a bug). NSG-default `AllowVnetInBound` permits VirtualNetwork‚ÜíVirtualNetwork on every port, so adding new control-plane services on the scheduler subnet (LDAP 389/TCP from login + compute subnets in v0.18.0) needs NO NSG rule changes provided source + destination are inside the cluster VNet. Same applies to the `loginNsg` which has explicit allow-22 rules on top of the default. Only `Internet‚ÜíVNet` (e.g. SSH from operator laptop) requires an explicit rule.

- **OpenLDAP 2.6 requires a STRUCTURAL objectClass on every entry** (v0.18.1 fix). An entry whose only objectClass is `extensibleObject` (AUXILIARY) is rejected with `Object class violation (65) ‚Äî no structural object class provided`. Common fix for "bag-of-attributes" entries (like the `cn=uidNext` counter): combine a minimal STRUCTURAL class such as `device` (which MUST only `cn`, and we already provide it) with `extensibleObject` to allow arbitrary attributes (e.g. `uidNumber`). Live-reproduced on v0.18.0 ‚Äî first failed entry in the LDIF aborted the entire `install-scheduler.sh` under `set -euo pipefail`, leaving the cluster half-provisioned (no `/var/log/azcluster/ready`, no `/shared/home`, no `chmod` on `/shared`, all downstream RBAC and `azcluster user` ops broken).

- **`ldapadd` idempotency**: re-running a previously-loaded LDIF returns exit `68` (`LDAP_ALREADY_EXISTS`). Combined with `set -e` this makes naive `if ! ldapsearch | grep -q ...` guards brittle: the lookup is racy right after `apt install slapd` (slapd's `cn=schema,cn=config` settles asynchronously), and once the guard misses, the second-run add also fails. Pattern that works: drop the guard, run `ldapadd -c -f file` (`-c` = continue on per-entry errors), capture the exit status with `set +e; ldapadd ...; rc=$?; set -e`, and explicitly tolerate `rc == 68` while bailing on any other non-zero.

- **`/shared` from ANF mounts as `drwxrwx--- nobody:nogroup` (0770) by default**, blocking non-`nobody` users from traversing it. Anything under `/shared` (e.g. `/shared/home/<user>` created by `pam_mkhomedir` with the right owner/perms) is unreachable for an LDAP-resolved user until the parent allows traversal. Fix: `chmod 0755 /shared` in the scheduler bootstrap before populating any user-facing subdirectory. Individual user home dirs stay 750 ‚Äî only the parent needs `o+x`.

- **OpenSSH `ssh -J ... bash -lc '$(cat)'` silently drops piped stdin** (v0.18.1 fix). When a Rust process invokes `ssh -o ... -J <login> <scheduler> -- bash -lc '<cmd>'` with `Stdio::piped()` on stdin and the remote command reads via command substitution (`X=$(cat)`), the subshell created by `$()` does NOT inherit the ssh channel's pipe stdin under ProxyJump + no-PTY, so `cat` reads **0 bytes** and `X` is empty. The same payload works when the remote command uses a bare pipeline (`cat | wc -c`) or heredoc (`<<EOF`). Diagnostic recipe: `echo -n "X" | ssh -J L T bash -lc 'V=$(cat); echo got=${#V}'` prints `got=0`, while `echo -n "X" | ssh -J L T bash -lc 'cat | wc -c'` prints `1`. Fix in azcluster CLI: drop stdin entirely; base64-encode the payload in Rust and inline it on the remote command line via `printf %s '<b64>' | base64 -d`. Adds ~20 LOC for an RFC 4648 encoder; no external dep needed (the CLI's auto-generated passwords use a shell-safe alphabet, so the b64 string survives single-quoting trivially).

## v0.19.2 gotchas (--extra-package, scheduler SSSD, env-var quoting)

- **SSSD `ldap_uri = ldapi:///` requires SASL EXTERNAL config that we don't supply** (v0.19.2 live-validation finding). The first cut of the scheduler SSSD block used `ldap_uri = ldapi:///` figuring that since slapd listens on the UNIX socket and SSSD is a local client, no TCP loopback was needed. `sssctl domain-status` reported `Online status: Offline / LDAP: not connected` and `getent passwd <ldap-user>` returned nothing even though `ldapsearch -H ldapi:///` worked from the same shell. Reason: SSSD's LDAP provider can speak LDAPI but only with `ldap_sasl_mech = EXTERNAL` (autobind via SO_PEERCRED), which we didn't configure ‚Äî it defaulted to anonymous-bind-over-LDAPI which slapd rejects. Fix: use `ldap_uri = ldap://127.0.0.1`. slapd already listens on `0.0.0.0:389` (and the NSG default `AllowVnetInBound` covers it), and loopback is never wire-visible so the cleartext concern is moot. Same anonymous-bind pattern as the login + compute SSSD configs (which already use `ldap://${AZCLUSTER_SCHEDULER}`).

- **Env vars whose value contains whitespace MUST be quoted in env files sourced under `set -e`** (v0.19.2 fix). `cloud-init/{scheduler,login,compute}.yaml.tmpl` previously emitted `EXTRA_PACKAGES={{EXTRA_PACKAGES}}` in install.env / cluster.env. With `--extra-package git-lfs --extra-package python3.12-venv`, the template expands to `EXTRA_PACKAGES=git-lfs python3.12-venv`. When `install-scheduler.sh` does `. /etc/azcluster/install.env` under `set -euo pipefail`, bash parses that line as a SCOPED ENV ASSIGNMENT for one command, not a global export ‚Äî so it tries to run `python3.12-venv` as a command with `EXTRA_PACKAGES=git-lfs` in its env. That exits 127 (`command not found`), `set -e` kills the source, and `install-scheduler.sh` aborts at line 1 with no useful error in `/var/log/azcluster/install.log` (the log redirect happens AFTER the source). Fix: emit `EXTRA_PACKAGES="{{EXTRA_PACKAGES}}"` (quoted). General rule: any cloud-init template variable that may contain whitespace MUST be emitted quoted in env files.

- **Defensive `az login --identity` in scheduler bootstrap** (v0.19.2 fix). The scheduler cloud-init unconditionally ran `az login --identity --client-id {{UAI_CLIENT_ID}}`. Under `--no-monitoring --no-accounting` test mode the UAI has no subscription-level RBAC, so the login returned non-zero, `set -e` killed `install-scheduler.sh`, and the cluster never finished bootstrap. The line was vestigial ‚Äî scheduler doesn't invoke any `az` subcommands later (monitoring/accounting paths handle their own login). Fix: append `|| true 2>&1`. Pattern: any `az` call in cloud-init that isn't followed by a hard dependency MUST be defensive.

- **Per-user enroot CACHE path uses runtime `$(id -u)` expansion**, not a literal cloud-init substitution. The line in `/etc/enroot/enroot.conf` reads `ENROOT_CACHE_PATH /var/lib/enroot-data/cache/user-$(id -u)` ‚Äî enroot evaluates the `$()` at container-import time as the invoking user, so each user gets `/var/lib/enroot-data/cache/user-20001`, `/var/lib/enroot-data/cache/user-1000`, etc. The parent dir `/var/lib/enroot-data/cache/` is 1777-sticky (matches the v0.13.5 RAID-0 fix) so the subdirs are auto-created on first import without root involvement. Do NOT bake a fixed UID into the path at cloud-init time ‚Äî that would defeat the per-user isolation.

- **NGC container import on /mnt/nvme NVMe RAID takes ~20 min for the FIRST ~20 GB image, not "seconds"** (v0.19.2 live-finding). The v0.13.5 gotcha claimed "first-time import of a ~20 GB image completes in tens of seconds, not minutes" ‚Äî that's wrong; the NGC pull dominates and is bottlenecked by NGC's CDN + the extraction-to-squashfs step. Subsequent imports of the SAME image complete in seconds (squashfs cached). Don't size benchmark wall-clocks assuming sub-minute imports.

- **Pyxis `--container-name` is per-node, not per-cluster** (v0.19.2 live-finding). A pre-import step like `srun --container-image=X --container-name=Y --nodes=1 --ntasks=1 bash -c imported` imports the container ONLY on the node Slurm picks. A subsequent `srun --container-name=Y --nodes=2 ...` then fails on the second node with `a container with name "pyxis_Y" does not exist, and --container-image is not set`. Pattern that works: import on every node first (`srun --nodes=N --ntasks=N --ntasks-per-node=1 --container-image=X --container-name=Y bash -c true`), THEN run the multi-node step that references the name. Or: pass `--container-image=X` to every srun and accept the per-step container creation overhead.

- **bare-metal PMIx env vs container PMIx env**. Bare-metal: `srun --mpi=pmix bash -c 'echo $PMIX_RANK'` correctly prints rank 0..N-1 across ranks (PMIx populates PMIX_RANK in the stepd-injected env, no PMIX_SIZE). `srun --mpi=pmi2 bash -c 'echo $PMI_RANK / $PMI_SIZE'` prints both. Container: needs the `50-slurm-pmi.sh` enroot hook to copy `PMIX_*` / `PMI_*` env into the container at start (v0.13.6 wired this). When the hook is present but the import races (per-node container name semantics, see above), per-rank env can be partially populated. For containerised multi-node work, prefer `--mpi=pmi2` + a torchrun-style rendezvous over TCP if the workload supports it ‚Äî that's what the v0.19.1 walkthrough's container path uses.

## v0.19.3 gotchas (sacctmgr auto-registration + dual-layer uid cache)

- **`slurmctld` assoc_mgr caches username‚Üíuid for process lifetime; `scontrol reconfigure` does NOT invalidate it** (v0.19.3 live-finding). When you delete LDAP user `paul` (uid 20007) and re-add with the same username (uid 20009), `slurmctld` keeps resolving `paul` ‚Üí 20007 until the process is restarted. `sbatch` then either accepts but the job runs as the wrong uid, or fails with `Invalid account or account/partition combination specified` if you've also cycled the sacctmgr account. The only way to refresh assoc_mgr's uid cache is `sudo systemctl restart slurmctld` (the process re-runs `getpwnam(paul)` on startup). Cost: ~3 s; zero impact on running jobs (slurmd retains workload state across slurmctld restart). Pattern for any CLI that mutates LDAP usernames+sacctmgr together: flush scheduler-side SSSD (`sss_cache -u <user> && sss_cache -E`) BEFORE restarting slurmctld, otherwise the restart re-resolves against a stale SSSD cache and the bug persists. `v0.19.3` `crates/azcluster-cli/src/user.rs` `build_sacctmgr_{add,remove}_cmd` encode this exact sequence; tests assert ordering (sss_cache before restart).
- **Slurm `sacctmgr` duplicate-account stderr uses capital-A "Already existing", NOT "already exists"** (v0.19.3 live-finding, on Slurm 25.11). Idempotency case-matches must accept both casings. Exact stderr: `" Already existing account paul on cluster v193a"` (leading space). The `sacctmgr_run` helper in `user.rs` matches `*"already exists"*|*"Already existing"*|*"Nothing new added"*|*"Nothing deleted"*` as idempotent (return 0).
- **Slurm `sacctmgr` transient errors** when slurmdbd is mid-bootstrap or the cluster hasn't yet been added to the accounting DB: `"Connection refused"`, `"Unable to contact slurmdbd"`, `"slurmdbd is down"`, `"cluster has not been added"`, `"is not registered"`. v0.19.3 retries up to 12 √ó 5 s = 60 s on these. Fatal everything else.
- **Re-adding a removed LDAP user reuses the username but gets a NEW uidNumber** (LDAP `cn=uidNext` counter is monotonic). The previous home dir `/shared/home/<user>` on ANF is owned by the OLD uid; the new uid cannot read it (`Could not chdir to home directory ... Permission denied`). v0.19.3 does NOT auto-`chown` the home dir on re-add ‚Äî operator must `sudo chown -R <user>: /shared/home/<user>` manually if home contents need preserving. Documented footgun, not a bug; the trade-off is that we don't risk silently giving the new uid access to files the old uid created (e.g. `.ssh/authorized_keys` could otherwise be hijacked).

## v0.19.4 gotchas (UCX_TLS=tcp breaks container HPC-X/MPI)

- **`UCX_TLS=tcp` in `/etc/enroot/environ.d/50-nccl.env` hangs every `*_mpi` NCCL test and every HPC-X-based MPI workload inside Pyxis containers** (v0.19.4 fix). The env file was added in v0.13.5 alongside the per-container NCCL tunings; `UCX_TLS=tcp` was carried over from a TCP-only debug config and never re-examined because the v0.13.5/v0.13.8 validation only exercised the bare `all_reduce_perf` binary (which uses NCCL's OOB bootstrap + IBext plugin, NOT UCX). Once you run the `_mpi` variant (`/usr/local/bin/all_reduce_perf_mpi` shipped in NGC NeMo/PyTorch images), HPC-X's UCX kicks in for the MPI bootstrap, sees `UCX_TLS=tcp`, and tries to find TCP transport on every `UCX_NET_DEVICES` interface. `mlx5_ib0..7` are IB-only ‚Äî no TCP socket ‚Äî so UCX errors `ucp_context.c:1582 UCX ERROR no usable transports/devices (asked tcp on network:mlx5_ib0:1,mlx5_ib1:1,...)` from every rank. OMPI then falls back to its TCP BTL, which routes correctly via eth0 but hangs in `btl_tcp_endpoint.c:625:mca_btl_tcp_endpoint_recv_connect_ack: received unexpected process identifier [[X,Y],N]` (process IDs collide because OMPI's pid-derived rank IDs aren't stable across the Pyxis container's REMAP_ROOT namespace). ## v0.21.1 gotchas (azcluster scp + login rebind fast-path)

- **`azcluster login --subscription <id>` fast-path** (v0.21.1). Management-scope access tokens (`aud=https://management.azure.com/`) are principal-scoped, not subscription-scoped ‚Äî so rebinding the cache entry under a new sub id is a 6 ms file mutation, no re-auth needed. `auth/token_provider::try_rebind_cached(target_sub_id, tenant_filter)` returns `Ok(Some)` if the cache holds an account for any visible sub under the same principal (filtered by tenant when provided), refreshes via `refresh_with_token` only when within the 5-min expiry window, then re-keys the entry. `login()` consults this before launching any interactive flow. Critical workaround for **Microsoft tenants where Conditional Access blocks device-code flow** (AADSTS53003): operator runs `azcluster login` once interactively in a browser-equipped shell, then `azcluster login --subscription <id>` works from any TTY (including SSH-without-X11). DO NOT remove the `match` over `Option::is_none_or` ‚Äî workspace MSRV clippy gate (`clippy::incompatible_msrv`) fails on `is_none_or` (stabilised in 1.82, MSRV is older).
- **`azcluster scp` syntax mirrors real `scp`, not `cp`** (v0.21.1). The first-colon-before-first-slash rule decides remote vs local: `./a:b` is local (slash before colon), `a:b/c` is remote node `a` path `b/c` (colon before slash). Empty node defaults to `login` (`:/shared/foo`). Other accepted nodes: `scheduler`, any compute hostname (e.g. `vmss-<cluster>-<pool>NNNNNN` ‚Äî Azure VNet DNS auto-registers these because cloud-init sets the hostname before Slurm config, and the login VM resolves them via the VNet DNS suffix). A single invocation may target only ONE remote node; remote-to-remote disallowed (operator must `ssh && scp` for that). Enforced in `scp()` at parse time.
- **scp route resolution** lives in `resolve_scp_route(state, node, use_bastion) -> (proxy_target: Option<String>, jump_login_host: Option<String>, dest_host: String)`. Six combinations:
  - `login` + bastion: `--target login`, no jump, dest `127.0.0.1`.
  - `login` + no bastion: no proxy, no jump, dest = `login_public_ip`.
  - `scheduler` + bastion: `--target scheduler` (DIRECT bastion‚Üíscheduler VM, no login hop), no jump, dest = `scheduler_private_ip`. Mirrors `azcluster ssh --scheduler` semantics ‚Äî Bastion can tunnel to any VM in the VNet by ARM resource id.
  - `scheduler` + no bastion: no proxy, jump = `azureuser@<login_public_ip>`, dest = `scheduler_private_ip`.
  - `<compute>` + bastion: `--target login`, jump = `azureuser@127.0.0.1`, dest = compute hostname. Compute ALWAYS traverses login (Bastion has no VNet DNS resolver for VMSS instance hostnames; resolution happens on login via VNet DNS).
  - `<compute>` + no bastion: no proxy, jump = `azureuser@<login_public_ip>`, dest = compute hostname.
- **scp command surface**: dispatches to OS `scp` (not implemented in Rust) with composed `-o ProxyCommand=...` / `-o ProxyJump=...` flags. The `bastion-proxy` invocation is `${self_exe_path} bastion-proxy --cluster <name> --target {login|scheduler}` ‚Äî same hidden subcommand as `azcluster ssh`. Unit-tested without Azure via `resolve_scp_route` + `parse_scp_path` (11 tests in `crates/azcluster-cli/src/main.rs::tests::{parse_scp_*,resolve_scp_*}`).
- **Auto-route helper reused**: `should_use_bastion(state, no_bastion_flag) = !no_bastion_flag && state.bastion_enabled && state.login_public_ip.is_none()`. Same function as `ssh`/`exec`/`tunnel` ‚Äî DRY across all four commands.

## v0.21.0 native Azure Bastion (no plugin)

- **`--bastion` flag on `azcluster deploy`** provisions Standard SKU Bastion + Standard Static public IP into a new `AzureBastionSubnet = cidrSubnet(vnetAddressPrefix, 26, 1)` (= `10.42.0.64/26`). `enableTunneling: true` is required for the token-based WS bridge; deploys without it cannot mint tunnel tokens (HTTP 400). Adds ~3-5 min wall-clock.
- **`AzureBastionSubnet` is a Microsoft-reserved name.** Bicep `name: 'AzureBastionSubnet'` MUST match exactly (case-sensitive); ARM rejects any other name with `BastionSubnetMustBeNamedAzureBastionSubnet`. Minimum /26.
- **`AllowAzureLoadBalancerInBound` on the cluster's internal NSG is sufficient** for Bastion's intra-VNet SSH-to-target traffic (port 22 from Bastion subnet ‚Üí login/scheduler/compute subnets). The Bastion-managed `bastionNetworkSecurityGroup` handles the public-facing rules (HTTPS 443 from Internet + Azure Bastion service tags). No NSG edits needed on the cluster side.
- **Tunnel-token API is `POST https://<bastion-fqdn>/api/tokens` with `application/x-www-form-urlencoded` body** ‚Äî NOT JSON, NOT bearer-auth. Form fields: `resourceId` (full target VM ARM resource id), `protocol=tcptunnel`, `workloadHostPort=22`, `aztoken=<operator-AAD-bearer>`. Optional: `token=<previous-token>` for refresh, `X-Node-Id` HTTP header for sticky session. Response JSON: `{authToken, nodeId}`. Token TTL ~3 min. The standard `Authorization: Bearer` header DOES NOT WORK on this endpoint ‚Äî must be in the form body as `aztoken`.
- **WS endpoint is `wss://<bastion-fqdn>/webtunnelv2/<authToken>?X-Node-Id=<nodeId>`**. **Do NOT use `tokio-tungstenite` (or any strict-RFC WS client) for this ‚Äî Azure Bastion's 101 Switching Protocols response omits the `Sec-WebSocket-Accept` header.** `tokio-tungstenite` rejects with `WebSocketProtocolError(InvalidHeader)`. Use hand-rolled WS framing on top of `tokio-rustls` (live in `crates/azcluster-cli/src/bastion/tunnel.rs`): client‚Üíserver frames MUST be masked (4-byte XOR key); server‚Üíclient frames are unmasked; payload-length encoding 7-bit / 7+16-bit / 7+64-bit; opcodes BINARY=0x02 (data), CLOSE=0x08, PING=0x09, PONG=0x0A.
- **Hidden `azcluster bastion-proxy` subcommand** is the bridge entry point. Designed exclusively as a stdio consumer for `ssh -o ProxyCommand="azcluster bastion-proxy --cluster N --target {login|scheduler}"`. Reads `ClusterState` for `bastion_dns_name`, mints token, opens WS, pipes stdio‚ÜîWS. Cleans up token on shutdown via `DELETE /api/tokens/<token>` (best-effort ‚Äî Bastion auto-reaps after TTL anyway). `--target scheduler` connects DIRECTLY to scheduler VM (Bastion can tunnel to any VM in the VNet by ARM resource id); does NOT need a `-J` jump through login. VM resource ID derived from cluster state by convention: `/subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/vm-<cluster>-{login|scheduler}`.
- **Auto-routing in `ssh`/`tunnel`/`exec`**: `should_use_bastion(state, no_bastion_flag) = !no_bastion_flag && state.bastion_enabled && state.login_public_ip.is_none()`. When true, the spawned `ssh` gets `-o ProxyCommand="<self> bastion-proxy ..."` and dials `azureuser@127.0.0.1` (login) or `azureuser@<scheduler-private-ip>` (scheduler). `--no-bastion` flag on each command opts out and surfaces the legacy "no public IP" error ‚Äî useful if Bastion is misbehaving and operator wants the original code path.
- **OAuth2 access token must have audience `https://management.azure.com/`** (the same management-scope token used for ARM REST). Bastion accepts this same audience for tunnel-token auth; no separate scope/audience needed. Token is fetched via `get_access_token()` which reuses the cached token from `azcluster login`.
- **CLI is otherwise pure-sync; bastion-proxy is the ONE async subcommand.** Uses `tokio::runtime::Builder::new_current_thread().enable_all().build()?.block_on(...)` to isolate the async runtime to just that subcommand. Do not async-ify the rest of the CLI to accommodate this.
- **dead_code allow on `run_tcp_listener` + `handle_one`** in `bastion/tunnel.rs` is intentional ‚Äî the TCP-listener mode (local-port-forward over Bastion) is not yet wired into a CLI surface but the code is ready to ship when needed.

## v0.21.3 gotchas (LDAP-user UX + jump identity)

- **Jump identity MUST match destination identity for `ssh`/`exec`/`scp` `--user <ldap-user>`** (v0.21.3 design). Earlier prototype used `jump_user = state.admin_username` and `connect_user = args.user.unwrap_or(admin)`, on the theory that admin (azureuser) always has the operator's key on login via Bicep `linuxConfiguration.ssh.publicKeys`. That's true for the deployer's laptop but BREAKS for any operator whose pubkey was only enrolled via `azcluster user add <user> --ssh-key ...` (i.e. the LDAP path) ‚Äî admin's authorized_keys wouldn't contain their key, so the ProxyJump hop fails with `Permission denied (publickey)`. Fix: `jump_user = connect_user` so a single identity authenticates at every hop. LDAP user `alice` ‚Üí login (SSSD resolves her pubkey) ‚Üí compute (SSSD resolves it again) all using the same `-i` key. Operator-as-admin keeps working because azureuser's authorized_keys has the deployer's pubkey AND `--user` defaults to admin.

- **`--scheduler --user <ldap-user>` does NOT work, by design**: the scheduler hosts the slapd LDAP server itself and runs NO SSSD client (login + compute are clients; scheduler is the source of truth). LDAP users therefore have no `getpwnam` resolution on scheduler, no `/shared/home/<user>`, and `sshd` rejects the key lookup. Documented as a known limitation in CHANGELOG + README "First user" section. Not worth adding SSSD to scheduler ‚Äî job submission is from login (which has it), scheduler is admin-only by convention. If a future use case needs scheduler shell access for LDAP users, the fix is to add an SSSD block to `cloud-init/scheduler.yaml.tmpl` mirroring login's (pointing at `ldap://127.0.0.1`), and ensure pam_mkhomedir runs in scheduler's pam stack.

- **`--host <hostname>` resolves via VNet DNS on login**, not on the operator's laptop. Azure VNet's auto-DNS registers VMSS instance hostnames (`<cluster>-<pool>-<NNNN>`) into the VNet's private DNS zone, but only resolvers inside the VNet see them ‚Äî `dig +short <cluster>-cpu-0001` from the operator's laptop returns nothing. The `--host` flag therefore always uses a ProxyJump through login (or bastion-proxy ‚Üí login), even when login has a public IP. Don't try to "optimize" by dialing the compute hostname directly from the operator ‚Äî there's no DNS to resolve it.

- **`azcluster exec` does NOT forward the SSH agent by default** (v0.21.3 design). Earlier `ssh` had `-A` baked in; new `exec` makes it opt-in via `--forward-agent` / `-A`. Rationale: `exec` is typically used for one-shot commands where the remote shell shouldn't need to ssh anywhere else; defaulting to agent-forward is unaudited and potentially confusing for ops workflows. If you need nested ssh from within an `exec` payload (rare: usually a sign the workload belongs in a script on `/shared`), add `-A` explicitly.

## v0.21.2 gotchas (hygiene trio)

- **Cloud-init `runcmd` 0077 umask + `set -euo pipefail` mid-script abort silently breaks `/etc/slurm/*.conf` perms** (v0.21.2 fix; sibling of v0.13.9). The chmod 0644 block near the end of `install-scheduler.sh` (and analogous plugstack.conf chmod in login/compute) only runs when the script reaches it. Concrete reproducer: deploying with `--azcluster-version vX.Y.Z` BEFORE that release tag has been published ‚Üí `curl -fsSL` on the spank_pyxis.so download returns 404 ‚Üí `-f` flag aborts the script at line ~229 ‚Üí chmod at line ~241 never runs ‚Üí slurm.conf stays 0600 ‚Üí every non-root `srun`/`sinfo` dies with `Permission denied`. Fix: `trap 'chmod 0644 /etc/slurm/<files> 2>/dev/null || true' EXIT` immediately after `set -euo pipefail`, so chmod runs regardless of where the script aborts. Applied to all three install scripts. Does NOT fix the underlying tarball-404 race ‚Äî operator still has a half-installed cluster ‚Äî but prevents the most common downstream UX failure.

- **`azcluster timings` recursion broken after v0.20.0 native ARM REST switch** (v0.21.2 fix). Pre-v0.20.0 the CLI shelled out to `az deployment {sub,group} operation list` which always populated top-level `targetResource.resourceGroup`/`resourceType`/`resourceName`. Native ARM REST `GET .../deployments/<name>/operations?api-version=2024-03-01` ALSO populates them for resource-typed ops, BUT for sub-scope nested `Microsoft.Resources/deployments` entries `resourceGroup` is empty (sub-scope deployments don't live in any RG). The v0.13.x recursion gate `if rtype == "Microsoft.Resources/deployments" && !rg.is_empty()` silently skipped every nested module ‚Üí captured ops collapsed to just the root (`{count: 1, total: -0}`). Plus float quirk: `iter().filter(...).sum::<f64>()` over empty serialised as `-0` via serde_json. Fix: (1) new `parse_target_id` helper extracts `(rg, type, name)` from the ARM ID string for both rg-scoped and sub-scope shapes, used as fallback when explicit fields are missing; (2) `collect_sub_operations` recurses via `list_subscription_deployment_operations` when parsed rg is empty (sub-scope nested) vs `list_resource_group_deployment_operations` otherwise; (3) `let total = if total == 0.0 { 0.0 } else { total };` normalises `-0.0`. 3 new unit tests in `timings::tests`. v19walk (pre-refactor) captured 35 ops / 3063s; expect similar post-fix.

- **`azcluster status` bootstrap probe** showed stale `curl 404` from `install.log` tail when bootstrap had failed; bypassed bastion auto-route entirely (v0.21.2 fix). Now: (a) checks `/var/log/azcluster/ready` first (written at end of every install script after all steps complete) ‚Üí prints `READY` instead of misleading log tail; otherwise falls back to install.log tail with an `in-progress | last log: ...` prefix; (b) honors `should_use_bastion` + `bastion-proxy` ssh `ProxyCommand` so probe works for `--bastion` clusters with no public IP; (c) probe enabled whenever login is reachable (`login_public_ip.is_some() || (bastion_enabled && bastion_dns_name.is_some())`).

## v0.22.0 gotchas (Key Vault + stateless CLI)

- **KV name MUST be derived deterministically client-side, threaded as a Bicep param, AND stamped into the `azcluster:kv` RG tag** (v0.22.0 design). Pattern: `kv-azc-<8-hex-blake3(sub_id|name|location)>`. Hard 24-char limit on Key Vault names ‚Äî 8 hex chars max. Don't try to derive it Bicep-side from a hash of the deployment id; ARM doesn't expose a hash function and `uniqueString()` collides too easily with other resources in the same RG. The `azcluster:kv` RG tag is the **only** way `azcluster list` and `azcluster <subcommand> --no-cache` can find the vault from the subscription alone ‚Äî without it the CLI would have to enumerate every KV in every RG, which is slow + scope-leaky.
- **ARM RG `PATCH` is 200-sync, not 202-async** (v0.22.0 confirmed). `PATCH /subscriptions/{}/resourcegroups/{name}?api-version=2024-03-01` with a tags-only body returns 200 with the updated RG envelope inline. No `Azure-AsyncOperation` header to poll. Use `arm_client::patch_and_wait` which handles both shapes ‚Äî it short-circuits on 200 and only polls on 202. Don't roll your own polling for RG tag PATCHes.
- **Key Vault tokens are a DIFFERENT audience from ARM management tokens** (v0.22.0 design). Vault: `https://vault.azure.net/.default`. ARM: `https://management.azure.com/.default`. The vault endpoint server-enforces audience and rejects ARM tokens with HTTP 401 `WWW-Authenticate: Bearer authorization_uri="..."` ‚Äî silent failure if you forget. `get_vault_token()` mints + caches vault-scope tokens independently of `get_access_token()`. Both reuse the same cached OAuth2 principal at `~/.azure/azcli_tokens.json` so the user only authenticates once.
- **Two KV secrets per cluster, both JSON-encoded**: `cluster-manifest` (full `ClusterState`) and `secrets-bundle` (LDAP admin password + MySQL admin password if accounting + `admin_ssh_public_key` + `admin_ssh_private_key`, all OpenSSH PEM). Don't split into more secrets ‚Äî Key Vault charges per secret and the per-secret 25 KiB limit is comfortably above what we need (~3 KiB combined). `ClusterSecrets::admin_ssh_{public,private}_key` are `Option<String>` with `#[serde(default)]` so deploys with pre-existing secrets (resume path) don't break on schema evolution.
- **Admin SSH keypair is generated fresh per deploy, NEVER read from `~/.ssh/*`** (v0.22.0 invariant). User insists: "create a new one - don't take the one from the local computer". Use `ssh-key` 0.6 with features `["ed25519", "alloc"]` ‚Äî note `alloc` is required for `to_openssh()` (PEM serialisation), the default feature set only exposes raw key bytes. The private key materialises lazily to `~/.azcluster/keys/<cluster>` (file mode `0600`, parent dir mode `0700`) on first `azcluster ssh/exec/scp/tunnel` ‚Äî never written by `azcluster deploy` itself.
- **`fetch_admin_private_key(name)`** checks `~/.azcluster/keys/<name>` first; if absent, calls `resolve_cluster(name)?` + `KeyVaultClient::get_secret("secrets-bundle")` + parses + writes. Important: do NOT make `azcluster deploy` itself write the private key locally ‚Äî operators may deploy from one machine and run `azcluster ssh` from another, and the materialise-on-use pattern keeps both paths uniform.
- **All 16 `ClusterState::load(...)` sites in `main.rs` were swapped to `resolve_cluster(...)` via a mix of ast-grep (15 trivial `?`-form sites) + 2 manual edits (`status` uses `.ok()` form, `delete` uses `match {‚Ä¶}` form for fallback). `delete` retains its `ClusterState::load` fallback path so pre-v0.22 clusters (no `azcluster:managed` tag) can still be torn down; every other subcommand hard-errors on resolve failure.
- **All 6 `if let Some(id) = &args.identity { -i ... }` blocks** (ssh, tunnel, exec, scp, logs, validate) were replaced atomically with `let identity = resolve_identity(args.identity.as_deref(), &args.name)?; cmd.args(["-i", &identity.display().to_string()]);` ‚Äî single ast-grep run, idempotent. User-supplied `-i` still wins; default is the lazily-materialised admin key.
- **Global `--no-cache` flag** is stored in a `static AtomicBool NO_CACHE` set in `main()` before dispatch. Plumbing it through every handler signature would have touched 16 sites; the atomic is read once per resolution in `resolve_cluster()`. Defensible because the flag is genuinely process-global ‚Äî there's no scenario where one subcommand wants cache-on and another cache-off in the same invocation.
- **`derive_kv_name` uses blake3** (not sha256) for two reasons: (1) faster, (2) `blake3` is already a transitive dep of `rustls` via `ring` so it adds nothing to the binary. We take 8 hex chars (32 bits) from the 256-bit digest ‚Äî collision probability across a single subscription's RGs is negligible (operator would need ~65k clusters to hit a single collision at 50% probability, and the failure mode is a deploy-time error not a silent data swap because Bicep ARM rejects duplicate KV names).
- **Clean break, no migration.** Pre-v0.22 clusters lack the `azcluster:managed` tag ‚Üí `list_resource_groups_by_tag` returns empty ‚Üí `resolve_cluster` errors. Operator must `azcluster delete <name>` (which falls back to `ClusterState::load` for tear-down) then re-deploy on v0.22. Don't add a v0.21.x compatibility shim ‚Äî it doubles the resolution surface area for one release of grace value.
- **`bicep/cluster.bicep` lines 240-242** output `keyVaultName`, `keyVaultUri`, `keyVaultId` from the keyvault module. `bicep/main.bicep` plumbs the `keyVaultName` Bicep param through to the cluster module (computed CLI-side, not derived in Bicep ‚Äî see "deterministic" gotcha above). `bicep/main.json` MUST be regenerated and committed after editing any of these (CI's `bicep` job diffs against the committed copy and fails on drift).
- **RBAC propagation: same 5-10 min lag as Monitoring Data Reader** (v0.13.0 carry-forward). The Bicep `keyvault.bicep` module grants the deployer Key Vault Secrets Officer (`b86a8fe4-44ce-4948-aee5-eccb2c155cd7`) on the vault scope. Immediately after deploy, `KeyVaultClient::set_secret` may return HTTP 403 `Forbidden` for ~30s-2min until the role assignment propagates. The CLI does NOT currently retry this (single-PUT, fails loudly) ‚Äî adding 60s of retry with 5s backoff is a backlog item if v0.22 live-validation reveals flakiness. Workaround: re-run `azcluster deploy` (now idempotent via resume path), which on the second invocation will catch the secrets push.

## v0.22.2+ gotchas (azcluster purge-kv + ARM bodyless POST Content-Length)

- **Azure ARM frontend rejects bodyless POST with `HTTP 411 Length Required`** (live-caught on the first `azcluster purge-kv` invocation against `kv-azc-90188ada` in southafricanorth). `reqwest::blocking::Client` does NOT add `Content-Length: 0` on POSTs with no body. Azure's frontend (the layer in front of `Microsoft.KeyVault`, likely IIS-flavoured per the HTML error shape `<HTML><HEAD><TITLE>Length Required</TITLE>`) rejects with 411. Fix in `ArmClient::purge_deleted_vault`: append `.header(reqwest::header::CONTENT_LENGTH, "0")` to the builder. **General rule: every ARM POST that does not include a body MUST set `Content-Length: 0` explicitly.** The existing `post()` helper accepts `body: Option<Value>` and serialises `None` as no body ‚Äî same bug would surface if any caller passes `None`; currently no caller does, but if one is added the helper itself should set the header.

- **`Microsoft.KeyVault/locations/{loc}/deletedVaults/{name}/purge` is a 202 long-running operation** (~6 minutes observed for a `kv-azc-*` vault with ~3 secrets in `southafricanorth`). The CLI correctly delegates the 202 case to `wait_for_async_operation` (10s polling, 30min cap). Bulk `--all` is serial, so multi-vault runs can exceed any short shell `timeout` an operator wraps around the call. The interactive `'yes'` gate + the `Permanently purge ...` warning set sufficient expectations; no further UX changes planned unless complaints surface.

- **`Microsoft.KeyVault/deletedVaults` list endpoint is subscription-wide, not location-scoped.** `GET /subscriptions/{sub}/providers/Microsoft.KeyVault/deletedVaults?api-version=2023-07-01` returns every soft-deleted vault in the subscription regardless of region ‚Äî more useful than the location-scoped variant because operators rarely remember which region a torn-down cluster was in. CLI filters client-side via `filter_purge_kv_candidates`: (1) `kv-azc-` prefix to scope to azcluster-managed vaults, (2) optional exact name match when `--name` derives it from sub+name+location, (3) optional location (case-insensitive).

- **`kv-azc-*` prefix discipline is the ONLY safety net protecting non-azcluster vaults from accidental `purge-kv --all`.** If `crypto::derive_kv_name` ever changes prefix, MUST grep `kv-azc-` in `main.rs` (`filter_purge_kv_candidates`) and AGENTS.md gotchas and update both. Hardcoded in `main.rs`; deliberately not configurable.

## v0.22.4 gotchas (live deploy-progress recursion + ARM op `resourceGroup` absent)

- **ARM deployment-operation envelopes return `properties.targetResource` with only `id`, `resourceType`, `resourceName` ‚Äî `resourceGroup` is NOT populated.** Live-confirmed v0.22.4 against `azcluster-v224a-20260526-144125`: every targetResource is exactly those 3-4 fields (the cluster-level nested deployment op also has `apiVersion`). Any code that needs the RG to recurse into rg-scoped nested deployments MUST parse it out of `id`'s path segment `/resourceGroups/<rg>/...`. Same trap that bit `timings::parse_target_id` in v0.21.2 ‚Äî now also bit the live-progress recursive walker in `arm/client.rs::nested_module_target` until v0.22.4 added the same id-path fallback. **Rule for any future ARM walker code**: never trust `targetResource.resourceGroup`; always have an `id`-parsing fallback. Three unit tests in `arm/client.rs::tests` pin the contract: `nested_module_target_falls_back_to_id_when_resource_group_field_absent`, `_sub_scope_has_empty_rg`, `_non_deployment_returns_none`.

- **`tee` + `&` background pipelines from the agent's `bash` tool get reaped when the spawning subshell exits** (v0.22.4 lesson). `./azcluster deploy ... 2>&1 | tee log &` does NOT survive the bash tool's per-call subshell ‚Äî the orphaned pipeline dies on next return. Symptoms: `ps -p <pid>` shows nothing, log file truncated mid-deploy, ARM deployment completes successfully on its own. For long-running CLI invocations that need stdout capture across multiple agent turns, use the `interactive_bash` (tmux) tool with `send-keys` to a detached session. The deploy itself is unaffected (ARM keeps running) ‚Äî only the local progress display is lost. Run `azcluster resume --name <name>` afterwards to re-attach via the pending marker.

## v0.22.1 gotchas (deploy keypair race + ssh `-J` identity propagation)

- **`finalize_deploy()` rebuilds `ClusterSecrets` from in-memory `existing_secrets`, which is `None` on fresh deploys** (v0.22.1 fix). `deploy()` (main.rs ~1003) generates the admin ed25519 keypair, persists it via `secrets.save(&args.name)`, then submits ARM. After ARM completes, `finalize_deploy()` runs and reconstructs `ClusterSecrets { admin_ssh_public_key, admin_ssh_private_key, ldap_admin_password, mysql_admin_password }` from `existing_secrets` (the snapshot captured BEFORE the keypair was generated). On a fresh deploy `existing_secrets` is `None`, so the keypair fields default to empty strings; the subsequent `secrets.save()` clobbers the on-disk file written at line 1003, and the KV upload pushes empty keys. Effect: `azcluster ssh/exec` fails with `secrets-bundle in vault '<kv>' has no admin_ssh_private_key`. Re-running deploy then generates a FRESH keypair (local secrets file now empty) but ARM rejects with `PropertyChangeNotAllowed` because `osProfile.linuxConfiguration.ssh.publicKeys` is immutable on existing VMs. Fix: `finalize_deploy()` first calls `ClusterSecrets::load_optional(&args.name)` to pick up the freshly-persisted on-disk copy, and only falls back to `existing_secrets` when the disk copy is missing. Sibling lesson: any post-ARM step that mutates state MUST load from canonical storage (disk or KV), NEVER from a function-local snapshot captured before the deploy.

- **OpenSSH `-J <jump>` does NOT propagate `-i <identity>` to the jump-hop ssh** (v0.22.1 fix). The inner `ssh -W %h:%p <jump>` that OpenSSH spawns to handle `-J` falls back to the agent / `~/.ssh/id_*`, ignoring the outer `-i`. v0.21.x worked because the admin SSH key was a copy of the operator's `~/.ssh/id_ed25519` (already in the agent). v0.22 generates the admin keypair fresh per deploy and stores it ONLY in Key Vault + `~/.azcluster/keys/<cluster>` ‚Äî no agent, so jump hops fail with `Permission denied (publickey)`. Diagnostic: `ssh -i ~/.azcluster/keys/X -J azureuser@<login> azureuser@<scheduler> hostname` fails; `ssh -i ~/.azcluster/keys/X -o ProxyCommand="ssh -W %h:%p -i ~/.azcluster/keys/X -o IdentitiesOnly=yes azureuser@<login>" azureuser@<scheduler> hostname` succeeds. Fix: helper `add_ssh_jump_with_identity(cmd, identity, jump_target)` emits `-o ProxyCommand="ssh -W %h:%p -i <key> -o IdentitiesOnly=yes -o StrictHostKeyChecking=accept-new -o UserKnownHostsFile=/dev/null -o LogLevel=ERROR <jump>"` (NOT `-J`). Applied to all 8 jump sites: `ssh()` (√ó3 ‚Äî scheduler/host/login-bastion), `exec()` (√ó3 ‚Äî same), `scp()` (√ó1), `bootstrap_probe()` (√ó1). `IdentitiesOnly=yes` is critical: without it, ssh tries the agent first and gets stuck if any unrelated keys are loaded with the same comment field. Generalisable rule: **for any Rust CLI that drives ssh with non-default identities through a jump host, NEVER use `-J`; always compose `-o ProxyCommand` with explicit `-i` on the inner ssh.**

- **`bootstrap_probe()` was running without `-i`**, silently relying on the operator's ssh-agent and returning `ERR (Permission denied)` on every status call from a fresh laptop. v0.22.1 makes it resolve the admin identity via `resolve_identity(None, &state.name)` and inject `-i <key> -o IdentitiesOnly=yes` into the outer ssh AND use the new jump-with-identity helper. If the identity can't be resolved (e.g. operator hasn't run `azcluster login` or KV RBAC isn't propagated), the probe prints `SKIP (no admin key: <reason>)` and `status` continues ‚Äî best-effort by design.
