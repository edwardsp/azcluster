# AGENTS.md

Instructions for AI agents working on this repository. Keep this file current.

## Documentation Discipline (MANDATORY)

Every change MUST update three artifacts in lockstep:

1. **`CHANGELOG.md`** — add entry under `## [Unreleased]` for every user-visible or operator-visible change. On tag, rename `Unreleased` to the new version with the release date, and open a fresh `Unreleased` section. Categories: `Added`, `Changed`, `Fixed`, `Removed`, `Deprecated`, `Security`.
2. **`README.md`** — keep the top-of-file pointer line current (`**Current release:**`), keep `Quickstart`, `Deploy options`, `Operator commands`, `Architecture`, `Repo layout` honest. If a flag/command/default changes, update the relevant table or command example. **Do not** add release-note "status blocks" at the top — those belong in `CHANGELOG.md`.
3. **`AGENTS.md`** (this file) — when a new workflow, convention, or guard-rail emerges, add it here so the next agent inherits it. Per-version gotchas at the bottom are the institutional debugging memory; append a new section per release if you hit a non-obvious bug.

Additional doc surface:

- **`doc/full-walkthrough-plan.md`** — canonical version-agnostic end-to-end walkthrough. Updated when an sbatch script changes or a new step is added. Every concrete sbatch lives here verbatim so version-specific walkthroughs (`doc/full-walkthrough-vX.Y.Z.md`) can reproduce.
- **`doc/full-walkthrough-vX.Y.Z.md`** + matching `doc/full-walkthrough-vX.Y.Z/` chart directory — committed once per release that completes a clean end-to-end run. Latest: `v0.24.20`.
- **`doc/healthchecks.md`** — operator-facing reference for the `azhealthcheck` binary, the 5 checks, the Prometheus textfile exporter, and the Grafana dashboard wiring. Update when adding/removing a check or changing a severity classification.

If a PR touches code but skips any of these three required artifacts (CHANGELOG, README, AGENTS), it is incomplete. The doc/ surface updates are required only when the underlying contract changes.

## Release Workflow

1. Land all `Unreleased` work; verify the four gates below.
2. Edit `CHANGELOG.md`: rename `## [Unreleased]` → `## [X.Y.Z] - YYYY-MM-DD`, add a fresh empty `## [Unreleased]` block at the top.
3. Bump version: `Cargo.toml` (workspace `[workspace.package] version = "X.Y.Z"`) and the `--azcluster-version` CLI default in `crates/azcluster-cli/src/main.rs`.
4. Regenerate `bicep/main.json` (`az bicep build --file bicep/main.bicep --outfile bicep/main.json`) if any `bicep/*.bicep` or `cloud-init/*.tmpl` changed — the CLI's `build.rs` embeds it and the local build fails without it. It is **not committed** (gitignored); CI and the release pipeline regenerate it from a pinned Bicep version, so there is nothing to commit and no drift check.
5. Update README's `**Current release:**` line + the install snippet `VERSION=vX.Y.Z`.
6. Commit, `git tag vX.Y.Z && git push origin main --tags` — CI publishes the release.
7. Live-validate on a fresh cluster in the current test region (today: `eastus`) before declaring done. See "Live-test workflow" below.

## Live-test workflow

- **Region**: `eastus` is the default test region as of v0.24.x. Capacity for the `Standard_D*as_v5` family is intermittent — when scheduler/login deploys fail with `SkuNotAvailable`, override with `--scheduler-sku Standard_HB120rs_v3 --login-sku Standard_HB120rs_v3`. HBv3 capacity is reliably available because the GPU pool needs ND_v5 in the same region.
- **Cluster name**: convention `v<MAJOR><MINOR><PATCH>walk` (e.g. `v2420walk` for v0.24.20). The plan canonical command is `azcluster deploy --name <name> --pool name=gpu,sku=Standard_ND96isr_H100_v5,count=2,default --bastion`; add `--location eastus --grafana-location eastus --scheduler-sku Standard_HB120rs_v3 --login-sku Standard_HB120rs_v3` for the eastus capacity workaround.
- **Token files**: `hf_token.txt` and `ngc_key.txt` at the repo root (both gitignored). NEVER echo their contents; pass them via `azcluster scp ... :/shared/home/clusteradmin/.hf-token` per the walkthrough plan.
- **Capture wall-clock + sacct + charts** as the walkthrough plan documents. Commit version-specific results under `doc/full-walkthrough-vX.Y.Z.md` + `doc/full-walkthrough-vX.Y.Z/` once the run is clean.
- **Tear down immediately** after capture: `azcluster delete <name>` + `azcluster purge-kv --name <name> --location <region> --yes` to release H100 quota and the KV name.

## Hard Rules

- **No CycleCloud / Jetpack / CCWS references** in any public artifact (README, code, commits, tags, release notes). Past mentions were scrubbed; don't reintroduce them. Historical references in `CHANGELOG.md` entries and AGENTS.md gotcha sections are fine because they document the debugging trail.
- **No personal identifiers or tenant-specific values** in committed files. Subscription IDs, real RG names, emails, tenant IDs all stay out of git.
- **No `az` shell-out from `crates/azcluster-cli/`.** v0.20.0 removed all 17 `Command::new("az")` sites in favour of native ARM REST + OAuth2. CI clippy with `-D warnings` enforces no dead-imports. Local dev commands in README/AGENTS that use `az bicep build` are fine — that's the build-time Bicep transpiler, not a runtime dependency.
- **Public IPs off by default** for login. Opt-in via `--login-public-ip`; default path is Bastion (`--bastion`).
- **All code Rust** except Bicep (transpiled to ARM JSON) + bootstrap shell in cloud-init templates.
- **Research checkouts** live under `research/` (gitignored). Planning artifacts under `.sisyphus/` (gitignored). Walkthrough logs/charts under `walkthrough-logs/` and `walkthrough-charts/` (also gitignored; they're scratch from live runs).
- **Minimize comments**; let code be self-documenting. Necessary exceptions: clap doc-comments (render as `--help` text), complex algorithms with non-obvious invariants, security-critical sections, ARM REST request body contracts (where field placement matters — see `arm/client.rs` for the canonical example).
- **Never suppress errors silently.** No `let _ = ...` on `Result` without a justifying comment; no `#[allow(...)]` blanket; no `.unwrap()` on user input paths. `expect("...")` is acceptable when the message documents why the invariant holds.
- **Never commit** unless the user explicitly asks. Force-add (`git add -f`) requires explicit user direction — the walkthrough docs under `doc/full-walkthrough-v*/` are gitignored-by-pattern but committed on user request once a clean run is captured.
- **Autonomous versioning is enabled** (user directive): when a coherent unit of work is complete, verified, changelogged, and live-validated, the agent SHOULD bump the version, commit, tag, and push without waiting for per-release permission. Apply SemVer strictly: feature batch → minor; bugfix-only → patch; breaking → major.
- **Walkthrough plan is the source of truth.** When debugging a failed walkthrough run, re-read `doc/full-walkthrough-plan.md` BEFORE proposing flag changes. The canonical command in the plan is what worked in the most-recent green release; deviations need explicit justification.

# YAML sanity for cloud-init templates (placeholders → dummy values):

## Azure / Infra Gotchas

- **AzSecPack policy** auto-attaches `AzSecPackAutoConfigUA-<region>` UAI to every VM/VMSS. Any PUT on a VMSS `identity` collection triggers `LinkedAuthorizationFailed` unless the caller has `Managed Identity Operator` on that UAI (out-of-band). Use VMSS scale ARM verb that doesn't touch `identity`, not the property-patch verb.
- **VMSS Flex + SystemAssigned identity** is rejected in subscriptions with AzSecPack/UAI-only policy (`InvalidParameter` on `identity`). VMs are fine; VMSS must use UserAssigned (or no MI).
- **Slurm configless mode** does NOT distribute `plugstack.conf.d/`. Write `/etc/slurm/plugstack.conf` (absolute path) on every node that uses Pyxis (compute + login + scheduler).
- **Pyxis spank library** (`spank_pyxis.so`) must be installed on EVERY node that may submit `srun` — including scheduler — because `plugstack.conf` loads at submit time. Forgetting it on scheduler crashes any `srun` invoked from there with `Dlopen of plugin file failed`.
- **Enroot directory perms** need `1777` (sticky world-write) on `/var/lib/enroot{,/runtime}`, `/var/lib/enroot-data{,/cache}`, and `/run/enroot` — top-level AND subdirs. Plus `0666` on `/dev/infiniband/*` via udev rule.
- **VMSS Flex VMs** surface as `Microsoft.Compute/virtualMachines` named `vmss-<cluster>-<pool>_<hex>`, NOT under the VMSS resource. The Slurm-side hostname is `<cluster>-<pool>-NNNN` (auto-set by cloud-init); the two names are unrelated. VNet DNS resolves the Slurm name only from inside the VNet.
- **Image**: `microsoft-dsvm:ubuntu-hpc:2404` (default), `2204` fallback. Ships drivers + OFED + NCCL + `nvidia-smi` (even on CPU SKUs — use `nvidia-smi -L | grep -cE '^GPU [0-9]+:'` as the GPU presence check).
- **Slurm 25.11 + Pyxis 0.21.0** ABI match. NVIDIA Pyxis 0.24.0 exists; only bump if Slurm 26 is needed.
- **Slurm 25.11 dynamic nodes**: `slurmd --conf "...Partitions=<pool>"` is rejected. Use the NodeSet+Feature pattern: emit `NodeSet=<pool>set Feature=pool_<pool>` and `PartitionName=<pool> Nodes=<pool>set ...` in `slurm.conf`; the compute slurmd registers with `--conf "...Feature=pool_<pool>"`.
- **Managed Grafana region coverage**: `Microsoft.Dashboard/grafana` is NOT available in many regions (e.g. `southafricanorth`). Use `--grafana-location <supported-region>` when needed.
- **Monitoring Data Reader role GUID** is `b0d8363b-8ddd-447d-831f-62ca05bff136`. Verify role GUIDs via ARM REST before baking into Bicep.
- **AMW ingestion RBAC**: `Monitoring Metrics Publisher` MUST be granted on the AMW's **default Data Collection Rule** (in the Azure-managed sister RG `MA_<amwName>_<location>_managed`), NOT on the AMW account itself. Role at the AMW scope passes role-listing checks but ingestion still returns 403.
- **AMW ingestion RBAC propagation**: 5-10 min on a freshly-created MI + DCR. After propagation, `systemctl restart prometheus` on the node is required to drop the cached bearer token.
- **Cloud-init `runcmd` 0077 umask** produces 0600 files from `cat > file <<EOF` heredocs. Both Prometheus (non-root `prometheus` user) and Slurm (non-root submitter parses `/etc/slurm/*.conf` locally for spank plugins) fail with permission denied. Always follow heredocs with explicit `chmod 0644`. Better: use the `trap '... || true' EXIT` pattern at the top of the install script so chmod runs even if the script aborts mid-way.
- **Cloud-init script silent abort**: scripts running under `set -euo pipefail` can abort on any transient failure (curl/IMDS/apt hiccup, slapd race, etc.) and cloud-init reports `status: done` because the *cloud-init phase* completed — it doesn't track the inner script's exit code through `tee`. v0.24.12 wraps each `install-X.sh` invocation in a 10-attempt retry loop with 30 s gap; the script exits 0 fast at the top if `/var/log/azcluster/ready` already exists. Any new install-script logic must be idempotent (apt is naturally so; useradd/groupadd already use `getent || useradd`; mkdir is `-p`).
- **apt-lists lock** at `/var/lib/apt/lists/lock` is NOT covered by `-o DPkg::Lock::Timeout=600` (that only handles the dpkg locks). v0.24.11 added an `apt_wait()` shell function that `fuser`-polls all 3 locks (dpkg-frontend, dpkg, apt-lists) plus an `aptget()` wrapper called instead of bare `apt-get`. Generalisable rule: any cloud-init script that runs apt-get MUST go through a wrapper that polls fuser on `/var/lib/apt/lists/lock` as well, not just dpkg-frontend.
- **Compute `/etc/hosts` 127.0.1.1** breaks cross-node Gloo/PyTorch rendezvous (Megatron-Bridge uses Gloo for CP groups during init). After `hostnamectl set-hostname`, `sed -i '/^127\.0\.1\.1/d' /etc/hosts` + write `${IPV4} ${NEW_HOSTNAME}` (eth0 IP, NOT 127.0.1.1). Scheduler/login don't need this — slurm.conf resolves them via Azure DNS suffix.
- **Prometheus `azuread.managed_identity` remote_write** works directly against the AMW DCE ingestion URL — no AMA or DCR custom scrape config needed for VMs. The UAI MUST be attached to the VM AND hold Metrics Publisher on the DCR. Audience is implicit (`https://monitor.azure.com/`).
- **Grafana dashboard JSON** lives under `grafana/dashboards/`. Keep panel ids stable across edits, use the datasource template variable named `DS_PROMETHEUS`, and never hardcode Azure Managed Grafana datasource UIDs. Filter all queries by `nodename` label (not `instance` — each node scrapes its own colocated Prometheus on `127.0.0.1:9100` so the `instance` label is identical across the fleet).
- **`bicep/main.json` is generated, NOT committed** (gitignored via `bicep/**/*.json`; the old `!bicep/main.json` exception was removed). The CLI embeds it at compile time via `include_str!`, and `crates/azcluster-cli/build.rs` `panic!`s with actionable instructions (`az bicep build --file bicep/main.bicep --outfile bicep/main.json`) if the file is absent — so a from-source build fails loudly instead of embedding a stale template. CI's `rust` job and `release.yml`'s `build` + `assets` jobs each regenerate it from a **pinned** Bicep version (`v0.43.8`) before `cargo build`. There is no committed copy and therefore no drift check (the old drift `diff` was removed). Runtime is still `az`-free; the Bicep dependency is build-time only, and only for people building from source (end users get the prebuilt binary with the template already baked in).
- **Cloud-init template variables containing whitespace** (e.g. `EXTRA_PACKAGES`) MUST be emitted quoted in env files (`EXTRA_PACKAGES="{{EXTRA_PACKAGES}}"`). Unquoted, bash parses `EXTRA_PACKAGES=git-lfs python3.12-venv` as a scoped env assignment for the next command, not a global export — and the install script aborts under `set -e`.
- **`Command::new("ssh").args(["host", "--", "bash", "-lc", "cmd"])` is BROKEN.** OpenSSH whitespace-joins all positional remote args, so the remote sees `bash -lc cmd arg1 arg2` and parses it as `-c 'cmd'` (ignoring everything after). Pass multi-token remote shell commands as a single string positional arg: `cmd.args(["host", "tail -n1 /path 2>/dev/null || echo MISSING"])`.
- **YAML write_files heredoc terminators**: cloud-init `write_files` with `content: |` block scalars strips the FIRST line's leading whitespace as the indent and removes that prefix from every subsequent line. Any `EOF` terminator MUST sit at exactly that base indent column in the template so it lands at column 0 in the materialised script - bash heredocs without `<<-` only recognise the closer at column 0. Extra indentation from being nested inside `if`/`for` (e.g. 6-space base + 2-space if-body = 8-space `EOF`) silently breaks the heredoc and chains errors until bash hits EOF with "syntax error: unexpected end of file". Always put `EOF` at the template's base indent column even inside `if` blocks.
- **AMG Grafana Admin for dashboard import**: `az grafana dashboard create` calls the Grafana HTTP API (`POST /api/dashboards/db`), which requires a Grafana role on the AMG. ARM contributor / owner is not enough. Grant the deployer principal `Grafana Admin` (`22926164-76b3-42b3-bc55-97df8dab3e41`) on the AMG resource scope, and expect 1-3 min propagation - retry with back-off on `401 NoRoleAssignedException`.
- **`Microsoft.Dashboard/grafana/dashboards@*` is NOT a real ARM resource**: BCP081 ("resource type does not have types available") is the warning, and ARM preflight returns `ResourceTypeRegistrationNotFound`. Import dashboards via the Grafana HTTP API (`az grafana dashboard create` or direct REST), not Bicep. The CLI does this post-deploy with the dashboard JSONs embedded via `include_str!`.
- **Shared filesystem modes**: `--shared-storage anf` (default) provisions Azure NetApp Files; `--shared-storage nfs-scheduler` exports `/shared` from the scheduler via `nfs-kernel-server`. The latter is test-only (no HA, SPOF on scheduler) but saves ~12 min off provisioning. Login + compute always mount with a 5-minute retry loop because the scheduler is still installing its NFS server while they boot in `nfs-scheduler` mode.
- **Deployment timing**: every successful `azcluster deploy` writes `~/.config/azcluster/deployments/<cluster>/<utc-stamp>.json` with per-resource `duration_seconds` and a `trend.tsv` for comparison. Inspect via `azcluster timings <cluster>`. Source: `az deployment operation {sub,group} list` recursed across nested modules; `properties.duration` parsed from ISO-8601 (`PT##H##M##S`).
- **`--no-monitoring` / `--no-accounting`**: monitoring + accounting default to ON. Pass these flags to skip them during rapid test deploys that don't depend on observability or accounting. Accounting (v0.13.0+) provisions an Azure Database for MySQL Flexible Server + runs `slurmdbd` on the scheduler.
- **MySQL Flex TLS CA**: Azure MySQL Flexible Server presents a DigiCert Global Root CA chain, which is **already in Ubuntu's `ca-certificates`** bundle. Set `StorageParameters=SSL_CA=/etc/ssl/certs/ca-certificates.crt` in `slurmdbd.conf`; do NOT try to download `https://dl.cacerts.digicert.com/DigiCertGlobalRootCA.crt.pem` — that host returns a cert whose SAN doesn't match the hostname and `curl` aborts with `SSL: no alternative certificate subject name matches target host name`. v0.13.0/v0.13.1 had this bug; fixed in v0.13.2.
- **Slurm accounting (v0.13.0)**: MySQL Flexible Server (`mysql-<cluster>`, `Standard_B2ms`, MySQL 8.0.21, 50 GB autogrow, `publicNetworkAccess=Disabled`) is provisioned in a delegated subnet `10.42.8.0/29` (delegation `Microsoft.DBforMySQL/flexibleServers`). The CLI auto-generates the admin password from `/dev/urandom` (32 random chars from an ambiguity-free alphabet plus literal `Aa1!` to satisfy Azure's "≥3 character classes" policy — Azure rejects passwords with only 2 classes) and passes it as a `@secure()` Bicep parameter. On the scheduler, the password lands in `/etc/azcluster/accounting.password` (mode 0600 root:root via cloud-init `write_files`) and is read into `slurmdbd.conf` (mode 0600 slurm:slurm) at boot, then the shell var is `unset`. TLS is required: the bootstrap downloads `https://dl.cacerts.digicert.com/DigiCertGlobalRootCA.crt.pem` to `/etc/slurm/ssl/` and `slurmdbd.conf` sets `StorageParameters=SSL_CA=…`. Bootstrap order MUST be: install `slurm-smd-slurmdbd` → write `slurmdbd.conf` → wait for `:3306` reachability (60×5s probe via `/dev/tcp`) → start `slurmdbd` → sleep 5s → start `slurmctld` → `sacctmgr -i add cluster`. Starting `slurmctld` before `slurmdbd` with `AccountingStorageEnforce=associations,limits,qos` set will cause slurmctld to refuse to start. Three MySQL server params are tuned per slurmdbd recommendations: `innodb_lock_wait_timeout=900`, `max_allowed_packet=16777216`, `log_bin_trust_function_creators=ON`.
- **`aka.ms/InstallAzureCLIDeb` is FORBIDDEN inside `set -euo pipefail` bootstraps** (v0.13.1 fix). The pipe script runs `apt-get install` internally without our `DPkg::Lock::Timeout=600`, so it races `apt-daily`/`unattended-upgrades` and aborts the entire bootstrap with `Could not get lock /var/lib/dpkg/lock-frontend`. Cloud-init still reports "finished" because runcmd pipes through `tee`. Always install azure-cli via the explicit MS apt repo (`packages.microsoft.com/repos/azure-cli/`) using our own `apt-get -o DPkg::Lock::Timeout=600 install -y azure-cli`. The MS GPG key (`/usr/share/keyrings/microsoft-prod.gpg`) is already imported earlier in the scheduler bootstrap for the slurm repo, so reuse it.
- **`tee` masks `runcmd` exit codes.** The compute/scheduler bootstraps run `/opt/azcluster/install-X.sh 2>&1 | tee -a /var/log/azcluster/install.log`. If the script aborts with a non-zero exit, the pipe still succeeds because `tee` (the last stage) returns 0. cloud-init then reports `finished` and you get a half-configured node that registers (or doesn't) with Slurm in a broken state. To diagnose, always `tail /var/log/azcluster/install.log` on the node and grep for `Error|fail|exit` rather than trusting `cloud-init status`.
- **`set -euo pipefail` + bash command substitution + pipelines is a sharp edge** (v0.13.4 fix). `drv=$(ethtool -i lo | awk ...)` aborts the whole script when the `awk` pattern matches nothing on a virtual interface, because the pipe's exit status under `pipefail` is non-zero and `set -e` kills the script. Always add `|| true` *inside* the `$(...)` for diagnostic pipes whose failure is expected and tolerated.
- **Slurm configless distribution does NOT push `gres.conf`** (v0.13.4 fix). `SlurmctldParameters=enable_configless` only distributes `slurm.conf`. Both scheduler (for slurmctld's `GresTypes=` validation) and every compute node (for slurmd's local Gres discovery) must have their own `/etc/slurm/gres.conf`. The simplest version that works on Azure is `AutoDetect=nvml` on both sides — slurm-smd is built with NVML support, and slurmctld accepts NVML autodetect even though it has no GPUs itself.
- **Slurm caches a node's first registration** (v0.13.4 lesson). If a slurmd registers once with bad Gres/Parameters/CPUs, restarting slurmd alone does NOT re-trigger fresh registration; slurmctld holds the stale INVALID_REG state. Operator must `scontrol update nodename=<n> state=DOWN reason=... && scontrol delete nodename=<n>` from the scheduler, *then* restart slurmd on the compute node. Build automation around this if you're iterating on `--conf` strings.
- **`nvidia-smi --query-gpu=name` returns multi-token product strings** (v0.13.4 fix). On H100 it's "NVIDIA H100 80GB HBM3". Naive `awk '{print tolower($NF)}'` yields "hbm3" not "h100" — Slurm registers `Gres=gpu:hbm3:8` and every `--gres=gpu:h100:N` job stays PENDING with reason `Resources`. Parse the first token matching `/^[A-Za-z][0-9]+$/` to pick out the SKU code (h100, h200, a100, v100, etc.).
- **NDv5 needs `Parameters=l3cache_as_socket` on slurmd's `--conf`, NOT `SlurmdParameters=` in slurm.conf** (v0.13.4 fix). Setting it via slurm.conf is silently ignored by slurmd (the option is per-node and read from the conf-server reply, not the global config). Setting it on the slurmd command line via `--conf "... Parameters=l3cache_as_socket"` works. Also: dropping `CPUs=` from `--conf` is required for autodetected S:C:T to honour the L3-as-socket grouping; explicit `CPUs=96` collapses everything back to one socket and confuses NCCL's GPU↔NIC affinity.
- **The 64 GB root disk on NDv5 is too small for container imports** (v0.13.4 fix). The `microsoft-dsvm:ubuntu-hpc:2404` image leaves ~7 GB free on `/` after base install. A single 10 GB Pyxis container import (e.g. `ghcr.io/azure/ai-infrastructure-on-azure/nccl-test:latest`) ENOSPC's mid-extraction. Always relocate `/var/lib/enroot` and `/var/lib/enroot-data` to symlinks under `/mnt` (the Azure ephemeral disk, ~956 GB NVMe-backed on NDv5) when `/mnt` is mounted.
- **HPC-X is the in-image MPI on Azure HPC/AI VMs** (v0.13.4 lesson). `/opt/hpcx-v2.25.1-gcc-doca_ofed-ubuntu24.04-cuda13-x86_64/` ships with OpenMPI + UCX + a bundled PMIx 4.2.x that ABI-matches Slurm 25.11's `mpi_pmix_v4.so`. Use it for bare-metal multi-node MPI/NCCL jobs by `source $HPCX_DIR/hpcx-init.sh && hpcx_load` then `srun --mpi=pmix ...`. The marketplace image also ships `/opt/nccl-tests/build/all_reduce_perf` prebuilt — no need to compile NCCL tests yourself.
- **Cross-node containerised MPI via Pyxis needs slurmd PMIx env + enroot PMI hooks** (v0.13.6 fix; supersedes earlier "PMIx 4↔5 ABI" claim; **live-validated v0.13.6 on `paul-azcluster-h100c`**: 2-node × 8-GPU NeMo container all-reduce reached a single 16-rank PMIx world and exited cleanly; orthogonal IB-visibility bug surfaced — see next gotcha). The real failure mode on v0.13.4/v0.13.5 was NOT an ABI mismatch. - **Cloud-init duplicate-writer bug** (v0.16.1 fix). When adding a new block to a `runcmd:` that writes to a path some earlier prototype block also writes to, the LAST writer wins because runcmd is sequential. v0.16 added a wrapper at `/usr/local/sbin/azcluster-healthcheck` that delegated to the new `/usr/local/bin/azhealthcheck` Rust binary, but left a pre-existing legacy inline-shell wrapper writing the same path further down `cloud-init/compute.yaml.tmpl` in place. The legacy block ran after the v0.16 block on every boot and overwrote it, so the Rust binary was never invoked. Lesson: when adding a cloud-init writer for a path, `grep -n '<path>' cloud-init/*.tmpl` first and consolidate. The legacy script ALSO hit the `command -v nvidia-smi` gotcha — drained every CPU node every 5 min with `Reason=azcluster-healthcheck: nvidia-smi -L failed`. Live-reproduced on `paul-azcluster v16a` (southafricanorth, 2x Standard_D8as_v5) before the fix.

- **Heredoc inside cloud-init `content: |` block fails when delimiter is at column 0** (v0.18.0 lesson, sibling of the existing `write_files` heredoc terminator gotcha). The install-scheduler.sh script lives inside a `content: |` YAML block scalar with 6-space base indent. A bash heredoc body (e.g. `debconf-set-selections <<DEBCONF ... DEBCONF`) requires the closing delimiter at column 0 of the *materialized* script — meaning it must be at column 6 in the template. Putting the body lines and closer at column 0 of the template ejects them from the YAML block scalar (YAML uses the first content line's indent as the base; lines with less indent terminate the scalar). YAML still parses (the line at column 0 starts a new top-level mapping key that gets ignored or errors elsewhere) but the materialized script is missing the heredoc body. Two fixes: (1) keep all heredoc body + closer at the YAML base indent (column 6) — works if the consumer tolerates leading whitespace; (2) avoid the heredoc entirely — use `printf '%s\n' "line1" "line2" ... | <cmd>`. We picked (2) for `debconf-set-selections` in scheduler.yaml.tmpl. Validate end-to-end before deploying: `sed -E 's/\{\{[A-Z_]+\}\}/PLACEHOLDER/g' cloud-init/X.yaml.tmpl > /tmp/x.yaml && python3 -c "import yaml; yaml.safe_load(open('/tmp/x.yaml'))"` for YAML, then extract each `*.sh` write_file and `bash -n` it.

- **`internalNsg` allows all intra-VNet traffic by default** (v0.18.0 verification, not a bug). NSG-default `AllowVnetInBound` permits VirtualNetwork→VirtualNetwork on every port, so adding new control-plane services on the scheduler subnet (LDAP 389/TCP from login + compute subnets in v0.18.0) needs NO NSG rule changes provided source + destination are inside the cluster VNet. Same applies to the `loginNsg` which has explicit allow-22 rules on top of the default. Only `Internet→VNet` (e.g. SSH from operator laptop) requires an explicit rule.

- **OpenLDAP 2.6 requires a STRUCTURAL objectClass on every entry** (v0.18.1 fix). An entry whose only objectClass is `extensibleObject` (AUXILIARY) is rejected with `Object class violation (65) — no structural object class provided`. Common fix for "bag-of-attributes" entries (like the `cn=uidNext` counter): combine a minimal STRUCTURAL class such as `device` (which MUST only `cn`, and we already provide it) with `extensibleObject` to allow arbitrary attributes (e.g. `uidNumber`). Live-reproduced on v0.18.0 — first failed entry in the LDIF aborted the entire `install-scheduler.sh` under `set -euo pipefail`, leaving the cluster half-provisioned (no `/var/log/azcluster/ready`, no `/shared/home`, no `chmod` on `/shared`, all downstream RBAC and `azcluster user` ops broken).

- **`ldapadd` idempotency**: re-running a previously-loaded LDIF returns exit `68` (`LDAP_ALREADY_EXISTS`). Combined with `set -e` this makes naive `if ! ldapsearch | grep -q ...` guards brittle: the lookup is racy right after `apt install slapd` (slapd's `cn=schema,cn=config` settles asynchronously), and once the guard misses, the second-run add also fails. Pattern that works: drop the guard, run `ldapadd -c -f file` (`-c` = continue on per-entry errors), capture the exit status with `set +e; ldapadd ...; rc=$?; set -e`, and explicitly tolerate `rc == 68` while bailing on any other non-zero.

- **`/shared` from ANF mounts as `drwxrwx--- nobody:nogroup` (0770) by default**, blocking non-`nobody` users from traversing it. Anything under `/shared` (e.g. `/shared/home/<user>` created by `pam_mkhomedir` with the right owner/perms) is unreachable for an LDAP-resolved user until the parent allows traversal. Fix: `chmod 0755 /shared` in the scheduler bootstrap before populating any user-facing subdirectory. Individual user home dirs stay 750 — only the parent needs `o+x`.

- **OpenSSH `ssh -J ... bash -lc '$(cat)'` silently drops piped stdin** (v0.18.1 fix). When a Rust process invokes `ssh -o ... -J <login> <scheduler> -- bash -lc '<cmd>'` with `Stdio::piped()` on stdin and the remote command reads via command substitution (`X=$(cat)`), the subshell created by `$()` does NOT inherit the ssh channel's pipe stdin under ProxyJump + no-PTY, so `cat` reads **0 bytes** and `X` is empty. The same payload works when the remote command uses a bare pipeline (`cat | wc -c`) or heredoc (`<<EOF`). Diagnostic recipe: `echo -n "X" | ssh -J L T bash -lc 'V=$(cat); echo got=${#V}'` prints `got=0`, while `echo -n "X" | ssh -J L T bash -lc 'cat | wc -c'` prints `1`. Fix in azcluster CLI: drop stdin entirely; base64-encode the payload in Rust and inline it on the remote command line via `printf %s '<b64>' | base64 -d`. Adds ~20 LOC for an RFC 4648 encoder; no external dep needed (the CLI's auto-generated passwords use a shell-safe alphabet, so the b64 string survives single-quoting trivially).

## v0.19.2 gotchas (--extra-package, scheduler SSSD, env-var quoting)

- **SSSD `ldap_uri = ldapi:///` requires SASL EXTERNAL config that we don't supply** (v0.19.2 live-validation finding). The first cut of the scheduler SSSD block used `ldap_uri = ldapi:///` figuring that since slapd listens on the UNIX socket and SSSD is a local client, no TCP loopback was needed. `sssctl domain-status` reported `Online status: Offline / LDAP: not connected` and `getent passwd <ldap-user>` returned nothing even though `ldapsearch -H ldapi:///` worked from the same shell. Reason: SSSD's LDAP provider can speak LDAPI but only with `ldap_sasl_mech = EXTERNAL` (autobind via SO_PEERCRED), which we didn't configure — it defaulted to anonymous-bind-over-LDAPI which slapd rejects. Fix: use `ldap_uri = ldap://127.0.0.1`. slapd already listens on `0.0.0.0:389` (and the NSG default `AllowVnetInBound` covers it), and loopback is never wire-visible so the cleartext concern is moot. Same anonymous-bind pattern as the login + compute SSSD configs (which already use `ldap://${AZCLUSTER_SCHEDULER}`).

- **Env vars whose value contains whitespace MUST be quoted in env files sourced under `set -e`** (v0.19.2 fix). `cloud-init/{scheduler,login,compute}.yaml.tmpl` previously emitted `EXTRA_PACKAGES={{EXTRA_PACKAGES}}` in install.env / cluster.env. With `--extra-package git-lfs --extra-package python3.12-venv`, the template expands to `EXTRA_PACKAGES=git-lfs python3.12-venv`. When `install-scheduler.sh` does `. /etc/azcluster/install.env` under `set -euo pipefail`, bash parses that line as a SCOPED ENV ASSIGNMENT for one command, not a global export — so it tries to run `python3.12-venv` as a command with `EXTRA_PACKAGES=git-lfs` in its env. That exits 127 (`command not found`), `set -e` kills the source, and `install-scheduler.sh` aborts at line 1 with no useful error in `/var/log/azcluster/install.log` (the log redirect happens AFTER the source). Fix: emit `EXTRA_PACKAGES="{{EXTRA_PACKAGES}}"` (quoted). General rule: any cloud-init template variable that may contain whitespace MUST be emitted quoted in env files.

- **Defensive `az login --identity` in scheduler bootstrap** (v0.19.2 fix). The scheduler cloud-init unconditionally ran `az login --identity --client-id {{UAI_CLIENT_ID}}`. Under `--no-monitoring --no-accounting` test mode the UAI has no subscription-level RBAC, so the login returned non-zero, `set -e` killed `install-scheduler.sh`, and the cluster never finished bootstrap. The line was vestigial — scheduler doesn't invoke any `az` subcommands later (monitoring/accounting paths handle their own login). Fix: append `|| true 2>&1`. Pattern: any `az` call in cloud-init that isn't followed by a hard dependency MUST be defensive.

- **Per-user enroot CACHE path uses runtime `$(id -u)` expansion**, not a literal cloud-init substitution. The line in `/etc/enroot/enroot.conf` reads `ENROOT_CACHE_PATH /var/lib/enroot-data/cache/user-$(id -u)` — enroot evaluates the `$()` at container-import time as the invoking user, so each user gets `/var/lib/enroot-data/cache/user-20001`, `/var/lib/enroot-data/cache/user-1000`, etc. The parent dir `/var/lib/enroot-data/cache/` is 1777-sticky (matches the v0.13.5 RAID-0 fix) so the subdirs are auto-created on first import without root involvement. Do NOT bake a fixed UID into the path at cloud-init time — that would defeat the per-user isolation.

- **NGC container import on /mnt/nvme NVMe RAID takes ~20 min for the FIRST ~20 GB image, not "seconds"** (v0.19.2 live-finding). The v0.13.5 gotcha claimed "first-time import of a ~20 GB image completes in tens of seconds, not minutes" — that's wrong; the NGC pull dominates and is bottlenecked by NGC's CDN + the extraction-to-squashfs step. Subsequent imports of the SAME image complete in seconds (squashfs cached). Don't size benchmark wall-clocks assuming sub-minute imports.

- **Pyxis `--container-name` is per-node, not per-cluster** (v0.19.2 live-finding). A pre-import step like `srun --container-image=X --container-name=Y --nodes=1 --ntasks=1 bash -c imported` imports the container ONLY on the node Slurm picks. A subsequent `srun --container-name=Y --nodes=2 ...` then fails on the second node with `a container with name "pyxis_Y" does not exist, and --container-image is not set`. Pattern that works: import on every node first (`srun --nodes=N --ntasks=N --ntasks-per-node=1 --container-image=X --container-name=Y bash -c true`), THEN run the multi-node step that references the name. Or: pass `--container-image=X` to every srun and accept the per-step container creation overhead.

- **bare-metal PMIx env vs container PMIx env**. Bare-metal: `srun --mpi=pmix bash -c 'echo $PMIX_RANK'` correctly prints rank 0..N-1 across ranks (PMIx populates PMIX_RANK in the stepd-injected env, no PMIX_SIZE). `srun --mpi=pmi2 bash -c 'echo $PMI_RANK / $PMI_SIZE'` prints both. Container: needs the `50-slurm-pmi.sh` enroot hook to copy `PMIX_*` / `PMI_*` env into the container at start (v0.13.6 wired this). When the hook is present but the import races (per-node container name semantics, see above), per-rank env can be partially populated. For containerised multi-node work, prefer `--mpi=pmi2` + a torchrun-style rendezvous over TCP if the workload supports it — that's what the v0.19.1 walkthrough's container path uses.

## v0.19.3 gotchas (sacctmgr auto-registration + dual-layer uid cache)

- **`slurmctld` assoc_mgr caches username→uid for process lifetime; `scontrol reconfigure` does NOT invalidate it** (v0.19.3 live-finding). When you delete LDAP user `paul` (uid 20007) and re-add with the same username (uid 20009), `slurmctld` keeps resolving `paul` → 20007 until the process is restarted. `sbatch` then either accepts but the job runs as the wrong uid, or fails with `Invalid account or account/partition combination specified` if you've also cycled the sacctmgr account. The only way to refresh assoc_mgr's uid cache is `sudo systemctl restart slurmctld` (the process re-runs `getpwnam(paul)` on startup). Cost: ~3 s; zero impact on running jobs (slurmd retains workload state across slurmctld restart). Pattern for any CLI that mutates LDAP usernames+sacctmgr together: flush scheduler-side SSSD (`sss_cache -u <user> && sss_cache -E`) BEFORE restarting slurmctld, otherwise the restart re-resolves against a stale SSSD cache and the bug persists. `v0.19.3` `crates/azcluster-cli/src/user.rs` `build_sacctmgr_{add,remove}_cmd` encode this exact sequence; tests assert ordering (sss_cache before restart).
- **Slurm `sacctmgr` duplicate-account stderr uses capital-A "Already existing", NOT "already exists"** (v0.19.3 live-finding, on Slurm 25.11). Idempotency case-matches must accept both casings. Exact stderr: `" Already existing account paul on cluster v193a"` (leading space). The `sacctmgr_run` helper in `user.rs` matches `*"already exists"*|*"Already existing"*|*"Nothing new added"*|*"Nothing deleted"*` as idempotent (return 0).
- **Slurm `sacctmgr` transient errors** when slurmdbd is mid-bootstrap or the cluster hasn't yet been added to the accounting DB: `"Connection refused"`, `"Unable to contact slurmdbd"`, `"slurmdbd is down"`, `"cluster has not been added"`, `"is not registered"`. v0.19.3 retries up to 12 × 5 s = 60 s on these. Fatal everything else.
- **Re-adding a removed LDAP user reuses the username but gets a NEW uidNumber** (LDAP `cn=uidNext` counter is monotonic). The previous home dir `/shared/home/<user>` on ANF is owned by the OLD uid; the new uid cannot read it (`Could not chdir to home directory ... Permission denied`). v0.19.3 does NOT auto-`chown` the home dir on re-add — operator must `sudo chown -R <user>: /shared/home/<user>` manually if home contents need preserving. Documented footgun, not a bug; the trade-off is that we don't risk silently giving the new uid access to files the old uid created (e.g. `.ssh/authorized_keys` could otherwise be hijacked).

## v0.19.4 gotchas (UCX_TLS=tcp breaks container HPC-X/MPI)

- **`UCX_TLS=tcp` in `/etc/enroot/environ.d/50-nccl.env` hangs every `*_mpi` NCCL test and every HPC-X-based MPI workload inside Pyxis containers** (v0.19.4 fix). The env file was added in v0.13.5 alongside the per-container NCCL tunings; `UCX_TLS=tcp` was carried over from a TCP-only debug config and never re-examined because the v0.13.5/v0.13.8 validation only exercised the bare `all_reduce_perf` binary (which uses NCCL's OOB bootstrap + IBext plugin, NOT UCX). Once you run the `_mpi` variant (`/usr/local/bin/all_reduce_perf_mpi` shipped in NGC NeMo/PyTorch images), HPC-X's UCX kicks in for the MPI bootstrap, sees `UCX_TLS=tcp`, and tries to find TCP transport on every `UCX_NET_DEVICES` interface. `mlx5_ib0..7` are IB-only — no TCP socket — so UCX errors `ucp_context.c:1582 UCX ERROR no usable transports/devices (asked tcp on network:mlx5_ib0:1,mlx5_ib1:1,...)` from every rank. OMPI then falls back to its TCP BTL, which routes correctly via eth0 but hangs in `btl_tcp_endpoint.c:625:mca_btl_tcp_endpoint_recv_connect_ack: received unexpected process identifier [[X,Y],N]` (process IDs collide because OMPI's pid-derived rank IDs aren't stable across the Pyxis container's REMAP_ROOT namespace). The job sits RUNNING in Slurm indefinitely. Fix: drop the `UCX_TLS=tcp` line entirely from both `/etc/profile.d/nccl-azcluster.sh` AND `/etc/enroot/environ.d/50-nccl.env`. UCX auto-detects `rc,ud,sm,self` for IB devices when unset. NCCL's path is unaffected because NCCL uses its own `IBext_v10` plugin, not UCX. ## v0.21.1 gotchas (azcluster scp + login rebind fast-path)

- **`azcluster login --subscription <id>` fast-path** (v0.21.1). Management-scope access tokens (`aud=https://management.azure.com/`) are principal-scoped, not subscription-scoped — so rebinding the cache entry under a new sub id is a 6 ms file mutation, no re-auth needed. `auth/token_provider::try_rebind_cached(target_sub_id, tenant_filter)` returns `Ok(Some)` if the cache holds an account for any visible sub under the same principal (filtered by tenant when provided), refreshes via `refresh_with_token` only when within the 5-min expiry window, then re-keys the entry. `login()` consults this before launching any interactive flow. Critical workaround for **Microsoft tenants where Conditional Access blocks device-code flow** (AADSTS53003): operator runs `azcluster login` once interactively in a browser-equipped shell, then `azcluster login --subscription <id>` works from any TTY (including SSH-without-X11). DO NOT remove the `match` over `Option::is_none_or` — workspace MSRV clippy gate (`clippy::incompatible_msrv`) fails on `is_none_or` (stabilised in 1.82, MSRV is older).
- **`azcluster scp` syntax mirrors real `scp`, not `cp`** (v0.21.1). The first-colon-before-first-slash rule decides remote vs local: `./a:b` is local (slash before colon), `a:b/c` is remote node `a` path `b/c` (colon before slash). Empty node defaults to `login` (`:/shared/foo`). Other accepted nodes: `scheduler`, any compute hostname (e.g. `vmss-<cluster>-<pool>NNNNNN` — Azure VNet DNS auto-registers these because cloud-init sets the hostname before Slurm config, and the login VM resolves them via the VNet DNS suffix). A single invocation may target only ONE remote node; remote-to-remote disallowed (operator must `ssh && scp` for that). Enforced in `scp()` at parse time.
- **scp route resolution** lives in `resolve_scp_route(state, node, use_bastion) -> (proxy_target: Option<String>, jump_login_host: Option<String>, dest_host: String)`. Six combinations:
  - `login` + bastion: `--target login`, no jump, dest `127.0.0.1`.
  - `login` + no bastion: no proxy, no jump, dest = `login_public_ip`.
  - `scheduler` + bastion: `--target scheduler` (DIRECT bastion→scheduler VM, no login hop), no jump, dest = `scheduler_private_ip`. Mirrors `azcluster ssh --scheduler` semantics — Bastion can tunnel to any VM in the VNet by ARM resource id.
  - `scheduler` + no bastion: no proxy, jump = `azureuser@<login_public_ip>`, dest = `scheduler_private_ip`.
  - `<compute>` + bastion: `--target login`, jump = `azureuser@127.0.0.1`, dest = compute hostname. Compute ALWAYS traverses login (Bastion has no VNet DNS resolver for VMSS instance hostnames; resolution happens on login via VNet DNS).
  - `<compute>` + no bastion: no proxy, jump = `azureuser@<login_public_ip>`, dest = compute hostname.
- **scp command surface**: dispatches to OS `scp` (not implemented in Rust) with composed `-o ProxyCommand=...` / `-o ProxyJump=...` flags. The `bastion-proxy` invocation is `${self_exe_path} bastion-proxy --cluster <name> --target {login|scheduler}` — same hidden subcommand as `azcluster ssh`. Unit-tested without Azure via `resolve_scp_route` + `parse_scp_path` (11 tests in `crates/azcluster-cli/src/main.rs::tests::{parse_scp_*,resolve_scp_*}`).
- **Auto-route helper reused**: `should_use_bastion(state, no_bastion_flag) = !no_bastion_flag && state.bastion_enabled && state.login_public_ip.is_none()`. Same function as `ssh`/`exec`/`tunnel` — DRY across all four commands.

## v0.21.0 native Azure Bastion (no plugin)

- **`--bastion` flag on `azcluster deploy`** provisions Standard SKU Bastion + Standard Static public IP into a new `AzureBastionSubnet = cidrSubnet(vnetAddressPrefix, 26, 1)` (= `10.42.0.64/26`). `enableTunneling: true` is required for the token-based WS bridge; deploys without it cannot mint tunnel tokens (HTTP 400). Adds ~3-5 min wall-clock.
- **`AzureBastionSubnet` is a Microsoft-reserved name.** Bicep `name: 'AzureBastionSubnet'` MUST match exactly (case-sensitive); ARM rejects any other name with `BastionSubnetMustBeNamedAzureBastionSubnet`. Minimum /26.
- **`AllowAzureLoadBalancerInBound` on the cluster's internal NSG is sufficient** for Bastion's intra-VNet SSH-to-target traffic (port 22 from Bastion subnet → login/scheduler/compute subnets). The Bastion-managed `bastionNetworkSecurityGroup` handles the public-facing rules (HTTPS 443 from Internet + Azure Bastion service tags). No NSG edits needed on the cluster side.
- **Tunnel-token API is `POST https://<bastion-fqdn>/api/tokens` with `application/x-www-form-urlencoded` body** — NOT JSON, NOT bearer-auth. Form fields: `resourceId` (full target VM ARM resource id), `protocol=tcptunnel`, `workloadHostPort=22`, `aztoken=<operator-AAD-bearer>`. Optional: `token=<previous-token>` for refresh, `X-Node-Id` HTTP header for sticky session. Response JSON: `{authToken, nodeId}`. Token TTL ~3 min. The standard `Authorization: Bearer` header DOES NOT WORK on this endpoint — must be in the form body as `aztoken`.
- **WS endpoint is `wss://<bastion-fqdn>/webtunnelv2/<authToken>?X-Node-Id=<nodeId>`**. **Do NOT use `tokio-tungstenite` (or any strict-RFC WS client) for this — Azure Bastion's 101 Switching Protocols response omits the `Sec-WebSocket-Accept` header.** `tokio-tungstenite` rejects with `WebSocketProtocolError(InvalidHeader)`. Use hand-rolled WS framing on top of `tokio-rustls` (live in `crates/azcluster-cli/src/bastion/tunnel.rs`): client→server frames MUST be masked (4-byte XOR key); server→client frames are unmasked; payload-length encoding 7-bit / 7+16-bit / 7+64-bit; opcodes BINARY=0x02 (data), CLOSE=0x08, PING=0x09, PONG=0x0A.
- **Hidden `azcluster bastion-proxy` subcommand** is the bridge entry point. Designed exclusively as a stdio consumer for `ssh -o ProxyCommand="azcluster bastion-proxy --cluster N --target {login|scheduler}"`. Reads `ClusterState` for `bastion_dns_name`, mints token, opens WS, pipes stdio↔WS. Cleans up token on shutdown via `DELETE /api/tokens/<token>` (best-effort — Bastion auto-reaps after TTL anyway). `--target scheduler` connects DIRECTLY to scheduler VM (Bastion can tunnel to any VM in the VNet by ARM resource id); does NOT need a `-J` jump through login. VM resource ID derived from cluster state by convention: `/subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/vm-<cluster>-{login|scheduler}`.
- **Auto-routing in `ssh`/`tunnel`/`exec`**: `should_use_bastion(state, no_bastion_flag) = !no_bastion_flag && state.bastion_enabled && state.login_public_ip.is_none()`. When true, the spawned `ssh` gets `-o ProxyCommand="<self> bastion-proxy ..."` and dials `azureuser@127.0.0.1` (login) or `azureuser@<scheduler-private-ip>` (scheduler). `--no-bastion` flag on each command opts out and surfaces the legacy "no public IP" error — useful if Bastion is misbehaving and operator wants the original code path.
- **OAuth2 access token must have audience `https://management.azure.com/`** (the same management-scope token used for ARM REST). Bastion accepts this same audience for tunnel-token auth; no separate scope/audience needed. Token is fetched via `get_access_token()` which reuses the cached token from `azcluster login`.
- **CLI is otherwise pure-sync; bastion-proxy is the ONE async subcommand.** Uses `tokio::runtime::Builder::new_current_thread().enable_all().build()?.block_on(...)` to isolate the async runtime to just that subcommand. Do not async-ify the rest of the CLI to accommodate this.
- **dead_code allow on `run_tcp_listener` + `handle_one`** in `bastion/tunnel.rs` is intentional — the TCP-listener mode (local-port-forward over Bastion) is not yet wired into a CLI surface but the code is ready to ship when needed.

## v0.21.3 gotchas (LDAP-user UX + jump identity)

- **Jump identity MUST match destination identity for `ssh`/`exec`/`scp` `--user <ldap-user>`** (v0.21.3 design). Earlier prototype used `jump_user = state.admin_username` and `connect_user = args.user.unwrap_or(admin)`, on the theory that admin (azureuser) always has the operator's key on login via Bicep `linuxConfiguration.ssh.publicKeys`. That's true for the deployer's laptop but BREAKS for any operator whose pubkey was only enrolled via `azcluster user add <user> --ssh-key ...` (i.e. the LDAP path) — admin's authorized_keys wouldn't contain their key, so the ProxyJump hop fails with `Permission denied (publickey)`. Fix: `jump_user = connect_user` so a single identity authenticates at every hop. LDAP user `alice` → login (SSSD resolves her pubkey) → compute (SSSD resolves it again) all using the same `-i` key. Operator-as-admin keeps working because azureuser's authorized_keys has the deployer's pubkey AND `--user` defaults to admin.

- **`--scheduler --user <ldap-user>` does NOT work, by design**: the scheduler hosts the slapd LDAP server itself and runs NO SSSD client (login + compute are clients; scheduler is the source of truth). LDAP users therefore have no `getpwnam` resolution on scheduler, no `/shared/home/<user>`, and `sshd` rejects the key lookup. Documented as a known limitation in CHANGELOG + README "First user" section. Not worth adding SSSD to scheduler — job submission is from login (which has it), scheduler is admin-only by convention. If a future use case needs scheduler shell access for LDAP users, the fix is to add an SSSD block to `cloud-init/scheduler.yaml.tmpl` mirroring login's (pointing at `ldap://127.0.0.1`), and ensure pam_mkhomedir runs in scheduler's pam stack.

- **`--host <hostname>` resolves via VNet DNS on login**, not on the operator's laptop. Azure VNet's auto-DNS registers VMSS instance hostnames (`<cluster>-<pool>-<NNNN>`) into the VNet's private DNS zone, but only resolvers inside the VNet see them — `dig +short <cluster>-cpu-0001` from the operator's laptop returns nothing. The `--host` flag therefore always uses a ProxyJump through login (or bastion-proxy → login), even when login has a public IP. Don't try to "optimize" by dialing the compute hostname directly from the operator — there's no DNS to resolve it.

- **`azcluster exec` does NOT forward the SSH agent by default** (v0.21.3 design). Earlier `ssh` had `-A` baked in; new `exec` makes it opt-in via `--forward-agent` / `-A`. Rationale: `exec` is typically used for one-shot commands where the remote shell shouldn't need to ssh anywhere else; defaulting to agent-forward is unaudited and potentially confusing for ops workflows. If you need nested ssh from within an `exec` payload (rare: usually a sign the workload belongs in a script on `/shared`), add `-A` explicitly.

## v0.21.2 gotchas (hygiene trio)

- **Cloud-init `runcmd` 0077 umask + `set -euo pipefail` mid-script abort silently breaks `/etc/slurm/*.conf` perms** (v0.21.2 fix; sibling of v0.13.9). The chmod 0644 block near the end of `install-scheduler.sh` (and analogous plugstack.conf chmod in login/compute) only runs when the script reaches it. Concrete reproducer: deploying with `--azcluster-version vX.Y.Z` BEFORE that release tag has been published → `curl -fsSL` on the spank_pyxis.so download returns 404 → `-f` flag aborts the script at line ~229 → chmod at line ~241 never runs → slurm.conf stays 0600 → every non-root `srun`/`sinfo` dies with `Permission denied`. Fix: `trap 'chmod 0644 /etc/slurm/<files> 2>/dev/null || true' EXIT` immediately after `set -euo pipefail`, so chmod runs regardless of where the script aborts. Applied to all three install scripts. Does NOT fix the underlying tarball-404 race — operator still has a half-installed cluster — but prevents the most common downstream UX failure.

- **`azcluster timings` recursion broken after v0.20.0 native ARM REST switch** (v0.21.2 fix). Pre-v0.20.0 the CLI shelled out to `az deployment {sub,group} operation list` which always populated top-level `targetResource.resourceGroup`/`resourceType`/`resourceName`. Native ARM REST `GET .../deployments/<name>/operations?api-version=2024-03-01` ALSO populates them for resource-typed ops, BUT for sub-scope nested `Microsoft.Resources/deployments` entries `resourceGroup` is empty (sub-scope deployments don't live in any RG). The v0.13.x recursion gate `if rtype == "Microsoft.Resources/deployments" && !rg.is_empty()` silently skipped every nested module → captured ops collapsed to just the root (`{count: 1, total: -0}`). Plus float quirk: `iter().filter(...).sum::<f64>()` over empty serialised as `-0` via serde_json. Fix: (1) new `parse_target_id` helper extracts `(rg, type, name)` from the ARM ID string for both rg-scoped and sub-scope shapes, used as fallback when explicit fields are missing; (2) `collect_sub_operations` recurses via `list_subscription_deployment_operations` when parsed rg is empty (sub-scope nested) vs `list_resource_group_deployment_operations` otherwise; (3) `let total = if total == 0.0 { 0.0 } else { total };` normalises `-0.0`. 3 new unit tests in `timings::tests`. v19walk (pre-refactor) captured 35 ops / 3063s; expect similar post-fix.

- **`azcluster status` bootstrap probe** showed stale `curl 404` from `install.log` tail when bootstrap had failed; bypassed bastion auto-route entirely (v0.21.2 fix). Now: (a) checks `/var/log/azcluster/ready` first (written at end of every install script after all steps complete) → prints `READY` instead of misleading log tail; otherwise falls back to install.log tail with an `in-progress | last log: ...` prefix; (b) honors `should_use_bastion` + `bastion-proxy` ssh `ProxyCommand` so probe works for `--bastion` clusters with no public IP; (c) probe enabled whenever login is reachable (`login_public_ip.is_some() || (bastion_enabled && bastion_dns_name.is_some())`).

## v0.22.0 gotchas (Key Vault + stateless CLI)

- **KV name MUST be derived deterministically client-side, threaded as a Bicep param, AND stamped into the `azcluster:kv` RG tag** (v0.22.0 design). Pattern: `kv-azc-<8-hex-blake3(sub_id|name|location)>`. Hard 24-char limit on Key Vault names — 8 hex chars max. Don't try to derive it Bicep-side from a hash of the deployment id; ARM doesn't expose a hash function and `uniqueString()` collides too easily with other resources in the same RG. The `azcluster:kv` RG tag is the **only** way `azcluster list` and `azcluster <subcommand> --no-cache` can find the vault from the subscription alone — without it the CLI would have to enumerate every KV in every RG, which is slow + scope-leaky.
- **ARM RG `PATCH` is 200-sync, not 202-async** (v0.22.0 confirmed). `PATCH /subscriptions/{}/resourcegroups/{name}?api-version=2024-03-01` with a tags-only body returns 200 with the updated RG envelope inline. No `Azure-AsyncOperation` header to poll. Use `arm_client::patch_and_wait` which handles both shapes — it short-circuits on 200 and only polls on 202. Don't roll your own polling for RG tag PATCHes.
- **Key Vault tokens are a DIFFERENT audience from ARM management tokens** (v0.22.0 design). Vault: `https://vault.azure.net/.default`. ARM: `https://management.azure.com/.default`. The vault endpoint server-enforces audience and rejects ARM tokens with HTTP 401 `WWW-Authenticate: Bearer authorization_uri="..."` — silent failure if you forget. `get_vault_token()` mints + caches vault-scope tokens independently of `get_access_token()`. Both reuse the same cached OAuth2 principal at `~/.azure/azcli_tokens.json` so the user only authenticates once.
- **Two KV secrets per cluster, both JSON-encoded**: `cluster-manifest` (full `ClusterState`) and `secrets-bundle` (LDAP admin password + MySQL admin password if accounting + `admin_ssh_public_key` + `admin_ssh_private_key`, all OpenSSH PEM). Don't split into more secrets — Key Vault charges per secret and the per-secret 25 KiB limit is comfortably above what we need (~3 KiB combined). `ClusterSecrets::admin_ssh_{public,private}_key` are `Option<String>` with `#[serde(default)]` so deploys with pre-existing secrets (resume path) don't break on schema evolution.
- **Admin SSH keypair is generated fresh per deploy, NEVER read from `~/.ssh/*`** (v0.22.0 invariant). User insists: "create a new one - don't take the one from the local computer". Use `ssh-key` 0.6 with features `["ed25519", "alloc"]` — note `alloc` is required for `to_openssh()` (PEM serialisation), the default feature set only exposes raw key bytes. The private key materialises lazily to `~/.azcluster/keys/<cluster>` (file mode `0600`, parent dir mode `0700`) on first `azcluster ssh/exec/scp/tunnel` — never written by `azcluster deploy` itself.
- **`fetch_admin_private_key(name)`** checks `~/.azcluster/keys/<name>` first; if absent, calls `resolve_cluster(name)?` + `KeyVaultClient::get_secret("secrets-bundle")` + parses + writes. Important: do NOT make `azcluster deploy` itself write the private key locally — operators may deploy from one machine and run `azcluster ssh` from another, and the materialise-on-use pattern keeps both paths uniform.
- **All 16 `ClusterState::load(...)` sites in `main.rs` were swapped to `resolve_cluster(...)` via a mix of ast-grep (15 trivial `?`-form sites) + 2 manual edits (`status` uses `.ok()` form, `delete` uses `match {…}` form for fallback). `delete` retains its `ClusterState::load` fallback path so pre-v0.22 clusters (no `azcluster:managed` tag) can still be torn down; every other subcommand hard-errors on resolve failure.
- **All 6 `if let Some(id) = &args.identity { -i ... }` blocks** (ssh, tunnel, exec, scp, logs, validate) were replaced atomically with `let identity = resolve_identity(args.identity.as_deref(), &args.name)?; cmd.args(["-i", &identity.display().to_string()]);` — single ast-grep run, idempotent. User-supplied `-i` still wins; default is the lazily-materialised admin key.
- **Global `--no-cache` flag** is stored in a `static AtomicBool NO_CACHE` set in `main()` before dispatch. Plumbing it through every handler signature would have touched 16 sites; the atomic is read once per resolution in `resolve_cluster()`. Defensible because the flag is genuinely process-global — there's no scenario where one subcommand wants cache-on and another cache-off in the same invocation.
- **`derive_kv_name` uses blake3** (not sha256) for two reasons: (1) faster, (2) `blake3` is already a transitive dep of `rustls` via `ring` so it adds nothing to the binary. We take 8 hex chars (32 bits) from the 256-bit digest — collision probability across a single subscription's RGs is negligible (operator would need ~65k clusters to hit a single collision at 50% probability, and the failure mode is a deploy-time error not a silent data swap because Bicep ARM rejects duplicate KV names).
- **Clean break, no migration.** Pre-v0.22 clusters lack the `azcluster:managed` tag → `list_resource_groups_by_tag` returns empty → `resolve_cluster` errors. Operator must `azcluster delete <name>` (which falls back to `ClusterState::load` for tear-down) then re-deploy on v0.22. Don't add a v0.21.x compatibility shim — it doubles the resolution surface area for one release of grace value.
- **`bicep/cluster.bicep` lines 240-242** output `keyVaultName`, `keyVaultUri`, `keyVaultId` from the keyvault module. `bicep/main.bicep` plumbs the `keyVaultName` Bicep param through to the cluster module (computed CLI-side, not derived in Bicep — see "deterministic" gotcha above). `bicep/main.json` MUST be regenerated after editing any of these so the local CLI build embeds the new template (it is generated, not committed — see the de-commit gotcha).
- **RBAC propagation: same 5-10 min lag as Monitoring Data Reader** (v0.13.0 carry-forward). The Bicep `keyvault.bicep` module grants the deployer Key Vault Secrets Officer (`b86a8fe4-44ce-4948-aee5-eccb2c155cd7`) on the vault scope. Immediately after deploy, `KeyVaultClient::set_secret` may return HTTP 403 `Forbidden` for ~30s-2min until the role assignment propagates. The CLI does NOT currently retry this (single-PUT, fails loudly) — adding 60s of retry with 5s backoff is a backlog item if v0.22 live-validation reveals flakiness. Workaround: re-run `azcluster deploy` (now idempotent via resume path), which on the second invocation will catch the secrets push.

## v0.22.2+ gotchas (azcluster purge-kv + ARM bodyless POST Content-Length)

- **Azure ARM frontend rejects bodyless POST with `HTTP 411 Length Required`** (live-caught on the first `azcluster purge-kv` invocation against `kv-azc-90188ada` in southafricanorth). `reqwest::blocking::Client` does NOT add `Content-Length: 0` on POSTs with no body. Azure's frontend (the layer in front of `Microsoft.KeyVault`, likely IIS-flavoured per the HTML error shape `<HTML><HEAD><TITLE>Length Required</TITLE>`) rejects with 411. Fix in `ArmClient::purge_deleted_vault`: append `.header(reqwest::header::CONTENT_LENGTH, "0")` to the builder. **General rule: every ARM POST that does not include a body MUST set `Content-Length: 0` explicitly.** The existing `post()` helper accepts `body: Option<Value>` and serialises `None` as no body — same bug would surface if any caller passes `None`; currently no caller does, but if one is added the helper itself should set the header.

- **`Microsoft.KeyVault/locations/{loc}/deletedVaults/{name}/purge` is a 202 long-running operation** (~6 minutes observed for a `kv-azc-*` vault with ~3 secrets in `southafricanorth`). The CLI correctly delegates the 202 case to `wait_for_async_operation` (10s polling, 30min cap). Bulk `--all` is serial, so multi-vault runs can exceed any short shell `timeout` an operator wraps around the call. The interactive `'yes'` gate + the `Permanently purge ...` warning set sufficient expectations; no further UX changes planned unless complaints surface.

- **`Microsoft.KeyVault/deletedVaults` list endpoint is subscription-wide, not location-scoped.** `GET /subscriptions/{sub}/providers/Microsoft.KeyVault/deletedVaults?api-version=2023-07-01` returns every soft-deleted vault in the subscription regardless of region — more useful than the location-scoped variant because operators rarely remember which region a torn-down cluster was in. CLI filters client-side via `filter_purge_kv_candidates`: (1) `kv-azc-` prefix to scope to azcluster-managed vaults, (2) optional exact name match when `--name` derives it from sub+name+location, (3) optional location (case-insensitive).

- **`kv-azc-*` prefix discipline is the ONLY safety net protecting non-azcluster vaults from accidental `purge-kv --all`.** If `crypto::derive_kv_name` ever changes prefix, MUST grep `kv-azc-` in `main.rs` (`filter_purge_kv_candidates`) and AGENTS.md gotchas and update both. Hardcoded in `main.rs`; deliberately not configurable.

## v0.22.4 gotchas (live deploy-progress recursion + ARM op `resourceGroup` absent)

- **ARM deployment-operation envelopes return `properties.targetResource` with only `id`, `resourceType`, `resourceName` — `resourceGroup` is NOT populated.** Live-confirmed v0.22.4 against `azcluster-v224a-20260526-144125`: every targetResource is exactly those 3-4 fields (the cluster-level nested deployment op also has `apiVersion`). Any code that needs the RG to recurse into rg-scoped nested deployments MUST parse it out of `id`'s path segment `/resourceGroups/<rg>/...`. Same trap that bit `timings::parse_target_id` in v0.21.2 — now also bit the live-progress recursive walker in `arm/client.rs::nested_module_target` until v0.22.4 added the same id-path fallback. **Rule for any future ARM walker code**: never trust `targetResource.resourceGroup`; always have an `id`-parsing fallback. Three unit tests in `arm/client.rs::tests` pin the contract: `nested_module_target_falls_back_to_id_when_resource_group_field_absent`, `_sub_scope_has_empty_rg`, `_non_deployment_returns_none`.

- **`tee` + `&` background pipelines from the agent's `bash` tool get reaped when the spawning subshell exits** (v0.22.4 lesson). `./azcluster deploy ... 2>&1 | tee log &` does NOT survive the bash tool's per-call subshell — the orphaned pipeline dies on next return. Symptoms: `ps -p <pid>` shows nothing, log file truncated mid-deploy, ARM deployment completes successfully on its own. The deploy itself is unaffected (ARM keeps running) — only the local progress display is lost. Run `azcluster resume --name <name>` afterwards to re-attach via the pending marker.

## v0.22.1 gotchas (deploy keypair race + ssh `-J` identity propagation)

- **`finalize_deploy()` rebuilds `ClusterSecrets` from in-memory `existing_secrets`, which is `None` on fresh deploys** (v0.22.1 fix). `deploy()` (main.rs ~1003) generates the admin ed25519 keypair, persists it via `secrets.save(&args.name)`, then submits ARM. After ARM completes, `finalize_deploy()` runs and reconstructs `ClusterSecrets { admin_ssh_public_key, admin_ssh_private_key, ldap_admin_password, mysql_admin_password }` from `existing_secrets` (the snapshot captured BEFORE the keypair was generated). On a fresh deploy `existing_secrets` is `None`, so the keypair fields default to empty strings; the subsequent `secrets.save()` clobbers the on-disk file written at line 1003, and the KV upload pushes empty keys. Effect: `azcluster ssh/exec` fails with `secrets-bundle in vault '<kv>' has no admin_ssh_private_key`. Re-running deploy then generates a FRESH keypair (local secrets file now empty) but ARM rejects with `PropertyChangeNotAllowed` because `osProfile.linuxConfiguration.ssh.publicKeys` is immutable on existing VMs. Fix: `finalize_deploy()` first calls `ClusterSecrets::load_optional(&args.name)` to pick up the freshly-persisted on-disk copy, and only falls back to `existing_secrets` when the disk copy is missing. Sibling lesson: any post-ARM step that mutates state MUST load from canonical storage (disk or KV), NEVER from a function-local snapshot captured before the deploy.

- **OpenSSH `-J <jump>` does NOT propagate `-i <identity>` to the jump-hop ssh** (v0.22.1 fix). The inner `ssh -W %h:%p <jump>` that OpenSSH spawns to handle `-J` falls back to the agent / `~/.ssh/id_*`, ignoring the outer `-i`. v0.21.x worked because the admin SSH key was a copy of the operator's `~/.ssh/id_ed25519` (already in the agent). v0.22 generates the admin keypair fresh per deploy and stores it ONLY in Key Vault + `~/.azcluster/keys/<cluster>` — no agent, so jump hops fail with `Permission denied (publickey)`. Diagnostic: `ssh -i ~/.azcluster/keys/X -J azureuser@<login> azureuser@<scheduler> hostname` fails; `ssh -i ~/.azcluster/keys/X -o ProxyCommand="ssh -W %h:%p -i ~/.azcluster/keys/X -o IdentitiesOnly=yes azureuser@<login>" azureuser@<scheduler> hostname` succeeds. Fix: helper `add_ssh_jump_with_identity(cmd, identity, jump_target)` emits `-o ProxyCommand="ssh -W %h:%p -i <key> -o IdentitiesOnly=yes -o StrictHostKeyChecking=accept-new -o UserKnownHostsFile=/dev/null -o LogLevel=ERROR <jump>"` (NOT `-J`). Applied to all 8 jump sites: `ssh()` (×3 — scheduler/host/login-bastion), `exec()` (×3 — same), `scp()` (×1), `bootstrap_probe()` (×1). `IdentitiesOnly=yes` is critical: without it, ssh tries the agent first and gets stuck if any unrelated keys are loaded with the same comment field. Generalisable rule: **for any Rust CLI that drives ssh with non-default identities through a jump host, NEVER use `-J`; always compose `-o ProxyCommand` with explicit `-i` on the inner ssh.**

- **`bootstrap_probe()` was running without `-i`**, silently relying on the operator's ssh-agent and returning `ERR (Permission denied)` on every status call from a fresh laptop. v0.22.1 makes it resolve the admin identity via `resolve_identity(None, &state.name)` and inject `-i <key> -o IdentitiesOnly=yes` into the outer ssh AND use the new jump-with-identity helper. If the identity can't be resolved (e.g. operator hasn't run `azcluster login` or KV RBAC isn't propagated), the probe prints `SKIP (no admin key: <reason>)` and `status` continues — best-effort by design.



## v0.22.5 gotchas (ARM deployment-ops per-state-transition duplicates)

- **ARM's `Microsoft.Resources/deployments/<name>/operations` endpoint returns one operation per provisioningState TRANSITION per target, not one op per target.** A resource that goes Accepted → Running → Succeeded shows up as THREE distinct operation entries in the listing, each with the same `properties.targetResource.id` but different `properties.provisioningState` + different operation `id` + different timestamps. The v0.22.4 recursive walker (`arm/client.rs::collect_sub_ops_into` / `collect_group_ops_into`) emitted each op as a `DeploymentOp`, so the live TTY progress renderer showed every resource 2-3× per tick. Worse: when a nested module deployment had multiple state-transition ops, the walker recursed into the same child deployment N times — duplicating the entire descendant subtree N times. v0.22.5 adds `dedup_ops_by_target(ops: Vec<Value>) -> Vec<Value>` that keeps the LATEST (latest-seen-wins) op per `targetResource.id` while preserving first-seen DFS order, called at the head of both walker methods before iteration. Ops with no `targetResource.id` (a handful of root-level deployment lifecycle ops with `targetResource: null`) are silently dropped — they have no render representation anyway. Two unit tests pin the contract (`dedup_ops_by_target_keeps_latest_state_per_id_preserves_first_seen_order`, `dedup_ops_by_target_drops_ops_without_target_id`). **Rule for any future ARM operation-listing code**: ALWAYS dedup by `targetResource.id` before downstream consumption — assume the same target appears multiple times.

## v0.22.6 gotchas (user.rs ssh wrappers missed by v0.22.1 -J sweep)

- **`crates/azcluster-cli/src/user.rs` has its own ssh wrappers (`ssh_run` + `flush_login_sssd_cache`) that were missed by the v0.22.1 sweep** that fixed the OpenSSH `-J` non-propagation bug at the other 8 jump sites in `main.rs`. v0.22.0 generates a fresh admin ed25519 keypair per deploy and stores it in Key Vault (materialised lazily to `~/.azcluster/keys/<cluster>`), so the operator's ssh-agent does NOT hold it. v0.22.1 patched `ssh`/`exec`/`scp`/`tunnel`/`status` bootstrap probe to pass `-i <key>` AND emit an explicit `-o ProxyCommand="ssh -W %h:%p -i <key> -o IdentitiesOnly=yes ..." <login>` instead of `-J <login>` (because `-J`'s inner ssh inherits neither the outer `-i` nor `IdentitiesOnly=yes`). `user.rs` was overlooked: every `azcluster user {add,remove,list,sshkey {add,remove,list}}` invocation goes through `ssh_run`, which was still doing bare `ssh -J azureuser@<login> azureuser@<scheduler>` with no `-i`. Live-reproduced on `paul-eus-hb120-h100`: `azcluster user add ... --username paul --ssh-key ...` fails with `azureuser@<login>: Permission denied (publickey)` on any laptop where the operator has not manually `ssh-add ~/.azcluster/keys/<cluster>`. v0.22.6 fix: `ssh_run` and `flush_login_sssd_cache` now call `crate::resolve_identity(None, &state.name)` to get the KV-backed key path, append `-i <key> -o IdentitiesOnly=yes` to the outer ssh, and use `crate::add_ssh_jump_with_identity` for the jump hop. `resolve_identity` + `add_ssh_jump_with_identity` are now `pub(crate)` so user.rs (and any future submodule that drives ssh) can call them.
- **Generalisable rule for the next agent**: when adding any code path that spawns `ssh` to a cluster resource, grep for `Command::new("ssh")` in the entire `crates/azcluster-cli/src/` tree (NOT just `main.rs`) and verify ALL sites use `resolve_identity` + `add_ssh_jump_with_identity`. Submodules with their own ssh wrappers are the most likely place to forget. As of v0.22.6, the only sites are `main.rs` (×6 — ssh, exec, scp, tunnel, logs, bootstrap_probe) and `user.rs` (×2 — `ssh_run`, `flush_login_sssd_cache`); any new site MUST follow the same pattern.
- **`azcluster user add` does NOT require the new LDAP user to be logged-in/reachable.** The ssh transport is `azureuser@login → azureuser@scheduler -- ldapadd ...`. The new user is just an LDIF record written to slapd; no per-user ssh until they `azcluster ssh --user <name>` later (which uses the SSSD path + `sshPublicKey` attribute). Don't conflate "the cluster admin's key works" with "the new user can log in" — they're orthogonal.

## v0.22.7 gotchas (LDAP-user SSH needs different identity than admin)

- **The v0.22 admin SSH key (KV-backed, `~/.azcluster/keys/<cluster>`) ONLY authenticates `azureuser`.** v0.22.1 wired every operator-facing ssh path (`ssh`/`exec`/`scp`/`tunnel`/`logs`/status probe/`user.rs` wrappers) to unconditionally pass `-i <admin-key>` + `IdentitiesOnly=yes`. That breaks `--user <ldap-user>` (added back in v0.21.3): the LDAP user's `authorized_keys` is served by SSSD `sss_ssh_authorizedkeys` reading the `sshPublicKey` attribute of the user's LDAP entry, populated by `azcluster user {add,sshkey add} --ssh-key <file>` — typically the operator's `~/.ssh/id_rsa.pub` or `id_ed25519.pub`. The admin ed25519 is not in any LDAP user's `sshPublicKey`, so `-i <admin-key> -o IdentitiesOnly=yes` forces a key sshd will never accept and the connection fails with `Permission denied (publickey)` BEFORE ssh tries the agent or `~/.ssh/id_*` defaults. Live-reproduced on `paul-eus-hb120-h100` against LDAP user `paul`. v0.22.7 fix: new `resolve_identity_for_user(explicit, cluster_name, connect_user, admin_user) -> Result<Option<PathBuf>>` returns `None` when `connect_user != admin_user` and no explicit `--identity` is passed; callers conditionally emit `-i` only when `Some`. New `add_ssh_jump(cmd, identity: Option<&Path>, jump_target)` wrapper emits bare `-J <jump>` when identity is `None` (correct because the inner ssh uses the same default key discovery as the outer ssh — agent / `~/.ssh/id_*` — so the two hops authenticate consistently). Applied to `ssh`, `exec`, `scp`. Admin-only paths (`tunnel`, `logs`, `validate`, status probe, `user.rs` wrappers) keep using `resolve_identity` + `add_ssh_jump_with_identity` because they always connect as `azureuser` and need the KV key.
- **Generalisable rule for the next agent**: any new ssh code path that honours a `--user <name>` flag MUST use `resolve_identity_for_user` (not `resolve_identity`) and `add_ssh_jump` (not `add_ssh_jump_with_identity`). The distinction matters: forcing the admin key on a non-admin connect breaks the connection BEFORE ssh's default discovery runs (`IdentitiesOnly=yes` is implied by `-i` in ProxyCommand patterns) — there is no automatic fallback. Conversely, admin-only paths MUST keep using `resolve_identity` because the operator's local agent/keys are not guaranteed to authenticate `azureuser` (only the KV key does after v0.22).
- **Why bare `-J` is correct when identity is `None`**: the original v0.22.1 fix (always-emit `-o ProxyCommand="ssh -W %h:%p -i <key> -o IdentitiesOnly=yes ..."`) was needed because OpenSSH `-J` does NOT propagate the OUTER `-i` to the INNER ssh — the inner hop falls back to agent / `~/.ssh/id_*`, which under v0.22 is empty for `azureuser`. When the OUTER ssh ALSO has no `-i` (i.e. it's also relying on agent / `~/.ssh/id_*`), the inner hop's identical fallback is the right behaviour — both hops authenticate with the same key set. `add_ssh_jump(cmd, None, jump)` emits exactly `-J <jump>`; `add_ssh_jump(cmd, Some(key), jump)` emits the v0.22.1 ProxyCommand pattern. Same correctness invariant, two code paths.
- **Workaround for older clients (pre-v0.22.7)**: pass `--identity ~/.ssh/id_rsa` (or whichever local key matches the pubkey enrolled via `azcluster user add --ssh-key`) explicitly. The explicit-`-i` path was never broken; only the no-`-i` default was.

## v0.23.0 gotchas (storage + azcp + user-scoped paths)

- **Azure Storage account name constraints are STRICTER than Key Vault.** 3-24 chars, lowercase ASCII letters + digits ONLY (no hyphens, no underscores), globally unique across all of Azure. `crypto::derive_storage_account_name` returns `stazc<8-hex-blake3(sub|name|location)>` which is 13 chars and always satisfies the grammar. Operator override via `--storage-name <name>` is validated by `crypto::validate_storage_account_name` before submission to ARM — fail-fast at the CLI instead of waiting for ARM to reject. Same blake3 input domain as KV name keeps the two derivations stable together.

- **`allowSharedKeyAccess: false` is HARDCODED in `bicep/modules/storage.bicep`** to force AAD-based data-plane authentication. Some legacy tools break against this — older `azcopy` versions (pre-2024), Azure Storage Explorer pre-2023, and any code path that calls `getKeys`/`listKeys` REST or constructs an account-key signature. `azcp` is fine (it prefers Bearer tokens from IMDS over `AZURE_STORAGE_KEY` anyway). If a workflow needs the account key, it's not supported by azcluster — file a ticket.

- **Private Endpoint + Private DNS A-record race**: when storage is provisioned with PE on (the default), ARM creates the PE NIC + private IP + DNS A-record asynchronously. There is typically a 30-60 second window after the deployment reports `Succeeded` during which DNS lookups for `<account>.blob.core.windows.net` from inside the VNet return `NXDOMAIN` (the public DNS doesn't have a CNAME yet, and the private zone hasn't propagated). First `azcp` call from a freshly-provisioned cluster may fail with `dns lookup failed` — retry after 30-60s. We do NOT currently bake retry-on-NXDOMAIN into the cloud-init `azcp` install step because the install only downloads the binary; the first `azcp` invocation happens at user-submitted job time, by which point DNS has long settled.

- **HNS (ADLS Gen2) implications**: `--storage-hns` enables `isHnsEnabled: true` on the storage account, which adds the `dfs` endpoint alongside `blob` and enables hierarchical operations (directory rename, ACLs, POSIX-like semantics). When HNS is on AND public access is off, `bicep/modules/storage.bicep` provisions a SECOND PE + Private DNS zone for the `dfs` sub-resource. `azcp` as of v0.4.5 uses the `blob` endpoint exclusively — HNS gives no `azcp` benefit but doesn't break it either. HDInsight, Synapse, and Spark connectors are the typical HNS consumers.

- **Storage Blob Data Contributor RBAC propagation**: 5-10 min same as the AMW Monitoring Data Reader pattern (carry-forward from v0.13). The Bicep `modules/storage.bicep` grants `ba92f5b4-2d11-453d-a403-e96b0029c9fe` (Storage Blob Data Contributor) on the storage account scope to the cluster UAI principal. ARM returns success immediately; the role takes 5-10 min to propagate to the storage data plane. First `azcp copy` from a freshly-provisioned cluster may return `403 AuthorizationPermissionMismatch` — retry after the wait. We do NOT bake retry into cloud-init for the same reason as the DNS race above; surfaces at user-job time.

- **Cluster UAI is now attached to compute + login** (v0.23 change). Pre-v0.23, only the scheduler had `uai-<cluster>-scheduler` attached; compute + login had only the monitoring UAI (when monitoring was on) or nothing. v0.23 attaches the cluster UAI to compute VMSS and login VM as well, via `clusterUaiId` + `clusterUaiClientId` params threaded from `cluster.bicep` to `modules/compute.bicep` + `modules/login.bicep`. Without this, `azcp` on compute/login would have no principal that holds Storage Blob Data Contributor (the monitoring UAI does NOT have that role). The dual-UAI attachment is unconditional from v0.23 — even when `--no-storage` is set, the UAI is still attached (it does nothing without RBAC, so cost is zero). If a future release wants to make this conditional, gate it on `enableStorage` in the bicep module-call sites.

- **`UAI_CLIENT_ID` substitution is now wired into ALL THREE cloud-init templates** (compute, login, scheduler) — pre-v0.23 it was scheduler-only. `/etc/profile.d/azcluster-storage.sh` uses `{{UAI_CLIENT_ID}}` to set `AZURE_CLIENT_ID` so `azcp` (which uses IMDS multi-identity SELECTION) picks the right UAI when the VM has multiple attached (monitoring + cluster). Without `AZURE_CLIENT_ID`, IMDS returns the first UAI's token, which is the monitoring UAI's — wrong principal for storage RBAC.

- **User-scoped storage path convention**: `/data/users/<user>/` under the cluster blob container is the user-writeable area. `/data/shared/` is convention-only — RBAC is cluster-wide Contributor, so any user can write to `/data/shared/`, there is no per-prefix isolation. Per-prefix RBAC would require separate containers + per-container role assignments, which is significant Bicep complexity and not v0.23 scope. Per-node NVMe mirrors this: `/mnt/nvme/users/<user>/` is created by the Slurm prolog with `${USER}:${USER} 0700` ownership, so OS perms enforce user isolation on the local-FS side.

- **Slurm `PrologFlags=Alloc,Contain` is REQUIRED** for `SLURM_JOB_USER`/`SLURM_JOB_UID`/`SLURM_JOB_GID` to be populated in the prolog environment. The v0.23 prolog `/etc/slurm/prolog.d/10-azcluster-user-nvme.sh` `install -d -o "${SLURM_JOB_UID}" ...` would silently fail (with `install: invalid user ‘’`) if `PrologFlags` were absent or missing the `Alloc` keyword. `Contain` adds cgroup isolation, which is good hygiene but not strictly required for this prolog. Both flags are added together in `cloud-init/scheduler.yaml.tmpl`'s `slurm.conf` generation.

- **`/mnt/nvme/users/<user>/` is ephemeral** — destroyed on VM deallocation (the NVMe RAID itself is destroyed; see v0.13.5 gotcha for the RAID setup). Treat it as scratch. The Slurm prolog re-creates the directory on every job allocation, which means a user's old NVMe contents are LOST between jobs if the same VM happens to be reused. The blob path `/data/users/<user>/` is the durable copy — examples encode this: download blob → scratch on NVMe → run job → upload outputs back to blob.

- **Soft-delete recreation footgun** (mirrors the v0.22.3 KV gotcha): storage accounts default to 7-day soft-delete on blobs + containers. After `azcluster delete <name>`, the storage account itself is purged immediately (it's an RG-scoped resource), but if you recreate a cluster with the same name within the soft-delete window, the new storage account does NOT inherit the deleted blobs — those are gone unless explicitly restored. Soft-delete is for accidental deletion within a live storage account, not for cluster recreation. We do NOT need `azcluster purge-storage` because storage accounts (unlike Key Vaults) do not have a tombstone state at the subscription level — `delete` is final.

- **azcp install via release tarball is unauthenticated** (no SHA256 verification in v0.23.0). The current cloud-init block does `curl -fsSL -o /tmp/azcp.tar.gz "${AZCP_URL}"` and `tar -xzf /tmp/azcp.tar.gz` without a sidecar `.sha256` check. The release tarball ships a sidecar (`azcp-x86_64-unknown-linux-gnu.tar.gz.sha256`); a follow-up release should fetch it + `sha256sum -c` before extract. Documented limitation — same trust model as `aka.ms/InstallAzureCLIDeb` (v0.13.1 fixed AZ CLI but not yet this).

- **Example sbatch templates use `sed -i 's|^      ||'` to strip cloud-init indent.** The cloud-init heredoc body is indented 6 spaces (matching the YAML block-scalar base indent) so the YAML parses correctly. The actual sbatch files need to be at column 0 so `sbatch` parses them — hence the `sed` strip after writing. Same pattern as the `/etc/profile.d/azcluster-storage.sh` heredoc fix. If you add a new example template, you MUST add the matching `sed -i 's|^      ||'` line or the `#SBATCH` directives won't be recognized (Slurm needs `#SBATCH` at column 0).

- **`{{STORAGE_ACCOUNT_NAME}}` is empty string when `--no-storage` is set**, NOT the literal string "false". Bash `if [ -n "{{STORAGE_ACCOUNT_NAME}}" ]; then` correctly distinguishes empty from non-empty. The cloud-init templates use this idiom to gate the azcp install + storage env + slurm prolog blocks on whether storage was provisioned. Do NOT change the empty-string convention without also fixing every `[ -n ...]` check.

## v0.23.1 gotchas (from the first end-to-end walkthrough)

- **azcp tarball nests binary one dir deep.** The `azcp-x86_64-unknown-linux-gnu.tar.gz` archive from `edwardsp/azcp` v0.4.5 (and presumably later) contains `azcp-x86_64-unknown-linux-gnu/azcp`, NOT `./azcp`. v0.23.0's cloud-init had `tar -xzf … -C /usr/local/bin/ azcp` which fails with `tar: azcp: Not found in archive` and exits non-zero. Under `set -e` this killed install-{login,compute}.sh BEFORE the `touch /var/log/azcluster/ready` marker → `azcluster status` reports "in-progress" forever even though the box has booted. v0.23.1 fix in `cloud-init/{login,compute}.yaml.tmpl`: `tar -xzf /tmp/azcp.tar.gz --strip-components=1 -C /usr/local/bin/ azcp-x86_64-unknown-linux-gnu/azcp`.

- **`ENROOT_TEMP_PATH /run/enroot` (tmpfs) breaks big container imports.** mksquashfs uses `/tmp` (which is on the 61 GB root disk, NOT on the NVMe RAID) for its OWN intermediate scratch, separately from `ENROOT_TEMP_PATH`. A 16+ GB NeMo container import fills 17 GB free on `/` partway through layer extraction and ENOSPCs at `mkdir: cannot create directory '95': No space left on device`. v0.13.5 only relocated `ENROOT_RUNTIME_PATH`, `ENROOT_DATA_PATH`, and `ENROOT_CACHE_PATH` to `/mnt/nvme`. v0.23.1 also relocates `ENROOT_TEMP_PATH` to `${ENROOT_BASE}/enroot-temp` (so `/mnt/nvme/enroot-temp` when NVMe RAID is present, ~28 TB). This fixes NeMo, PyTorch NGC, and other large NGC container imports.

- **`package_update: true` races `unattended-upgrades`** on first boot on Ubuntu 24.04 noble (and probably earlier). Cloud-init's `packages:` install starts almost immediately after boot, but `apt-daily.service` and `unattended-upgrades.service` are ALSO triggered by post-boot timers, and they hold the dpkg lock for several minutes. Result: cloud-init's `package_update` returns `E: Could not get lock /var/lib/dpkg/lock-frontend. It is held by process N (apt)` and aborts. Our runcmd block that masks unattended-upgrades NEVER runs because cloud-init's package phase already failed. v0.23.1 fix: disable + mask unattended-upgrades + apt-daily timers in `bootcmd:` (which runs BEFORE `package_update`). Applied to all 3 templates (`cloud-init/{scheduler,login,compute}.yaml.tmpl`).

- **Inline shell-var append into YAML breaks block-scalar indentation.** The pattern `cat > prometheus.yml <<EOF … - targets: ['127.0.0.1:9100']${SCRAPE_GPU} … EOF` looks like a clean way to optionally append a `dcgm_exporter` scrape config, but `${SCRAPE_GPU}` is a single `$'\n        - job_name: ...\n          ...'` string whose newline indent context drifts away from the surrounding block-scalar's indent. Result: dcgm `- job_name:` ends up at the wrong indent level (under `- targets:` instead of at `scrape_configs:` level), Prometheus can't parse the YAML, service restarts forever. v0.23.1 fix: write the base config first with `cat > … <<EOF` (no inline append), then conditionally `cat >> … <<EOF` a separate block for dcgm. Same indent class throughout.

- **Slurm sbatch with `#!/bin/bash` and `set -u` loses `/etc/profile.d/*.sh` exports.** Slurm spawns the batch script as a non-login non-interactive shell, so profile.d is not sourced. With `set -u` (any reasonable sbatch hygiene), `${AZCLUSTER_USER_BLOB_URL}` is "unbound variable" and the job exits at line 1. v0.23.1 fix: example sbatch templates use `#!/bin/bash -l` (login shell). Also ensures `srun --export=` includes the storage env vars so they propagate into pyxis containers.

- **`azcp-cluster` source URL is a prefix, not a single file.** Passing `${AZCLUSTER_USER_BLOB_URL}/sqsh/nccl-test.sqsh` directly resolves to a blob name exactly equal to the prefix; azcp-cluster's list step treats it as a "directory-marker blob (name == source prefix or ends in '/')" and skips it → `[shard] rank 0: 0 ranges, 0 bytes`. v0.23.1 convention: upload the sqsh to a per-sqsh subdir (`users/<u>/sqsh/<name>/<name>.sqsh`) and point `azcp-cluster` at the subdir (`users/<u>/sqsh/<name>/`). The `azcp-build-and-publish-sqsh.sbatch` + `azcp-cluster-distribute-sqsh.sbatch` templates encode this.

- **Grafana Admin RBAC propagation > 5 min.** First-deploy Grafana dashboard import retries `401 Unauthorized` for the duration. v0.23.0 budget was 10×30s=5min, hit timeout on every fresh deploy in `eastus`. v0.23.1 budget is 20×60s=20min. Cosmetic — Grafana is reachable regardless; just the auto-import of bundled dashboards needs more patience.

- **`azcluster validate` did not auto-route via Bastion.** When deployed with `--bastion` (no login public IP), validate failed with `cluster X has no login public IP. Redeploy with --login-public-ip.` even though `ssh`/`exec`/`scp` already auto-route. v0.23.1 makes validate use the same `bastion-proxy` ProxyCommand when `should_use_bastion(state, false)` is true.

- **`azcluster list` matched Azure-managed sister RGs.** Azure Monitor Workspace auto-creates `MA_<amwName>_<location>_managed` to hold the default DCR + ingestion endpoint. ARM tag propagation from our cluster deployment leaks `azcluster:*` tags onto this managed RG. v0.23.1 adds `is_azure_managed_rg(name)` filter that rejects RGs whose name starts with `MA_`, `MC_`, `AzureBackupRG_`, `NetworkWatcherRG`, or `databricks-rg-`. Tested against `MA_amw_eastus_managed`, `MC_aks_cluster_eastus`, `NetworkWatcherRG`, `databricks-rg-workspace-id`.

- **Big NGC container imports are slow because of NGC CDN, not local I/O.** First-time import of `nvcr.io/nvidia/nemo:25.07.02` (16 GiB compressed sqsh, ~25 GiB unpacked, 103 layers) takes ~56 min on a fresh node with cold cache. Breakdown: ~40 min single-stream layer download from `nvcr.io`, ~16 min layer extraction + mksquashfs. NVMe RAID-0 (28 TB at ~50 GB/s) is NOT the bottleneck. Once the sqsh exists in `/data/users/<u>/sqsh/<name>/`, `azcp-cluster` broadcasts it to every node in ~17 s via IB. The slow path (one-time per sqsh) is intentionally separated from the fast path (every job).

- **`azcp-cluster` broadcast on small clusters is NVMe-read-bound, not IB-bound.** Single-rank-per-node `azcp-cluster` works by range-sharding the download from blob across all ranks, then exchanging the shards via `MPI_Ibcast` over IB so every rank ends with the full set on local NVMe. The bytes still have to be **read from each rank's local NVMe** before they can be broadcast out. The more nodes, the smaller each rank's read share. At N=2, each rank reads ~50% of the bytes while concurrently writing the other ~50% — NVMe read is the bottleneck and the broadcast tops out around 40 Gbps. At N=16, each rank reads ~6.25% and writes ~93.75%, and the upstream tuning doc (https://github.com/edwardsp/azcp/blob/main/docs/cluster-h100-tuning.md) measures 110 Gbps. **2 nodes is the worst case.** Document this if an operator at N=2 thinks their fabric is broken; redirect them to NCCL `all_reduce_perf` for an IB-fabric-bound number.

- **`dmesg` requires root or CAP_SYSLOG (kernel 4.10+).** azhealthcheck's `gpu_xid` and `kmsg` checks shell out to read the kernel ring buffer. Slurm's `HealthCheckProgram` invokes `azcluster-healthcheck` (the wrapper) as root via slurmd, so production scheduled runs work — but operators running `azcluster exec --user clusteradmin -- ... /usr/local/sbin/azcluster-healthcheck` interactively get `dmesg: read kernel buffer failed: Operation not permitted` and a spurious WARN. v0.24.5 switches both check call-sites to `journalctl --dmesg --no-pager --since "1 hour ago"`, which works for root unconditionally and for any user in the `systemd-journal` group. Note that `journalctl -k`, `/var/log/kern.log`, and `/dev/kmsg` all have the same EPERM constraint for non-root users — `journalctl --dmesg` is the only path that respects journald's group-based ACL.

- **Azure accelerated networking always shows one boot-time `carrier_down_count` on eth0.** SR-IOV passthrough means the kernel hot-switches eth0 from the basic emulated NIC to the SR-IOV VF early in boot. `/sys/class/net/eth0/carrier_down_count` ends up at exactly 1 and stays there for the VM's lifetime. azhealthcheck's `network` check pre-v0.24.5 counted ANY non-zero carrier transition as a warning and permanently dragged the cluster's health dashboard. Threshold is now `>= 2` — a real link flap incident shows multiple transitions, the boot-time switch shows exactly 1.

- **Pre-v0.24.4 clusters ship dashboards filtered by `instance`, which is permanently broken on multi-node azcluster deploys.** Every compute node scrapes its own colocated Prometheus at `127.0.0.1:9100` so the `instance` label is identically `127.0.0.1:9100` across the fleet — the `instance=~"$instance"` filter the dashboards used was useless for cross-node disambiguation. v0.24.4 switched all dashboards to filter by `nodename` (set via Prometheus's `external_labels.nodename = $(hostname)`). Pre-v0.24.4 clusters either need to redeploy on v0.24.4+ OR `az grafana dashboard create --definition @grafana/dashboards/<name>.json --overwrite` against the live AMG to re-import the fixed JSONs. v0.24.5+ `azcluster deploy --skip-arm` is the canonical path: cluster stays as-is, post-deploy hooks rerun, dashboards get the fixed `nodename` filter.

## v0.24.11 gotchas (apt-lists lock + prolog parent dir)

- **`-o DPkg::Lock::Timeout=600` does NOT cover `/var/lib/apt/lists/lock`** (v0.24.11 fix). The flag only handles the **dpkg** locks (`/var/lib/dpkg/lock-frontend`, `/var/lib/dpkg/lock`). apt's own metadata lock at `/var/lib/apt/lists/lock` (held by `apt-get update`, `apt-cache`, and during sources-list refresh inside `unattended-upgrade`) is NOT covered — and that's what bit v2411walk on `eastus`: the v0.24.7 `bootcmd` `pkill` of `unattended-upgrade` killed the *first* invocation, but a NEW apt process (kernel-update post-install trigger) started after bootcmd and was holding apt-lists lock when `install-compute.sh` ran. Symptom: `E: Could not get lock /var/lib/apt/lists/lock. It is held by process N (apt)` mid-script, `set -e` aborts, node never reaches `touch /var/log/azcluster/ready`, slurmd never registers. Fix: every install script (compute, login, scheduler) now defines `apt_wait()` (fuser-polls all 3 locks: dpkg-frontend, dpkg, apt-lists; 10-min cap, 5s interval) + an `aptget()` wrapper called instead of bare `apt-get`. Generalisable rule: **any cloud-init script that runs apt-get MUST go through a wrapper that polls fuser on `/var/lib/apt/lists/lock` as well, not just dpkg-frontend.**

- **`install -d <leaf>` does NOT auto-create parent directories** (v0.24.11 fix). Coreutils `install` advertises mkdir-like behaviour but only for the leaf directory specified — if intermediate parents are missing, it errors with `install: cannot create directory '/parent/leaf': No such file or directory`. The Slurm prolog `/etc/slurm/prolog.d/10-azcluster-user-nvme.sh` called `install -d -o $UID -g $GID -m 0700 /mnt/nvme/users/${SLURM_JOB_USER}` on every job. cloud-init creates `/mnt/nvme` (the RAID-0 mountpoint) but not `/mnt/nvme/users/`, so the FIRST prolog invocation on a freshly-booted compute node failed, Slurm marked the node `DRAIN reason=Prolog error`, and the operator had to manually `scontrol update nodename=<n> state=resume`. Fix: prepend `mkdir -p /mnt/nvme/users && chmod 0755 /mnt/nvme/users` to the prolog, then call `install -d` for the per-user leaf. Distinct from `mkdir -p` which DOES auto-create parents. Live-reproduced on v2411walk gpu-0003.

## v0.24.12 gotchas (install-script idempotency via runcmd retry loop)

- **Install scripts under `set -euo pipefail` will silently abort on ANY transient failure**, not just the apt-lock contention v0.24.11 fixed. v2411walk's second deploy reproduced this with a DIFFERENT abort point each time: install-scheduler.sh died between the azcluster-server tarball SHA verification and slapd install (probably IMDS hiccup on the `RG=$(curl IMDS resourceGroupName)` call at line 591); install-compute.sh on gpu-0003 died between `systemctl enable prometheus` and the prolog write at line 798 (unknown root cause; possibly cloud-init timeout or transient apt). cloud-init reports `status: done` because the *cloud-init* phase completed — it doesn't track the inner script's exit code through `tee`. v0.24.12 wraps each install-script invocation in cloud-init `runcmd` with a 10-attempt retry loop (30 s gap, 5 min total cap). Each retry re-runs the script; the script exits 0 fast at the top if `/var/log/azcluster/ready` already exists. Apt installs are naturally idempotent, useradd/groupadd already use `getent || useradd`, mkdir is `-p`, ln -s is guarded with `[ -L ... ] ||` — so re-runs are safe. Generalisable rule: any cloud-init script doing multi-step setup MUST be wrapped in a retry loop AND have an early-exit at the top, because there is no other mechanism for cloud-init to retry on inner-script failure.

## v0.24.15 gotchas (cluster-internal SSH keypair per LDAP user)

- **Goal**: every LDAP user (default `clusteradmin`/`clusteruser` + every `azcluster user add`) gets a node-local ed25519 keypair so inter-node ssh works WITHOUT the operator's laptop key being on the cluster. Implementation: a SECOND value is added to the user's `sshPublicKey` LDAP attribute, prefixed with `from="10.42.0.0/16,127.0.0.1/32" ` (VNet + loopback). `AuthorizedKeysCommand /usr/bin/sss_ssh_authorizedkeys` on login + compute returns BOTH values to sshd; sshd honours the `from=` restriction per authorized_keys(5), so this key cannot be used from outside the VNet even if exfiltrated.
- **Private key lives at `/shared/home/<user>/.ssh/id_ed25519`** (ANF or scheduler-NFS). Mode 0700 dir / 0600 priv / 0644 pub, chown'd to the user uid/gid. The pubkey is ALSO appended to `~/.ssh/authorized_keys` so the same user can ssh from one node to another using the home-dir-resident key without needing the LDAP-served value (saves the SSSD round-trip and works even if SSSD cache is cold).
- **`install -d` does NOT auto-create parents** (v0.24.11 lesson, applies here too). The provisioning script does `install -d -m 0755 /shared/home` BEFORE the per-user `install -d -m 0700 -o $UID -g $GID /shared/home/<u>` because ANF's default `/shared` permissions are `0755 root:root` but the `/shared/home` subdir may not exist on a fresh deploy until the FIRST user is added.
- **`sudo -u \#UID -g \#GID ssh-keygen`** works on login + compute (SSSD clients can resolve LDAP uids via NSS) but **DOES NOT WORK on the scheduler** — slapd is local but no NSS client resolves LDAP uids, so `sudo -u '#20001'` errors with `sudo: unknown user #20001` (v0.24.16 live-caught + fixed). On the scheduler, run `ssh-keygen` as root then `chown $UID:$GID` + explicit `chmod 0600/0644`. This is the only difference between the scheduler bootstrap loop (`cloud-init/scheduler.yaml.tmpl`) and the operator `azcluster user add` path (`crates/azcluster-cli/src/user.rs::build_internal_keypair_provision_cmd`) — both now use the root + chown pattern.
- **`from="..."` restriction in LDAP `sshPublicKey`**: the LDIF value is `from="10.42.0.0/16,127.0.0.1/32" ssh-ed25519 AAAA... azcluster-internal-<cluster>-<user>`. ldapsearch returns the value verbatim; sss_ssh_authorizedkeys passes it through; sshd parses authorized_keys(5) options. Verified semantics — ssh from a source IP outside the listed CIDRs fails on key validation.
- **Idempotency**: scheduler bootstrap re-runs (v0.24.12 retry loop) skip keygen if `/shared/home/<u>/.ssh/id_ed25519` exists, and skip the ldapmodify add if `ldapsearch` already shows the pubkey in `sshPublicKey`. `azcluster user add` runs the same provisioning step after the main `ldapadd` succeeds; the user is freshly created so neither file nor LDAP value exists yet, but the same guards make the call safe to retry.
- **Generalisable rule**: any future LDAP user-management code that adds a `sshPublicKey` value MUST consider whether to apply a `from=` restriction. Operator-supplied keys (via `--ssh-key`) are NOT restricted because the operator may want to ssh from their laptop's public IP. Cluster-internal keys ARE restricted because there is no legitimate reason for a cluster-generated keypair to authenticate from outside the VNet.
- **`azcluster user sshkey remove` is currently a footgun for the internal key**: the LDAP-served value is `from="10.42.0.0/16,127.0.0.1/32" ssh-ed25519 AAAA... azcluster-internal-...`. An operator who runs `azcluster user sshkey list` and then `sshkey remove` with the FULL value will delete the internal key as well as any operator-supplied keys. The on-disk `/shared/home/<u>/.ssh/{id_ed25519,authorized_keys}` survive, so inter-node ssh keeps working via authorized_keys, but inter-node SSSD-served auth breaks until the user is re-added. Documented footgun, not a v0.24.15 bug. Future fix: filter out `azcluster-internal-*` keys from `sshkey list` output OR add a `--include-internal` flag.

### AKS target live-validation findings (first clean run: 2× Standard_ND96isr_H200_v5 / mexicocentral, NCCL all-reduce **481.5 GB/s busbw**)

These are the bugs surfaced and fixed during the first end-to-end live `azcluster deploy --target aks` + `azcluster validate`. Every one was invisible to `cargo test` / `az bicep build` and only appeared on real H200 hardware.

- **Do NOT pin `kubernetesVersion` to a minor.** Pinning `1.30` resolved to `1.30.14`, which mexicocentral now offers **LTS-only** → `K8sVersionNotSupported` preflight failure. Fix: `kubernetesVersion` defaults to `''` in `bicep/aks-main.bicep`, and `modules/aks.bicep` omits the property via `union(empty(kubernetesVersion) ? {} : {kubernetesVersion: ...}, {...})` so AKS picks the region-default GA version (got `1.34.8`). The CLI passes `""`.
- **AKS `runCommand` returns `202` with an EMPTY body + a `Location` header** pointing at `.../commandResults/{id}`. Parsing the 202 body as JSON fails (`EOF while parsing a value at line 1 column 0`). `ArmClient::aks_run_command` reads the result URL from the `Location`/`Azure-AsyncOperation` header and polls it leniently (tolerating bodyless 202s while running).
- **`runCommand` executes the script under `/bin/sh` (dash), NOT bash.** `set -o pipefail` aborts with `set: Illegal option -o pipefail` (exit 2). All operator + validate scripts use `set -eu` (POSIX) and avoid bashisms (`$(seq ...)`, `[ ]`, `case`, param-expansion are fine).
- **`kubectl rollout status` does NOT accept `--all`** (only `kubectl wait` does). The Network/GPU operator stages rely on their own readiness loops (NicClusterPolicy `.status.state==ready`; node `nvidia.com/gpu` allocatable) instead.
- **The NFD `NodeFeatureRule` must select IB NICs by PCI class `0207` (InfiniBand), not device IDs, AND the GPU pool must NOT be tainted.** Two compounding bugs: (1) the reference rule matched device IDs `101c/101e` (ConnectX-6) but ND H200 v5 IB NICs are `101e` *and* class `0207`, while the Azure AccelNet Ethernet VF on every node (incl. system nodes) is vendor `15b3` device `1018` class `0200` — so a device-ID or vendor-only match either misses the IB NICs or mislabels system nodes. Match `vendor In [15b3]` **AND** `class In [0207]` to label exactly the 8 IB NICs on GPU nodes. (2) the GPU pool had `nodeTaints: ['sku=gpu:NoSchedule']`, which the MOFED/SR-IOV/GPU-operator DaemonSets do not tolerate → they only scheduled on (untainted) system nodes. Removed the taint. Symptom of both: NicClusterPolicy `notReady`, `rdma/ib` never advertised. After the fix: `rdma/ib: 8` allocatable per GPU node, NicClusterPolicy `ready`.
- **On AKS the IB devices are named `mlx5_0..7`, NOT the Slurm/Pyxis `mlx5_ib0..7`.** Setting `NCCL_IB_HCA=mlx5_ib` matches nothing → NCCL reports `NET/IB : No device found` and **silently falls back to `NET/Socket eth0`** (the recurring v0.13.8/v0.19.4 TCP-fallback class). Fix: do NOT set `NCCL_IB_HCA`; let NCCL auto-detect the `mlx5_*` IB devices. The `validate` log parser counts both `mlx5_N` and `mlx5_ibN` tokens in `NET/IB : Using .../IB/SHARP` lines.
- **The MPIJob launcher hits a worker-DNS race.** `launcherCreationPolicy: WaitForWorkersReady` starts the launcher when worker pods are *Running*, but CoreDNS endpoint propagation for the headless-service hostnames lags a few seconds → `mpirun` fails in ~3 s with `ssh: Could not resolve hostname <mpijob>-worker-0.<mpijob>.default.svc`. The launcher command **must** loop on `getent hosts` for every `/etc/mpi/hostfile` entry before invoking `mpirun`.
- **Worker pods need a large `/dev/shm`.** Default pod shm is 64 MB → NCCL aborts at scale with `failed to extend /dev/shm/nccl-* : No space left on device`. Mount `emptyDir{medium: Memory}` at `/dev/shm` on the workers (gives a memory-backed tmpfs sized to the node).
- **Use `all_reduce_perf_mpi` with the FULL NGC env, and `-mca coll_hcoll_enable 0`.** In `ghcr.io/azure/ai-infrastructure-on-azure/nccl-test`, the MPI-aggregating binary is the distributed exec; without the reference env (esp. `UCX_TLS=rc`, `UCX_NET_DEVICES=mlx5_0:1`, `NCCL_TOPO_FILE=/etc/topology/ndv5-topo.xml`, the SHARP vars) and `mpirun -mca coll_hcoll_enable 0`, it **segfaults**. The plain `all_reduce_perf` reports `busbw 0` (runs as nranks=1 per process). The full recipe lives in `crates/azcluster-cli/src/aks/manifests/nccl-validate-mpijob.yaml`, mirroring `infrastructure_validations/aks/NCCL/helm/nccl-test` from the reference repo. Result on 2× ND H200 v5: **481.5 GB/s avg busbw**, all 8 `mlx5_0..7` IB/SHARP devices, no TCP fallback — matching/exceeding the Slurm H100 baseline (440 GB/s).
- **Debugging tip:** `az aks command invoke` is slow/flaky for iterative debugging. Pull the admin kubeconfig (`az aks get-credentials --admin`) — the API server is public and `disableLocalAccounts: false` — and use `kubectl` directly. To inspect a fast-failing launcher, wrap its command as `bash -c '<cmd>; sleep 3600'` so the pod persists for `kubectl logs`/`exec`.

Dashboards ship as a separate release artifact `azcluster-dashboards-${VERSION}.tar.gz` (packaged by the `assets` job in `.github/workflows/release.yml`, listed in the same `SHA256SUMS`), fetched by `cloud-init/scheduler.yaml.tmpl` from `https://github.com/${AZCLUSTER_REPO}/releases/download/${AZCLUSTER_VERSION}/`. **Key implication for testing**: the release tarball MUST exist publicly before any test deploy can succeed — push a pre-release tag (`vX.Y.Z-pre.N`) on the feature branch, wait ~10 min for the `release.yml` workflow, then `azcluster deploy --azcluster-version vX.Y.Z-pre.N`. **Implication for operators**: client no longer needs `Grafana Admin` on the AMG; only KV Secrets Officer (already granted) and ARM contributor. The CLI's `import_dashboards`, `ensure_grafana_folder`, `get_grafana_token`, and `DASHBOARDS` const are removed; `finalize_deploy` keeps a 3-line load-bearing comment in the empty monitoring branch documenting why it's empty to prevent regression.

- **`bicep/main.json` is no longer committed — it is generated at build time and the Rust build fails loudly if it is missing.** Earlier (v0.20.0) `main.json` was committed with a `.gitignore` exception so that `cargo build`/`cargo test` didn't require Bicep, and a CI `bicep` job `diff`ed the regenerated file against the committed copy to catch drift. That drift check was permanently red on a cosmetic `_generator.version` + `templateHash` stamp whenever the runner's Bicep version differed from whatever produced the committed file. The de-commit removes that whole failure class: (1) `.gitignore` no longer has `!bicep/main.json`, so `bicep/**/*.json` ignores it; (2) `git rm --cached bicep/main.json` untracked it (file stays on disk locally); (3) new `crates/azcluster-cli/build.rs` checks `$CARGO_MANIFEST_DIR/../../bicep/main.json` exists and `panic!`s with `az bicep build --file bicep/main.bicep --outfile bicep/main.json` (and the standalone-`bicep` equivalent) instructions if absent, plus `cargo:rerun-if-changed` on the path; (4) the CI drift `diff` step was deleted. The CLI still embeds the file via `include_str!("../../../bicep/main.json")` at `crates/azcluster-cli/src/main.rs:830` — `build.rs` only guards its presence, it does NOT run Bicep itself (kept dependency-free and deterministic). **Who needs Bicep now**: only people building the CLI from source. End users installing the prebuilt release tarball get the template already baked into the binary and need neither `az` nor Bicep. Runtime is still 100% `az`-free.
- **CI/release regenerate `main.json` from a PINNED Bicep before every `cargo build`.** `ci.yml`'s `rust` job and `release.yml`'s `build` + `assets` jobs each run `az bicep install --version ${BICEP_VERSION}` + `az bicep build --file bicep/main.bicep --outfile bicep/main.json` after checkout, before any cargo step. `BICEP_VERSION` is pinned to `v0.43.8` in all four job scopes — keep them in lockstep when bumping. The `bicep` lint job still installs the pinned Bicep and builds every `bicep/*.bicep` for syntax, but no longer diffs `main.json` against a committed copy (there isn't one).
- **Regenerating at a newer Bicep only changes two metadata fields.** Going `0.41.2` → `0.43.8` rewrote ONLY `_generator.version` + `templateHash`; zero resource/template content diff (verified). So a contributor on a slightly different Bicep patch will produce a byte-different-but-semantically-identical `main.json` — which is exactly why committing it was painful and why de-committing is the right call. When bumping the pinned `BICEP_VERSION`, expect (and ignore) those two fields shifting.
- **`build.rs` was added with no `Cargo.toml` change** — cargo auto-detects `crates/azcluster-cli/build.rs`. The 3-line doc comment at the top is a deliberate AGENTS.md comment-discipline exception (it documents the non-obvious "why does the CLI need a build script" invariant). Don't delete it.

### v0.24.20 live-walkthrough findings (DGXC Megatron-Bridge training + matched-params container NCCL)

Captured from a clean 2-node ND96isr_H100_v5 eastus run (`v2420walk`, 2026-06-08); see `doc/full-walkthrough-v0.24.20.md`.

- **DGXC training uses Megatron-Bridge, NOT NeMo — different log format AND a different parser.** The DGXC training container `nvcr.io/nvidia/nemo:26.04.00` runs Megatron-Bridge, which logs per-iteration timing as `Step Time : <N>s GPU utilization: <T>MODEL_TFLOP/s/GPU` followed by ` [<ts>] iteration <i>/<N> | ... | elapsed time per iteration (ms): <ms> | ... | lm loss: <loss> | ...`, NOT NeMo's `train_step_timing in s:`. The matching parser is `<LLMB_INSTALL>/llmb_repo/common/parse_train_timing_mbridge.sh` (the `_mbridge` suffix is the tell — distinct from NeMo's `parse_train_timing.sh`), and it takes the **experiments DIRECTORY** as its argument (recurses for `log-*.out`), averaging iterations 35-44. Pointing it at a single log file returns nothing. Verified: Llama-3.1-8B BF16 tp1/pp1/cp2/mbs1 hit **541.81 MODEL_TFLOP/s/GPU** (16-GPU/gbs256) and **542.66** (8-GPU/gbs128), both Success — ~99.8% per-GPU scaling efficiency single→2-node.
- **DGXC pretrain uses MOCK token data — lm loss converges absurdly fast; do not read it as a real-training signal.** lm loss falls from `1.259187E+01` (iteration 1) to `1.976016E-02` (iteration 50) in 50 steps because the dataset is synthetic. The headline metric is MODEL_TFLOP/s/GPU + step-time stability (σ ≈ 13-20 ms over 10 sampled iters), NOT the loss curve.
- **Container NCCL reaches bare-metal IB ONLY when the benchmark params MATCH plain-VM.** The v0.24.12 walkthrough compared a 16 GiB×10-iter plain-VM run against a 1 GiB×20-iter container smoke — not apples-to-apples, which made the plain (461) vs container (428) numbers look like a container penalty. v2420walk ran BOTH with identical `all_reduce_perf -b 16G -e 16G -f 2 -g 1 -N 10` (16 ranks / 2 nodes, `srun --mpi=pmix --ntasks=16 --ntasks-per-node=8`): plain VM **440.21 GB/s** avg busbw (NCCL RDMA Plugin v11, in-image HPC-X), container `nvcr.io/nvidia/nemo:25.07.02` **451.079 GB/s** (NCCL RDMA Plugin v10). Both negotiate all 8 `mlx5_ib{0..7}:1/IB/SHARP` and peak ~466.8 GB/s on the 16 GiB row; the ~2.5% spread is ramp noise, NOT a TCP fallback. Container prerequisites are the carry-forward v0.13.8 set: `MELLANOX_VISIBLE_DEVICES=all` + enroot `99-mellanox.sh` hook + `--container-mounts=/shared:/shared,/mnt/nvme:/mnt/nvme`. Note the two NCCL RDMA plugin **minor versions differ** (v11 vs v10) yet both reach SHARP — plugin minor version is not the determinant of fabric performance, so don't chase it when a container underperforms; check IB device visibility (`ibv_devinfo` inside the container) first.
- **`run-dsr1.sh` (DGXC DeepSeek-R1 SGLang launcher) hardcodes `HEAD_IP` to the first compute node's IP.** On v2420walk that was `10.42.4.6` (gpu-0003 = first node in the allocation = NODEID0). If the hardcoded IP doesn't match the actual first compute node, the multi-node SGLang TP=16 rendezvous never forms. Operators copying the example MUST edit `HEAD_IP` (or derive it: `scontrol show hostnames "$SLURM_JOB_NODELIST" | head -1` resolved via VNet DNS). Separately, multi-node SGLang shutdown surfaces a **non-zero worker exit on teardown** — the bench JSON is written BEFORE shutdown, so a non-zero sbatch ExitCode does NOT mean the benchmark failed; verify by the presence of the `dsr1-fp8-h100-tp16-c64.json` output. On v2420walk job 25 OOM'd (ExitCode 137) during conc tuning; job 26 completed clean: **487.81 tok/s output, 123.34 ms median TPOT, 640/640 requests** at conc=64.
