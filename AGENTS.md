# AGENTS.md

Instructions for AI agents working on this repository. Keep this file current.

## Documentation Discipline (MANDATORY)

Every change MUST update three artifacts in lockstep:

1. **`CHANGELOG.md`** — add entry under `## [Unreleased]` for every user-visible or operator-visible change. On tag, rename `Unreleased` to the new version with the release date, and open a fresh `Unreleased` section. Categories: `Added`, `Changed`, `Fixed`, `Removed`, `Deprecated`, `Security`.
2. **`README.md`** — keep `Status`, `Quickstart`, `Architecture`, `Repo Layout` honest. If a phase ships, update the Status block. If a flag/command changes, update the Quickstart.
3. **`AGENTS.md`** (this file) — when a new workflow, convention, or guard-rail emerges, add it here so the next agent inherits it.

If a PR touches code but skips any of these three, it is incomplete.

## Release Workflow

1. Land all `Unreleased` work; verify `cargo fmt && cargo clippy -- -D warnings && cargo test --workspace` and `az bicep build` clean across all modules.
2. Edit `CHANGELOG.md`: rename `## [Unreleased]` → `## [X.Y.Z] - YYYY-MM-DD`, add a fresh empty `## [Unreleased]` block at the top, update the link references at the bottom.
3. Edit `README.md` Status block if phase advanced.
4. Bump version: `crates/*/Cargo.toml`, the `--azcluster-version` CLI default in `crates/azcluster-cli/src/main.rs`.
5. Commit, tag `vX.Y.Z`, push tag — CI publishes the release.
6. Live-validate on `paul-azcluster`/`southafricanorth` before declaring done.

## Hard Rules (carried from earlier sessions)

- **No CycleCloud / Jetpack / CCWS references** in any public artifact (README, code, commits, tags, release notes). Past mentions were already scrubbed; don't reintroduce them.
- **No personal identifiers or tenant-specific values** in committed files. Subscription IDs, RG names beyond the documented `paul-azcluster`, emails, etc. stay out of git.
- **Public IPs off by default**; opt-in via `--login-public-ip`.
- **All code Rust** except Bicep + bootstrap shell in cloud-init.
- **Research checkouts** live under `research/` (gitignored). Planning artifacts under `.sisyphus/` (gitignored).
- **Minimize comments**; let code be self-documenting. Necessary exceptions: clap doc-comments (they render as `--help` text), complex algorithms, security-critical sections.
- **Never suppress type errors** (`as any`, `#[allow]` blanket, `unwrap()` on user input paths).
- **Never commit** unless the user explicitly asks.
- **Autonomous versioning is enabled** (user directive, AFK): when a coherent unit of work is complete, verified, and changelogged, the agent SHOULD bump the version, commit, tag, and push. Do not wait for explicit per-release permission. Apply SemVer: feature batch → minor; bugfix-only → patch; breaking → major.

## Azure / Infra Gotchas

- AzSecPack policy auto-attaches `AzSecPackAutoConfigUA-<region>` UAI to every VM/VMSS. Any PUT on a VMSS `identity` collection triggers `LinkedAuthorizationFailed` unless the caller has `Managed Identity Operator` on that UAI (out-of-band). **Use `az vmss scale --new-capacity`, not `az vmss update --set sku.capacity`.**
- Slurm configless mode does **not** distribute `plugstack.conf.d/`. Write `/etc/slurm/plugstack.conf` (absolute path) on every node that uses Pyxis (compute + login).
- Enroot needs `1777` on `/var/lib/enroot{,/runtime}`, `/var/lib/enroot-data{,/cache}`, and `/run/enroot` — top-level AND subdirs.
- VMSS Flex VMs surface as `Microsoft.Compute/virtualMachines` named `vmss-<cluster>-<pool>_<hex>`, not under the VMSS resource.
- Image: `microsoft-dsvm:ubuntu-hpc:2404` (default), `2204` fallback.
- Slurm 25.11 + Pyxis 0.21.0 ABI match. NVIDIA Pyxis 0.24.0 exists; only bump if Slurm 26 is needed.
- Slurm 25.11 dynamic nodes: `slurmd --conf "...Partitions=<pool>"` is rejected ("Failed to parse nodeline"). Use the **NodeSet+Feature** pattern: emit `NodeSet=<pool>set Feature=pool_<pool>` and `PartitionName=<pool> Nodes=<pool>set ...` in `slurm.conf`; the compute slurmd registers with `--conf "...Feature=pool_<pool>"` and slurmctld places it in the matching NodeSet/partition.
- Pyxis spank library (`spank_pyxis.so`) must be installed on **every** node that may submit `srun` — scheduler, login, and compute — because `plugstack.conf` loads at submit time. Forgetting it on scheduler crashes any `srun` invoked from there with `Dlopen of plugin file failed`.
- The `microsoft-dsvm:ubuntu-hpc` image ships `nvidia-smi` even on CPU SKUs, so `command -v nvidia-smi` cannot be used as a GPU presence check. Use `nvidia-smi -L | grep -cE '^GPU [0-9]+:' || true` instead.
- **Managed Grafana region coverage**: `Microsoft.Dashboard/grafana` is NOT available in `southafricanorth` (and several other regions). Use `--grafana-location uksouth` (or another supported region) when the cluster region lacks AMG.
- **Monitoring Data Reader role GUID** is `b0d8363b-8ddd-447d-831f-62ca05bff136` (NOT the `...51537` value some docs list). Verify role GUIDs with `az role definition list --name "..."` before baking into Bicep.
- **AMW ingestion RBAC scope**: `Monitoring Metrics Publisher` MUST be granted on the AMW's **default Data Collection Rule**, not on the AMW account itself. Role at the AMW scope passes role-listing checks but the ingestion endpoint still returns `403 The authentication token provided does not have access to ingest data for the data collection rule with immutable Id 'dcr-...'`. The default DCR lives in the Azure-managed sister RG `MA_<amwName>_<location>_managed`.
- **AMW ingestion RBAC propagation**: 5-10 minutes on a freshly-created MI + DCR. After it propagates, `systemctl restart prometheus` is required - prometheus and IMDS cache the bearer token, and the cached token's authorization is decided server-side at request time but the connection state appears to survive the role landing. A direct `curl` test (`POST` empty body to the ingestion URL with an IMDS token) is the fastest way to confirm whether the failure is RBAC vs config.
- **Prometheus 3.x cloud-init perms**: `cat > /opt/prometheus/prometheus.yml <<EOF ... EOF` in a cloud-init `runcmd` produces a file with mode `0600` (root's umask). The non-root `prometheus` service user cannot read it -> "permission denied" at startup. Always follow with explicit `chmod 0644`.
- **Slurm conf files cloud-init perms** (v0.13.9 fix). Same root cause as the Prometheus gotcha above: cloud-init `runcmd` inherits a 0077 umask so `cat > /etc/slurm/slurm.conf <<EOF` produces 0600 files. `srun`/`sinfo` run as the submitting (non-root) user and parse `/etc/slurm/slurm.conf` + `/etc/slurm/plugstack.conf` locally on the submit host — Pyxis spank plugins load at submit time, not exec time. With 0600 perms users get `error: s_p_parse_file: unable to read "/etc/slurm/slurm.conf": Permission denied` and `error: spank: Failed to open /etc/slurm/plugstack.conf: Permission denied`. Always `chmod 0644 /etc/slurm/*.conf` after writing them in cloud-init. Applies to scheduler (slurm.conf + plugstack.conf + cgroup.conf + gres.conf), login (plugstack.conf — slurm.conf comes via sackd configless), and compute (plugstack.conf — slurm.conf + gres.conf come via slurmd configless cache).
- **Compute `/etc/hosts` 127.0.1.1 breaks cross-node Gloo/PyTorch rendezvous** (v0.13.9 fix). The Ubuntu cloud-image default `/etc/hosts` has `127.0.1.1 <hostname>` after `hostnamectl set-hostname`. `torch.distributed.new_group(backend="gloo")` (Megatron-Bridge uses this for CP groups during init) calls `gethostbyname(hostname)` to learn its own advertised address, and Gloo `connectFullMesh` then expects every other rank to dial that address. With `127.0.1.1` every remote rank dials its own loopback and the rendezvous stalls with `Gloo connectFullMesh failed ... timed out connecting: SO_ERROR: Connection refused, remote=[127.0.1.1]:20901`. NCCL is unaffected (rendezvous is via `MASTER_ADDR`, set explicitly by Megatron-Bridge's launcher) — the failure is at the PyTorch-level Gloo PG used for control-plane groups. Live-validated on `paul-azcluster-h100d` Llama 3.1 8B 2-node BF16 (job 16 failed pre-fix, job 20 succeeded post-fix). Fix in `cloud-init/compute.yaml.tmpl`: after `sed -i '/^127\.0\.1\.1/d' /etc/hosts`, write `${IPV4} ${NEW_HOSTNAME}` (eth0 IP, NOT 127.0.1.1). Scheduler/login don't need this — slurm.conf resolves them via Azure DNS suffix.
- **Prometheus `azuread.managed_identity` remote_write**: works directly against the AMW DCE ingestion URL (`${dceEndpoint}/dataCollectionRules/${dcrImmutableId}/streams/Microsoft-PrometheusMetrics/api/v1/write?api-version=2023-04-24`). No AMA or DCR custom scrape config needed for VMs. Block shape:
  ```yaml
  remote_write:
    - url: "..."
      azuread:
        cloud: AzurePublic
        managed_identity:
          client_id: "<uai-client-id>"
  ```
  Audience is implicit (`https://monitor.azure.com/`). The UAI MUST be attached to the VM AND hold Metrics Publisher on the DCR (see above).
- **VMSS Flex + SystemAssigned identity** is rejected in subscriptions with AzSecPack/UAI-only policy (`InvalidParameter` on `identity`). VMs are fine; VMSS must use UserAssigned (or no MI).
- **Grafana dashboard JSON** lives under `grafana/dashboards/` and is provisioned by `bicep/modules/grafana-dashboards.bicep`. Keep panel ids stable across edits, use a datasource template variable named `DS_PROMETHEUS`, and never hardcode Azure Managed Grafana datasource UIDs.
- Phase 1+ test region: `southafricanorth`, RG `paul-azcluster`, max 2 GPU nodes.

If any of these fail, fix before claiming completion.
- **YAML write_files heredoc terminators**: cloud-init `write_files` with `content: |` block scalars strips the FIRST line's leading whitespace as the indent and removes that prefix from every subsequent line. Any `EOF` terminator MUST sit at exactly that base indent column in the template so it lands at column 0 in the materialised script - bash heredocs without `<<-` only recognise the closer at column 0. Extra indentation from being nested inside `if`/`for` (e.g. 6-space base + 2-space if-body = 8-space `EOF`) silently breaks the heredoc and chains errors until bash hits EOF with "syntax error: unexpected end of file". Always put `EOF` at the template's base indent column even inside `if` blocks.
- **AMG Grafana Admin for dashboard import**: `az grafana dashboard create` calls the Grafana HTTP API (`POST /api/dashboards/db`), which requires a Grafana role on the AMG. ARM contributor / owner is not enough. Grant the deployer principal `Grafana Admin` (`22926164-76b3-42b3-bc55-97df8dab3e41`) on the AMG resource scope, and expect 1-3 min propagation - retry with back-off on `401 NoRoleAssignedException`.
- **`Microsoft.Dashboard/grafana/dashboards@*` is NOT a real ARM resource**: BCP081 ("resource type does not have types available") is the warning, and ARM preflight returns `ResourceTypeRegistrationNotFound`. Import dashboards via the Grafana HTTP API (`az grafana dashboard create` or direct REST), not Bicep. The CLI does this post-deploy with the dashboard JSONs embedded via `include_str!`.
- **Shared filesystem modes**: `--shared-storage anf` (default) provisions Azure NetApp Files; `--shared-storage nfs-scheduler` exports `/shared` from the scheduler via `nfs-kernel-server`. The latter is test-only (no HA, SPOF on scheduler) but saves ~12 min off provisioning. Login + compute always mount with a 5-minute retry loop because the scheduler is still installing its NFS server while they boot in `nfs-scheduler` mode.
- **Deployment timing**: every successful `azcluster deploy` writes `~/.config/azcluster/deployments/<cluster>/<utc-stamp>.json` with per-resource `duration_seconds` and a `trend.tsv` for comparison. Inspect via `azcluster timings <cluster>`. Source: `az deployment operation {sub,group} list` recursed across nested modules; `properties.duration` parsed from ISO-8601 (`PT##H##M##S`).
- **`--no-monitoring` / `--no-accounting`**: monitoring + accounting default to ON. Pass these flags to skip them during rapid test deploys that don't depend on observability or accounting. Accounting (v0.13.0+) provisions an Azure Database for MySQL Flexible Server + runs `slurmdbd` on the scheduler.
- **MySQL Flex TLS CA**: Azure MySQL Flexible Server presents a DigiCert Global Root CA chain, which is **already in Ubuntu's `ca-certificates`** bundle. Set `StorageParameters=SSL_CA=/etc/ssl/certs/ca-certificates.crt` in `slurmdbd.conf`; do NOT try to download `https://dl.cacerts.digicert.com/DigiCertGlobalRootCA.crt.pem` — that host returns a cert whose SAN doesn't match the hostname and `curl` aborts with `SSL: no alternative certificate subject name matches target host name`. v0.13.0/v0.13.1 had this bug; fixed in v0.13.2.
- **Slurm accounting (v0.13.0)**: MySQL Flexible Server (`mysql-<cluster>`, `Standard_B2ms`, MySQL 8.0.21, 50 GB autogrow, `publicNetworkAccess=Disabled`) is provisioned in a delegated subnet `10.42.8.0/29` (delegation `Microsoft.DBforMySQL/flexibleServers`). The CLI auto-generates the admin password from `/dev/urandom` (32 random chars from an ambiguity-free alphabet plus literal `Aa1!` to satisfy Azure's "≥3 character classes" policy — Azure rejects passwords with only 2 classes) and passes it as a `@secure()` Bicep parameter. On the scheduler, the password lands in `/etc/azcluster/accounting.password` (mode 0600 root:root via cloud-init `write_files`) and is read into `slurmdbd.conf` (mode 0600 slurm:slurm) at boot, then the shell var is `unset`. TLS is required: the bootstrap downloads `https://dl.cacerts.digicert.com/DigiCertGlobalRootCA.crt.pem` to `/etc/slurm/ssl/` and `slurmdbd.conf` sets `StorageParameters=SSL_CA=…`. Bootstrap order MUST be: install `slurm-smd-slurmdbd` → write `slurmdbd.conf` → wait for `:3306` reachability (60×5s probe via `/dev/tcp`) → start `slurmdbd` → sleep 5s → start `slurmctld` → `sacctmgr -i add cluster`. Starting `slurmctld` before `slurmdbd` with `AccountingStorageEnforce=associations,limits,qos` set will cause slurmctld to refuse to start. Three MySQL server params are tuned per slurmdbd recommendations: `innodb_lock_wait_timeout=900`, `max_allowed_packet=16777216`, `log_bin_trust_function_creators=ON`.
- **`aka.ms/InstallAzureCLIDeb` is FORBIDDEN inside `set -euo pipefail` bootstraps** (v0.13.1 fix). The pipe script runs `apt-get install` internally without our `DPkg::Lock::Timeout=600`, so it races `apt-daily`/`unattended-upgrades` and aborts the entire bootstrap with `Could not get lock /var/lib/dpkg/lock-frontend`. Cloud-init still reports "finished" because runcmd pipes through `tee`. Always install azure-cli via the explicit MS apt repo (`packages.microsoft.com/repos/azure-cli/`) using our own `apt-get -o DPkg::Lock::Timeout=600 install -y azure-cli`. The MS GPG key (`/usr/share/keyrings/microsoft-prod.gpg`) is already imported earlier in the scheduler bootstrap for the slurm repo, so reuse it.
- **`tee` masks `runcmd` exit codes.** The compute/scheduler bootstraps run `/opt/azcluster/install-X.sh 2>&1 | tee -a /var/log/azcluster/install.log`. If the script aborts with a non-zero exit, the pipe still succeeds because `tee` (the last stage) returns 0. cloud-init then reports `finished` and you get a half-configured node that registers (or doesn't) with Slurm in a broken state. To diagnose, always `tail /var/log/azcluster/install.log` on the node and grep for `Error|fail|exit` rather than trusting `cloud-init status`.
- **`set -euo pipefail` + bash command substitution + pipelines is a sharp edge** (v0.13.4 fix). `drv=$(ethtool -i lo | awk ...)` aborts the whole script when the `awk` pattern matches nothing on a virtual interface, because the pipe's exit status under `pipefail` is non-zero and `set -e` kills the script. Always add `|| true` *inside* the `$(...)` for diagnostic pipes whose failure is expected and tolerated.
- **Slurm configless distribution does NOT push `gres.conf`** (v0.13.4 fix). `SlurmctldParameters=enable_configless` only distributes `slurm.conf`. Both scheduler (for slurmctld's `GresTypes=` validation) and every compute node (for slurmd's local Gres discovery) must have their own `/etc/slurm/gres.conf`. The simplest version that works on Azure is `AutoDetect=nvml` on both sides — slurm-smd is built with NVML support, and slurmctld accepts NVML autodetect even though it has no GPUs itself.
- **Slurm caches a node's first registration** (v0.13.4 lesson). If a slurmd registers once with bad Gres/Parameters/CPUs, restarting slurmd alone does NOT re-trigger fresh registration; slurmctld holds the stale INVALID_REG state. Operator must `scontrol update nodename=<n> state=DOWN reason=... && scontrol delete nodename=<n>` from the scheduler, *then* restart slurmd on the compute node. Build automation around this if you're iterating on `--conf` strings.
- **`nvidia-smi --query-gpu=name` returns multi-token product strings** (v0.13.4 fix). On H100 it's "NVIDIA H100 80GB HBM3". Naive `awk '{print tolower($NF)}'` yields "hbm3" not "h100" — Slurm registers `Gres=gpu:hbm3:8` and every `--gres=gpu:h100:N` job stays PENDING with reason `Resources`. Parse the first token matching `/^[A-Za-z][0-9]+$/` to pick out the SKU code (h100, h200, a100, v100, etc.).
- **NDv5 needs `Parameters=l3cache_as_socket` on slurmd's `--conf`, NOT `SlurmdParameters=` in slurm.conf** (v0.13.4 fix). Setting it via slurm.conf is silently ignored by slurmd (the option is per-node and read from the conf-server reply, not the global config). Setting it on the slurmd command line via `--conf "... Parameters=l3cache_as_socket"` works. Also: dropping `CPUs=` from `--conf` is required for autodetected S:C:T to honour the L3-as-socket grouping; explicit `CPUs=96` collapses everything back to one socket and confuses NCCL's GPU↔NIC affinity.
- **The 64 GB root disk on NDv5 is too small for container imports** (v0.13.4 fix). The `microsoft-dsvm:ubuntu-hpc:2404` image leaves ~7 GB free on `/` after base install. A single 10 GB Pyxis container import (e.g. `ghcr.io/azure/ai-infrastructure-on-azure/nccl-test:latest`) ENOSPC's mid-extraction. Always relocate `/var/lib/enroot` and `/var/lib/enroot-data` to symlinks under `/mnt` (the Azure ephemeral disk, ~956 GB NVMe-backed on NDv5) when `/mnt` is mounted.
- **HPC-X is the in-image MPI on Azure HPC/AI VMs** (v0.13.4 lesson). `/opt/hpcx-v2.25.1-gcc-doca_ofed-ubuntu24.04-cuda13-x86_64/` ships with OpenMPI + UCX + a bundled PMIx 4.2.x that ABI-matches Slurm 25.11's `mpi_pmix_v4.so`. Use it for bare-metal multi-node MPI/NCCL jobs by `source $HPCX_DIR/hpcx-init.sh && hpcx_load` then `srun --mpi=pmix ...`. The marketplace image also ships `/opt/nccl-tests/build/all_reduce_perf` prebuilt — no need to compile NCCL tests yourself.
- **Cross-node containerised MPI via Pyxis needs slurmd PMIx env + enroot PMI hooks** (v0.13.6 fix; supersedes earlier "PMIx 4↔5 ABI" claim; **live-validated v0.13.6 on `paul-azcluster-h100c`**: 2-node × 8-GPU NeMo container all-reduce reached a single 16-rank PMIx world and exited cleanly; orthogonal IB-visibility bug surfaced — see next gotcha). The real failure mode on v0.13.4/v0.13.5 was NOT an ABI mismatch. Slurm 25.11 ships `mpi_pmix_v4.so` linked against PMIx 4.2.9; ND v5 host has PMIx 4.2.9; **NGC containers `nvcr.io/nvidia/nemo:25.07.02` and `nvcr.io/nvidia/pytorch:24.10/24.11/25.04-py3` ship HPC-X 2.20/2.21 → PMIx 4.2.x** — the ABIs already match. The actual root cause is twofold: (1) slurmd has no `PMIX_MCA_*` env so the PMIx server tmpdir / security / GDS modules drift between ranks; (2) Pyxis needs the per-rank `PMIX_*` env propagated INTO the container plus the PMIx server socket bind-mounted. CCWS (proven on GB300 in Jan 2026) fixes this without rebuilding anything by appending `PMIX_MCA_ptl=^usock`, `PMIX_MCA_psec=none`, `PMIX_SYSTEM_TMPDIR=/var/empty`, `PMIX_MCA_gds=hash`, `HWLOC_COMPONENTS=-opencl` to slurmd's EnvironmentFile (`/etc/default/slurmd` on Ubuntu, `/etc/sysconfig/slurmd` on RHEL) AND installing the two upstream enroot hooks `50-slurm-pmi.sh` + `50-slurm-pytorch.sh` from `/usr/share/enroot/hooks.d/extra/` into `/etc/enroot/hooks.d/`. The `50-slurm-pmi.sh` hook copies all `PMIX_*` and `SLURM_*` env into the container's `${ENROOT_ENVIRON}` and bind-mounts `$PMIX_SERVER_TMPDIR` into the container via `${ENROOT_MOUNTS}` — without it Pyxis containers can't talk to the host PMIx server and each rank initialises as a singleton MPI world. Source: `research/azurehpc/experimental/cc_slurm_pyxis_enroot/.../files/slurmd_pmix.sh` + `research/cyclecloud-slurm/azure-slurm-install/install.py:926-1000+`. The `ghcr.io/azure/ai-infrastructure-on-azure/nccl-test:latest` image uses HPC-X 2.26, which bundles PMIx 4.2.9 (not PMIx 5.x). Confirmed via NVIDIA HPC-X 2.26 release notes and Azure HPC image specifications. All major NGC training containers (PyTorch, NeMo, TensorFlow) from 2024–2025 ship HPC-X 2.20–2.25, all carrying PMIx 4.2.x. No PMIx 5 found in production NGC containers as of May 2026. Rebuilding Slurm with `mpi_pmix_v5.so` is NOT required for v0.13.6.
- **NDv5 NCCL key env vars** (v0.13.4 confirmed working): `NCCL_IB_HCA=mlx5_ib` (NOT `mlx5` — that also matches IPoE devices), `NCCL_TOPO_FILE=/opt/microsoft/ndv5-topo.xml`, `UCX_NET_DEVICES=mlx5_ib0:1,mlx5_ib1:1,...,mlx5_ib7:1`. Look for `NVLS multicast support is available`, `Initialized NET plugin IBext_v11`, `NET/IB : Using [0]mlx5_ib0:1/IB/SHARP ... [7]mlx5_ib7:1/IB/SHARP`, and `NCCL RDMA Plugin v11` (HPC-X's `nccl_rdma_sharp_plugin`) in `NCCL_DEBUG=INFO`. Without those NCCL falls back to TCP/IPoE and you'll see drastically lower busbw than the 466.33 GB/s peak / 348.02 GB/s avg seen on 2-node H100.
- **ANF NFSv4.1 with anonymous UID mapping creates `nobody:nogroup` files**. Anything `root` writes onto `/shared` (e.g. example sbatches dropped by the scheduler bootstrap) lands as `nobody:nogroup 700` and is unreadable by `azureuser`. Always `chown azureuser:azureuser /shared/...` after writing as root, or write as `azureuser` via `runuser -u azureuser`.
- **NVMe Direct Disk RAID-0 (v0.13.5)**. ND96isr_H100_v5 ships 8× ~3.5 TB raw NVMe; detect via `lsblk -d -n -p -o NAME,MODEL | awk 'BEGIN{IGNORECASE=1} /Microsoft NVMe Direct Disk/ {print $1}'` (matches both v1 and v2 model strings). Compute bootstrap builds them as RAID-0 (`mdadm --create /dev/md/azcluster_nvme --level=0 --chunk=128 --metadata=1.2`), `mkfs.ext4 -L azcluster_nvme`, mounts at `/mnt/nvme` with `nofail,x-systemd.device-timeout=10`. Persisted in `/etc/mdadm/mdadm.conf` + `/etc/fstab`. Ephemeral — destroyed on deallocation; do NOT use for anything that must survive a stop/start. Enroot scratch precedence: `/mnt/nvme` > `/mnt` > `/var/lib`. With NVMe RAID, the ~20 GB NeMo container imports in seconds.
- **DGXC (NVIDIA dgxc-benchmarking) compat needs three Enroot pieces** (v0.13.5). (1) `/etc/enroot/enroot.conf` MUST set `ENROOT_REMAP_ROOT yes` in addition to `ENROOT_ROOTFS_WRITABLE yes`. (2) `/etc/enroot/environ.d/50-nccl.env` MUST carry NCCL/UCX env vars — `/etc/profile.d/*.sh` is NOT sourced inside Pyxis containers because the entrypoint is a non-login shell. (3) `/etc/enroot/mounts.d/50-azcluster.fstab` MUST bind-mount `/opt/microsoft` so `NCCL_TOPO_FILE=/opt/microsoft/ndv5-topo.xml` resolves inside the container. All three are written by `cloud-init/compute.yaml.tmpl` in the GPU block. Without any of these, NCCL silently falls back to TCP and you'll see ~10× slower throughput.
- **CCWS (cyclecloud-slurm-workspace) parity**. Pinned versions: Pyxis 0.21.0 + Enroot 4.0.1 (confirmed in `Azure/cyclecloud-slurm` → `azure-slurm-install/{ubuntu,rhel}.sh`). azcluster pins the same. - **NCCL inside Pyxis containers can't see IB on NDv5 H100 by default** (v0.13.6 finding; v0.13.7 partial fix; **v0.13.8 complete fix, live-validated 434 GB/s busbw multinode SHARP**). Symptom (pre-fix): `dgxc-nemo-multinode-smoke.sbatch` runs to completion (16-rank PMIx world forms cleanly) BUT `NCCL_DEBUG=INFO` shows `NET/IB : No device found.` on every rank and NCCL falls back to `OOB eth0:10.42.4.x` — single-NIC TCP instead of the 8× NDR400 IB fabric. **Real root cause (corrected v0.13.8)**: enroot's default `/dev` handling does NOT bind-mount `/dev/infiniband` into containers. The earlier v0.13.7 diagnosis (REMAP_ROOT + 0660 perms blocking uverbs access) was a real but secondary problem; without the device nodes inside the container at all, perms are moot. **Complete fix = two pieces, both shipped by `cloud-init/compute.yaml.tmpl`**: (1) `MELLANOX_VISIBLE_DEVICES=all` in `/etc/enroot/environ.d/50-nccl.env` triggers enroot's shipped `/etc/enroot/hooks.d/99-mellanox.sh` hook to bind-mount `/dev/infiniband/{uverbs,umad,issm}*` + `/dev/infiniband/rdma_cm` (plus matching sysfs entries) into every Pyxis container (this is the *operative* fix — without it, no device nodes in the container). (2) The v0.13.7 udev rule (`MODE="0666"` on `uverbs*`/`rdma_cm`/`ucm*`/`umad*`/`issm*` + immediate `chmod` + `udevadm trigger`) is retained because the mellanox hook bind-mounts preserving host perms (`0660 root:root` on ubuntu-hpc); with `ENROOT_REMAP_ROOT yes` the in-container root maps to a host non-root uid that cannot otherwise open them. Diagnostic: inside container `ibv_devinfo` should list 8 `mlx5_ib*` devices and `NCCL_DEBUG=INFO` should print `NET/IB : Using [0]mlx5_ib0:1/IB/SHARP ... [7]mlx5_ib7:1/IB/SHARP`. Validated on `paul-azcluster-h100d` (job 11, 2× ND96isr_H100_v5, `nvcr.io/nvidia/nemo:25.07.02`): `avg busbw=434.40 GB/s` at 1 GiB, 16-rank all-reduce with SHARP + GPUDirect RDMA. Equivalent CCWS pattern: same env var, same enroot hook (which ships with enroot 4.0.1 as `99-mellanox.sh`).

- **Cloud-init duplicate-writer bug** (v0.16.1 fix). When adding a new block to a `runcmd:` that writes to a path some earlier prototype block also writes to, the LAST writer wins because runcmd is sequential. v0.16 added a wrapper at `/usr/local/sbin/azcluster-healthcheck` that delegated to the new `/usr/local/bin/azhealthcheck` Rust binary, but left a pre-existing legacy inline-shell wrapper writing the same path further down `cloud-init/compute.yaml.tmpl` in place. The legacy block ran after the v0.16 block on every boot and overwrote it, so the Rust binary was never invoked. Lesson: when adding a cloud-init writer for a path, `grep -n '<path>' cloud-init/*.tmpl` first and consolidate. The legacy script ALSO hit the `command -v nvidia-smi` gotcha — drained every CPU node every 5 min with `Reason=azcluster-healthcheck: nvidia-smi -L failed`. Live-reproduced on `paul-azcluster v16a` (southafricanorth, 2x Standard_D8as_v5) before the fix.
